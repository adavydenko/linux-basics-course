# Корректорская вычитка русскоязычных текстов

Этот скилл — это чек-лист и набор правил для финальной вычитки русскоязычной технической документации. Он рассчитан на учебные пособия в формате Markdown с расширением `.ru.md`, но применим к любому русскоязычному техническому тексту.

## Принцип работы

Корректурой называется **финальная** правка — после того, как содержательная и стилистическая работа уже сделана. Корректор не переписывает абзацы и не меняет смысл; он ловит мелкие огрехи, накапливающиеся при крупной работе с текстом. Поэтому скилл устроен как чек-лист: каждый пункт — узкая категория с однозначным правилом «как должно быть», и его применение не должно менять смысл текста.

Если по ходу вычитки вы видите проблему, требующую содержательной правки (например, абзац построен «командой → командой → командой» без пояснений), — это не корректура. Не правьте такие места молча; добавьте их в отчёт как «требует ручного решения».

## Шаги вычитки

1. **Определите цель.** Это один файл, каталог, или весь проект? Соберите список путей.
2. **Прогоните автоматические правки.** Категории A1–A9 ниже — это безопасные замены, которые можно делать механически. Применяйте их через Edit/Grep по всему набору файлов.
3. **Соберите список ручных решений.** Категории M1–M5 — это места, где нужно человеческое суждение. Запишите их в отчёт по форме «файл:строка — что не так».
4. **Сообщите итог.** Сводка: сколько автоматических правок применено по каждой категории, сколько ручных решений требуется.

## A. Автоматические правки

Эти изменения безопасны и могут применяться через replace_all или массовый Edit.

### A1. Языковые метки кодовых блоков

Канон для технических пособий — единая метка `shell` для shell-команд. Внутри проекта язык должен быть одинаковым; вперемешку `bash`, `sh`, `console`, `terminal` создают визуальную пестроту.

Правила замены:

- ` ```bash ` → ` ```shell ` (Bourne Again Shell-команды показываются в shell-промпте);
- ` ```sh ` → ` ```shell ` (так же);
- ` ```console ` → ` ```shell ` (та же история);
- ` ```terminal ` → ` ```shell ` (то же);
- `bash` оставляется только при показе именно bash-скриптов (файл-скрипт целиком).

Для других языков канон такой:

- systemd unit-файлы → ` ```ini `;
- YAML-конфиги → ` ```yaml `;
- crontab → ` ```cron `;
- многострочный нейтральный вывод/конфиг без подсветки → ` ```text `.

### A2. Кавычки

В русском тексте — «ёлочки», в кодовых блоках — `"ASCII"` (как пишется в реальных командах).

Правила:

- `"..."` в обычном тексте → `«...»`;
- `'...'` в обычном тексте → `«...»` (если это не цитата кода);
- Внутри ` ```...``` `, ` `inline code` ` и атрибутов изображений `![alt]` — ничего не трогать;
- Внутри кода типа `«find -name`» — это ошибка наоборот, должно быть `"find -name"` (но такие случаи редки, осторожно).

### A3. Тире и дефис

- Между словами с пробелами (как тире) — должно быть длинное `—` (em dash), не короткое `-`. Пример: `работает только в пределах одной сети — коммутатор связывает`.
- В числовых диапазонах — короткое `–` (en dash) без пробелов: `5–10 секунд`, `2026–2027 гг.`;
- В составных словах — обычный дефис `-`: `IT-специалист`, `по-разному`.

Не трогать дефис внутри кода: `apt-get`, `multi-user.target`, `chown user:group`.

### A4. Лишние пробелы и переводы строк

- Два и более пробелов подряд внутри строки → один пробел (не считая выравнивания таблиц);
- Trailing whitespace в конце строк → удалить;
- Три и более пустых строк подряд → две;
- Конец файла без перевода строки → добавить.

### A5. Числа и единицы измерения

- Неразрывный пробел между числом и единицей: `100 МБ`, `5 мин`, `1 ГБ` — если документ это поддерживает; иначе обычный пробел.
- Между числами и `%` — без пробела: `5%`, `100%` (русская типографика);
- Между числом и `°C`, `°F` — без пробела.

### A6. Многоточие

- `...` (три точки) в обычном тексте → `…` (один символ);
- Внутри кодовых блоков — оставить `...`.

### A7. Типичные русские опечатки

Заменяй только при полной уверенности (грамматически корректное направление):

- «тоже самое» → «то же самое»;
- «также» / «так же» — не трогать без контекста (разные значения);
- «чтобы» / «что бы» — не трогать без контекста;
- «потому что» как одно слово «потомучто» → разделить;
- «в течении» (когда «в течение» — про время) — требует контекста, проверяй сам.

### A8. Заголовки

- В заголовках H1/H2/H3 (# / ## / ###) после знака `#` должен быть один пробел: `#Заголовок` → `# Заголовок`.
- Точка в конце заголовка — лишняя: `# Введение.` → `# Введение`.
- Пустая строка после заголовка обязательна (Markdown-стандарт).

### A9. Списки

- Между концом абзаца и началом списка — пустая строка;
- Маркер списка одинаковый по всему документу: либо `-`, либо `*`, не вперемешку;
- Нумерованные списки начинаются с `1.`, не `0.` и не `01.`.

## M. Места, требующие ручного решения

Эти проблемы нельзя править автоматически — они требуют контекста и осмысленного решения.

### M1. Согласование «вы»/безличное

Если в одном разделе автор пишет «вы откроете файл», а через два абзаца — «открывается файл», стиль рваный. Корректор отмечает места, но не правит — выбор между активным и безличным залогом меняет тон и должен быть сознательным.

В отчёт: `файл:строка — переключение залога: «...» рядом с «...»`.

### M2. Английские термины без русского эквивалента

Если английское слово встречается первый раз в тексте без перевода или пояснения, добавить его — задача автора, не корректора. Корректор отмечает.

Пример «требует решения»: `файл:строка — «namespaces» введён без русского эквивалента (например: «namespaces (пространства имён)»)`.

### M3. Аббревиатуры без расшифровки

То же — отмечаем при первом упоминании в файле. Если аббревиатура расшифрована в глоссарии или в предыдущем модуле, проверка нестрогая — но для печатного пособия расшифровка при первом упоминании в каждом разделе — норма хорошего тона.

### M4. «Обзывающий» стиль

Конструкции вроде «вот так делается X», «нужно сделать Y», «делайте Z» без объяснения зачем — отмечаются как кандидаты на переписывание. Не правим: альтернатива требует знания материала.

### M5. Последовательность команд без пояснений

«Выполните команду 1. Выполните команду 2. Выполните команду 3.» — если между командами нет связующего текста («Теперь, когда X, нужно Y»), это «команда-конвейер», которая для печати не годится. Отмечаем места.

## Формат отчёта

В конце вычитки выдай отчёт в таком виде:

```
## Сводка автоматических правок
- A1 (языковые метки кодовых блоков): N замен в M файлах
- A2 (кавычки): N замен
- A3 (тире и дефис): N замен
- A4 (пробелы и переводы): N мест
- ...

## Места, требующие ручного решения

### Файл: docs/.../X.ru.md
- строка K: M1 — переключение залога
- строка L: M2 — английский термин «...» без русского эквивалента

### Файл: docs/.../Y.ru.md
- строка K: M4 — «обзывающий» стиль в разделе «...»
- ...
```

Не комментируй каждое отдельное исправление в категориях A — пользователю важен агрегат. По M-категориям нужен файл и строка.

## Ограничения

- Не трогай содержимое кодовых блоков ` ```...``` ` и inline кода ` `...` ` — это технический контент, в нём опечатки и ASCII-кавычки часть синтаксиса.
- Не правь YAML-фронтматтер (между двумя `---`), путаница с типами там может поломать сборку.
- Не правь содержимое внутри атрибутов изображения `![alt](path)` — alt-текст имеет отдельный канон.
- Не правь содержимое HTML-тегов (если встречаются): `<details>`, `<summary>`, и т.п.
- В таблицах будь внимателен — выравнивание пробелами иногда визуально, иногда содержательно.

## Объяснение «зачем»

Этот скилл — компромисс между «придраться к каждой запятой» и «оставить как есть». Те 9 категорий, что вынесены в автоматические, — это места, где у русскоязычной типографики и Markdown-конвенций есть единственно правильный ответ; их правка экономит часы ручного просмотра и не несёт риска. Категории M — наоборот: они выглядят как «вот тут шероховато», но решение принимает автор, а не корректор.

Главное: корректура — не отмазка от плохого первого черновика, а финальный пас, после которого текст можно отдавать на верстку.
