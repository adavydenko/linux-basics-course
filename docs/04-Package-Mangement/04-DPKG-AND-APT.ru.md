# Пакетные менеджеры DPKG и APT

- [Доп. видео](https://kodekloud.com/topic/dpkg-and-apt/)

В мире RHEL мы видели пару «`rpm` плюс `yum`»: низкоуровневый инструмент и удобная надстройка над ним. В семействе Debian — Ubuntu, Debian, PureOS и похожих системах — устроено ровно так же, только инструменты называются иначе. Если уловить эту параллель, новый набор команд ложится на уже знакомую модель.

## Цели раздела

После раздела студент должен уметь:

- объяснить роль `dpkg`;
- понимать ограничение `dpkg`: он не решает зависимости автоматически;
- использовать `apt` для установки, обновления, удаления и поиска пакетов;
- понимать назначение `/etc/apt/sources.list`.

## DPKG

`dpkg` (Debian Package Manager, «менеджер пакетов Debian») — это низкоуровневый пакетный менеджер семейства Debian. Он работает напрямую с файлами пакетов `.deb` и в этой паре играет ту же роль, что `rpm` в мире RHEL.

### Работа с DPKG

Набор режимов `dpkg` тоже знаком — он покрывает весь жизненный цикл пакета:

1. Установка.
2. Удаление.
3. Обновление.
4. Просмотр списка пакетов.
5. Проверка статуса.
6. Проверка пакета.

| Режим | Что делает |
|---|---|
| Установка | поставить `.deb`-пакет |
| Удаление | удалить пакет |
| Обновление | обновить версию |
| Просмотр списка | показать установленные пакеты |
| Проверка статуса | узнать состояние пакета |
| Проверка пакета | сверить файлы пакета с системой |

## APT и APT-GET

И ограничение у `dpkg` то же самое, что у `rpm`: он не разрешает зависимости. Если устанавливаемому `.deb`-пакету нужны другие пакеты, а их в системе нет, установка просто завершится ошибкой — `dpkg` не пойдёт их искать.

Эту брешь закрывают инструменты уровнем выше — `apt` (Advanced Package Tool) и его более старый родственник `apt-get`. Они умеют ставить программу вместе со всеми зависимостями:

```shell
$ sudo apt install gimp
$ sudo apt-get install gimp
```

`apt` выступает как frontend — удобная «лицевая» часть системы управления пакетами, — а саму низкоуровневую установку под капотом всё равно выполняет `dpkg`. Это в точности та же конструкция, что связка `yum` и `rpm`: вы работаете с удобным инструментом, а грязную работу делает базовый.

Откуда `apt` берёт пакеты, описано в файле `/etc/apt/sources.list` и в дополнительных файлах каталога `/etc/apt/sources.list.d/` — это и есть список репозиториев APT.

## Частые команды

Разберём команды `apt`, которые встречаются чаще всего.

Первое, что стоит сделать, — обновить локальный список пакетов. `apt update` не трогает сами программы, а лишь скачивает из репозиториев свежую информацию о том, какие версии сейчас доступны:

```shell
$ sudo apt update
```

А вот `apt upgrade` уже устанавливает обновления для тех пакетов, что стоят в системе. Поэтому на практике эти две команды обычно идут парой: сначала `update`, затем `upgrade`:

```shell
$ sudo apt upgrade
```

Если нужно заглянуть в список источников пакетов и поправить его, удобна команда `apt edit-sources` — она открывает в редакторе основной файл `/etc/apt/sources.list`. В современных Ubuntu (22.04+) основной файл часто пустой, а реальные репозитории описаны отдельными `.list`-файлами в каталоге `/etc/apt/sources.list.d/`; чтобы посмотреть всё, заглядывают и туда:

```shell
$ sudo apt edit-sources
$ ls /etc/apt/sources.list.d/
```

Установка пакета:

```shell
$ sudo apt install telnet
```

Удаление пакета — зеркальная операция:

```shell
$ sudo apt remove telnet
```

Когда вы не уверены в точном имени пакета, его можно поискать в репозитории:

```shell
$ sudo apt search telnet
```

А чтобы посмотреть все пакеты, в названии которых встречается `telnet`, список можно отфильтровать через `grep`. `apt list` без опций перечисляет вообще все доступные пакеты — это длинный вывод, поэтому добавляют `2>/dev/null`, чтобы подавить служебное предупреждение «WARNING: apt does not have a stable CLI interface»:

```shell
$ apt list 2>/dev/null | grep telnet
```

## Разобранный пример: ставим `nginx` с нуля и проверяем

**Задача.** Поставить веб-сервер `nginx` на свежую Ubuntu-машину так, чтобы убедиться: сервис не просто установлен, а реально слушает порт 80.

**Решение.**

1. Перед любым `install` обновляем локальный кэш репозиториев: `sudo apt update`. Иначе `apt` может попытаться скачать версию пакета, которой уже нет в зеркале, и установка прервётся.
2. Ставим сам пакет: `sudo apt install nginx`. APT покажет список зависимостей и спросит подтверждение — отвечаем `y`. В скрипте на этом месте добавили бы `-y`, но при ручной работе полезно сначала прочитать, что именно собрались поставить.
3. Проверяем, что `nginx` появился в системе и каким пакетом он принесён: `dpkg -l | grep nginx`. Должна быть строка с статусом `ii` — «installed, ok».
4. Смотрим, запустился ли сервис: `systemctl status nginx`. На Ubuntu сервис стартует автоматически после установки; если этого не произошло — `sudo systemctl start nginx`.
5. Проверяем порт: `ss -tlnp | grep :80`. В выводе должна быть строка с `LISTEN` и процессом `nginx`. Это и есть подтверждение «сервер действительно работает», а не «пакет лежит на диске».

**Проверка.** `curl -I http://localhost` должен вернуть `HTTP/1.1 200 OK` и строку `Server: nginx`. Если возвращает ошибку соединения — порт закрыт фаерволом или сервис не запущен; если 404 — `nginx` работает, но не отдаёт стартовую страницу.

> **Типичные ошибки.**
>
> - **`sudo` забыт в `apt install`.** Без `sudo` APT не сможет писать в системные каталоги и завершится сообщением «Permission denied» или «Are you root?». Это самая частая ошибка новичка: команда вроде «не работает», хотя проблема одна — нет прав.
> - **`apt upgrade` без предварительного `apt update`.** `upgrade` ориентируется на локальный кэш; если кэш не обновлён, APT не увидит свежие версии и сообщит, что обновлять нечего. Привычка — всегда сначала `update`, потом `upgrade`.
> - **Установка `.deb`-файла напрямую через `dpkg -i`.** Если у пакета есть зависимости, `dpkg` остановится с ошибкой и оставит систему в «полу-установленном» состоянии. Лечится командой `sudo apt -f install` — она доставляет недостающее. На будущее: для одиночного `.deb` лучше использовать `sudo apt install ./package.deb` — APT сам подтянет зависимости.
> - **`apt remove` оставляет конфигурации.** После `apt remove nginx` файлы из `/etc/nginx/` остаются на месте — это сделано осознанно, чтобы не потерять настройки. Чтобы вычистить пакет полностью, нужен `apt purge nginx`.

## Самопроверка

- [ ] могу установить пакет так, чтобы перед этим обновить кэш и прочитать список зависимостей;
- [ ] объясню, чем `apt update` отличается от `apt upgrade`, и в каком порядке их запускают;
- [ ] могу проверить, что пакет установлен, через `dpkg -l` и через `apt list --installed`;
- [ ] объясню, в каких случаях `apt install ./package.deb` лучше, чем `dpkg -i package.deb`;
- [ ] прочитаю состояние пакета в выводе `dpkg -l` (колонка статуса `ii`, `rc`, `un`) и скажу, что оно означает;
- [ ] могу удалить пакет с сохранением конфигурации и без неё, выбрав правильную команду.

## Практическое задание

1. Выполните `sudo apt update` и объясните, почему это не обновляет сами программы.
2. Найдите пакет `telnet` через `apt search`.
3. Посмотрите файл `/etc/apt/sources.list`.
4. Объясните разницу между `dpkg -i package.deb` и `apt install package`.
