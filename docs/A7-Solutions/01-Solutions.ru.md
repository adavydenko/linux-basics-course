# Решения к банку упражнений

Документ для преподавателя. По каждому модулю представлены решения 3–5 задач — преимущественно аналитических, проектных и отладочных. Репродуктивные задачи решаются «по тексту модуля» и здесь не дублируются; для них в банке достаточно сверки с выводом команд из соответствующего раздела.

Нумерация задач совпадает с [банком упражнений](../A3-Exercises/01-Exercises.ru.md).

## Модуль 00. Настройка окружения

### Задача 4 (аналитическая). Две причины выполнять задания в контейнере

**Эталон.** Первая — изоляция: ошибки внутри контейнера не повреждают основную систему. Студент может удалить файл, сломать сервис, заполнить диск — и просто пересоздать контейнер за минуту. Вторая — воспроизводимость: образ один и тот же у всех студентов курса, поэтому пример из пособия гарантированно отрабатывает одинаково, без «у меня другая версия пакета».

**На что обращать внимание.** Студент часто называет третью причину — «безопасность» — и забывает про две первые. Безопасность тоже верна, но без изоляции и воспроизводимости она не главное преимущество.

### Задача 7 (отладка). `docker run linux-basics-lab` — контейнер сразу выходит

**Эталон.** Не хватает двух флагов: `-i` (interactive — держать stdin открытым) и `-t` (TTY — выделить псевдо-терминал). Без них `bash` запускается и тут же завершается, так как у него нет ни ввода, ни связанного терминала. Правильно: `docker run -it linux-basics-lab`. На практике часто добавляют ещё `--rm`, чтобы контейнер удалился после выхода, и не копился мусор.

**На что обращать внимание.** Если студент говорит «нужен `-d`», он перепутал интерактивный режим с фоновым. `-d` запустит контейнер в фоне, и в нём так же нечего делать — флаг полезен только для долгоживущих сервисов.

## Модуль 02. Shell, часть I

### Задача 5 (аналитическая). Абсолютный vs относительный путь

**Эталон.** Абсолютный путь начинается с `/` и однозначно указывает на файл независимо от текущего каталога (`/etc/hosts`, `/home/bob/notes.txt`). Относительный путь интерпретируется относительно текущего каталога (`../config`, `./script.sh`). Главное различие — переносимость: команда с абсолютным путём отработает одинаково из любого каталога; относительный путь даст разный результат в разных местах.

**На что обращать внимание.** Формулировка «относительный путь — без `/`» технически верна, но не отражает сути. Уточняющий вопрос: «а что вообще значит "относительно"? Относительно чего?»

### Задача 7 (проектирование). PS1 с пользователем, хостом, каталогом

**Эталон.** В `~/.profile` строка:

```shell
export PS1='[\u@\h \w]\$ '
```

Подстановки: `\u` — имя пользователя, `\h` — короткое имя хоста, `\w` — текущий каталог (с заменой `$HOME` на `~`), `\$` — `#` для root и `$` для остальных. Изменение применится со следующей сессии — или сейчас, если выполнить `source ~/.profile`.

**Альтернативы.** Можно положить в `~/.bashrc` вместо `.profile` — `.bashrc` читается интерактивными не-login оболочками. На сервере оба варианта дадут одинаковый эффект. Цветовое оформление через ANSI-коды (`\[\033[1;32m\]\u@\h\[\033[0m\]`) — допустимая надстройка, но не критерий правильности.

### Задача 8 (отладка). `cd` в скрипте не меняет каталог в вызывающей оболочке

**Эталон.** Скрипт выполняется в *дочернем* процессе оболочки. Изменения каталога, переменных окружения, состояния оболочки — всё это происходит в дочернем процессе и теряется при его завершении. Чтобы команды влияли на вызывающую оболочку, скрипт нужно либо *включить* (source): `source script.sh` или `. script.sh`; либо превратить в shell-функцию, объявленную в `.bashrc`.

**На что обращать внимание.** Студент путает «как работает `cd`» с «как работает скрипт». Хорошее уточнение: «А что произойдёт, если ты вместо `./script.sh` запустишь `. script.sh`?»

## Модуль 03. Ключевые понятия Linux

### Задача 4 (аналитическая). Ядерное и пользовательское пространство

**Эталон.** Ядерное пространство (kernel space) — это область памяти и режим работы процессора, в котором выполняется ядро: оно имеет полный доступ к памяти, оборудованию, системным ресурсам. Пользовательское пространство (user space) — память и режим, где работают обычные программы; их доступ к оборудованию опосредован. Граница между ними проходит по системному вызову (syscall): когда программе нужно открыть файл или прочитать сокет, она делает syscall — управление переходит в ядро, ядро выполняет операцию, возвращает результат в пользовательское пространство.

В примере `cat /etc/os-release` основные syscall'ы: `open()` для открытия файла, `read()` для чтения, `write()` для вывода в stdout, `close()` для закрытия. Между ними — пользовательский код самой `cat`, работающий в user space.

**На что обращать внимание.** Если студент говорит «через файлы» — это поверхностно. Глубже: «через интерфейс системных вызовов». Хороший наводящий вопрос: «А что делает программа, когда хочет узнать время? Она же не лезет к часам сама».

### Задача 6 (проектирование). Перевести Ubuntu из графического в текстовый режим

**Эталон.**

```shell
$ sudo systemctl set-default multi-user.target
$ sudo systemctl isolate multi-user.target
```

Первая команда меняет цель загрузки по умолчанию, чтобы изменение пережило перезагрузку. Вторая — переключает текущее состояние без перезагрузки. Чтобы только временно (до ребута): только `isolate`. Чтобы только постоянно (с перезагрузкой): только `set-default` + `reboot`.

**Альтернативы.** Можно отключить только display manager: `sudo systemctl disable gdm` (или `lightdm`, `sddm` в зависимости от окружения). Это менее радикально — система остаётся в `graphical.target`, но без графики на старте.

## Модуль 05. Shell, часть II

### Задача 5 (аналитическая). Порядок `>` и `2>&1`

**Эталон.** `cat file.txt > out.txt 2>&1` перенаправляет сначала stdout в `out.txt`, затем дублирует stderr на stdout — и stderr тоже пишется в `out.txt`. Если поменять местами — `cat file.txt 2>&1 > out.txt` — сначала stderr перенаправляется туда, куда указывает stdout (а это всё ещё терминал, потому что `>` ещё не сработал); потом stdout переключается на файл, но stderr остаётся в терминале. Порядок применения — слева направо, и каждое перенаправление снимает «снимок» текущего значения целевого fd.

**На что обращать внимание.** Студент часто запоминает как заклинание «2>&1 в конце». Хорошее уточнение: «а почему в bash есть короткая форма `&>` для того же — что она делает?»

### Задача 6 (проектирование). Конвейер для `/etc/passwd`

**Эталон.**

```shell
$ cut -d: -f1,3 /etc/passwd | awk -F: '$2>=1000 {print $1}' | sort | head -5
```

Шаги: `cut` оставляет первое и третье поля (имя и UID); `awk` фильтрует строки, где UID ≥ 1000; `sort` сортирует по имени; `head -5` берёт первые пять.

**Альтернативы.** Без `cut`: `awk -F: '$3>=1000 {print $1}' /etc/passwd | sort | head -5` — короче, awk сам умеет работать с полями. Эту версию обычно и выбирают опытные пользователи. Версия с `cut` показывает декомпозицию задачи на отдельные шаги, что полезно для обучения.

### Задача 7 (отладка). `grep -r 'TODO'` выводит бинарные файлы

**Эталон.** Флаг `-I` (большая i) пропускает бинарные файлы — они в выводе мешают и часто содержат «случайные» совпадения. Дополнительно полезен `-l` (lowercase L) — он печатает только имена файлов, без самих строк, что удобно для отчётов: «вот где у нас остались TODO».

```shell
$ grep -rIl 'TODO' /home/user/code/
```

**На что обращать внимание.** Студент часто решает задачу через `find ... | xargs grep` — это работает, но `grep -r` короче и поддерживает те же фильтры.

## Модуль 06. Безопасность и права

### Задача 4 (аналитическая). `DROP` vs `REJECT`

**Эталон.** `DROP` молча отбрасывает пакет — никакого ответа отправителю; со стороны выглядит как «сервера не существует», соединение висит до таймаута. `REJECT` тоже не пропускает пакет, но возвращает отправителю ICMP-сообщение «соединение отклонено», и тот сразу понимает, что порт закрыт.

`DROP` уместен на внешнем периметре сети — меньше информации потенциальному злоумышленнику, сканирование портов работает медленнее (приходится ждать таймаут на каждый порт). `REJECT` уместен внутри сети — там быстрая обратная связь важнее, и диагностика «почему не подключается» становится мгновенной вместо «висит минуту».

**На что обращать внимание.** Если студент говорит «DROP лучше», значит, не понял компромисс. Безопасность через медлительность — это слабая защита; против целеустремлённого атакующего она просто увеличивает время сканирования.

### Задача 6 (проектирование). Cron-задача для бэкапа

**Эталон.**

```cron
30 2 * * 0 /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

Где `/usr/local/bin/backup.sh` — отдельный скрипт, который вызывает `tar` или `rsync`. Прямо в crontab сложную команду не пишут — экранирование `%`, длина строки и читаемость превращают это в ад.

**Альтернативы.** systemd-таймер с `OnCalendar=Sun *-*-* 02:30:00` и `Persistent=true` — даёт лучшее журналирование и догонит пропущенный запуск, если машина была выключена. Если задача критична — выбор systemd-таймера правильнее; если задача рядовая — cron проще.

### Задача 7 (проектирование). Правила iptables

**Эталон.**

```shell
$ sudo iptables -A INPUT -i lo -j ACCEPT
$ sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
$ sudo iptables -A INPUT -p tcp -s 10.0.0.0/24 --dport 22 -j ACCEPT
$ sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
$ sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
$ sudo iptables -P INPUT DROP
$ sudo iptables -P OUTPUT ACCEPT
```

Важно: разрешения должны идти *до* запрещающего правила или политики по умолчанию `DROP`. Первое правило (`-i lo`) разрешает loopback — без него часть локальных сервисов перестаёт работать. Второе — разрешает уже установленные соединения, чтобы исходящий трафик мог получать ответы.

**На что обращать внимание.** Студент часто забывает про `lo` и `ESTABLISHED,RELATED`. Если эти два правила пропущены, в системе ломаются неочевидные вещи: `localhost`-DNS, ответы серверу при исходящем запросе.

### Задача 8 (отладка). `bob` создан, по SSH не заходит

**Эталон.** Три места:

1. **Пароль:** проверить через `sudo passwd -S bob`. Если статус `L` (locked) — пароль не задан или заблокирован; чинится `sudo passwd bob`.
2. **Оболочка:** `grep bob /etc/passwd` — если у пользователя `/usr/sbin/nologin` или `/bin/false`, вход запрещён по политике. Чинится `sudo chsh -s /bin/bash bob`.
3. **SSH-настройки:** `sudo grep -E '(AllowUsers|DenyUsers|PermitRootLogin|PasswordAuthentication)' /etc/ssh/sshd_config`. Если `AllowUsers` не включает `bob`, или `PasswordAuthentication no` без настроенных ключей — вход не пройдёт.

Журналы попыток: `sudo journalctl -u ssh --since "5 minutes ago"`. Там видно конкретную причину отказа.

**На что обращать внимание.** Если студент сразу лезет менять конфиги — это плохой ход. Хороший — сначала прочесть журнал, там обычно прямо написано «Failed password for bob» или «User bob not allowed».

## Модуль 07. Сети

### Задача 4 (аналитическая). `ping` по IP работает, по имени — нет

**Эталон.** Это однозначный признак: сеть в порядке (пакеты доходят), сломалось разрешение имён. Проверять надо DNS. Источники проблемы: пустой или неправильный `/etc/resolv.conf` (нет DNS-сервера); сам DNS-сервер недоступен (его проверить через `dig @<dns> hostname`); неправильный `/etc/nsswitch.conf` (строка `hosts:`); конкретный домен не разрешается, потому что у него нет A-записи.

**На что обращать внимание.** Студент часто говорит «значит, проблема в DNS» — и останавливается. Глубже: какой именно слой DNS? Локальный resolver? Сетевой DNS? Доменный сервер?

### Задача 7 (отладка). Перестало работать разрешение имён

**Эталон.** Проверять в порядке:

1. `cat /etc/resolv.conf` — есть ли вообще nameserver, и какой? На современных Ubuntu файл — символическая ссылка на `/run/systemd/resolve/stub-resolv.conf` с адресом `127.0.0.53`.
2. `resolvectl status` — состояние systemd-resolved, какие DNS-серверы установлены.
3. `ping <dns-сервер>` — доходит ли пакет до него?
4. `dig @8.8.8.8 example.com` — работает ли разрешение через публичный DNS? Если работает — проблема в локальном DNS, не в самой системе.
5. `/etc/nsswitch.conf` строка `hosts:` — в каком порядке проверяются источники.

**На что обращать внимание.** Хороший студент проверяет от близкого к далёкому. Плохой — сразу меняет конфиг «наугад».

## Модуль 08. Хранилища

### Задача 4 (аналитическая). MBR vs GPT

**Эталон.**

| Свойство | MBR | GPT |
|----------|-----|-----|
| Размер таблицы | до 2 ТБ | до 9.4 ЗБ (зеттабайт) |
| Число основных разделов | до 4 (или 3+extended) | до 128 без ограничений |
| Контрольная сумма таблицы | нет | есть (CRC32) |
| Резервная копия | нет | есть (в конце диска) |
| Загрузка | через BIOS | через UEFI |

Преимущества MBR — совместимость со старым железом, простота. Преимущества GPT — поддержка больших дисков, защита от повреждения (резервная копия и CRC), больше разделов без «костыля» в виде extended-контейнера.

**На что обращать внимание.** Если студент говорит «MBR устарел» — это упрощение. На встраиваемых системах, в учебных задачах, в виртуальных машинах MBR по-прежнему используется, потому что проще.

### Задача 7 (отладка). `fstab` сломал загрузку

**Эталон.** Три причины:

1. **Опечатка в имени устройства/UUID.** Если `/dev/sdb1` не существует, монтирование падает. Решение: использовать UUID (`blkid` → `UUID=...`) — он не меняется при перестановке дисков.
2. **Тип файловой системы неверный.** Например, `ext4` в строке, а на разделе на самом деле `xfs`. Решение: проверить через `blkid` или `lsblk -f` и поставить правильный тип.
3. **`pass=1` или `2` для несуществующего раздела.** При загрузке systemd попытается прогнать `fsck`, не сможет — и уйдёт в emergency mode. Решение: ставить `pass=0` для всего, что не root-раздел.

Защита от поломки: опция `nofail` в поле `<options>` — если раздел не монтируется, система продолжит загрузку без него:

```text
/dev/sdb1 /mnt/data ext4 defaults,nofail 0 0
```

**На что обращать внимание.** «Никогда не править fstab без тестовой загрузки» — практическое правило. Хорошая привычка — выполнить `sudo mount -a` после правки: если что-то не то, ошибка выйдет тут же, а не при следующей перезагрузке.

## Модуль 09. systemd

### Задача 4 (аналитическая). `start` vs `enable`

**Эталон.** `start` запускает службу здесь и сейчас, в текущем сеансе. После перезагрузки служба не поднимется — `start` ничего не записал в постоянную конфигурацию. `enable` добавляет службу в автозапуск (создаёт симлинк в каталоге `*.target.wants/`), но *не запускает* её в текущий момент. Поэтому при настройке нового сервиса обычно нужны обе команды — или один раз `enable --now`, который делает оба действия.

**На что обращать внимание.** Студент часто говорит «start — это для разового запуска, enable — для постоянного». Это неточно: `enable` сам по себе ничего не запускает; разовость/постоянство — это про то, переживёт ли запуск перезагрузку.

### Задача 6 (проектирование). `data-sync.service` + `data-sync.timer`

**Эталон.**

```ini
# /etc/systemd/system/data-sync.service
[Unit]
Description=Daily data sync

[Service]
Type=oneshot
ExecStart=/usr/local/bin/sync.sh
User=student
Restart=on-failure
RestartSec=30s
```

```ini
# /etc/systemd/system/data-sync.timer
[Unit]
Description=Daily timer for data sync

[Timer]
OnCalendar=*-*-* 04:00:00
Persistent=true
Unit=data-sync.service

[Install]
WantedBy=timers.target
```

После создания: `sudo systemctl daemon-reload && sudo systemctl enable --now data-sync.timer`.

**На что обращать внимание.** Студент часто пишет `Restart=on-failure` в сервисе с `Type=oneshot`. Это допустимо — systemd перезапустит, если скрипт упал, — но `RestartSec` обязательно: иначе при постоянных падениях systemd попадёт в петлю.

### Задача 7 (отладка). `Unit not found`

**Эталон.** Три причины:

1. **Файл не там.** Должен лежать в `/etc/systemd/system/` (для системных юнитов) или `~/.config/systemd/user/` (для пользовательских). Не в `/usr/lib/systemd/system/` — туда кладут только установленные пакетами.
2. **`daemon-reload` не сделан.** systemd закэшировал старый список юнитов. После создания файла обязательно `sudo systemctl daemon-reload`.
3. **Опечатка в имени.** `project.service` ≠ `Project.service` ≠ `project-service`. Проверить через `systemctl list-unit-files | grep -i project`.

**На что обращать внимание.** Если студент сразу пересоздаёт юнит — это перебор. Сначала: `systemctl status project.service 2>&1` — ошибка часто говорит конкретно «не нашёл такой файл по таким путям».

## Модуль 11. Обработка текста

### Задача 4 (аналитическая). `cut`, `awk`, `sed` — что выбрать?

**Эталон.** `cut -d: -f2 file` — самое простое для «достать второе поле»; не умеет ничего больше. `awk -F: '{print $2}' file` — то же, но даёт возможность условий и трансформаций: `awk -F: '$3>1000 {print $2}'`. `sed -E 's/^[^:]+:([^:]+):.*/\1/' file` — мощнее, использует регулярки; для простой задачи «достать поле» — оверкилл.

Правило: для пары `cut/awk` выбирают тот, что короче читается. `sed` — когда нужно не просто выбрать, а *преобразовать* (заменить, переставить).

**На что обращать внимание.** «Я всегда пишу awk» — плохой знак: студент не выбирает инструмент по задаче. Хороший признак — спрашивает «нужны ли условия?».

### Задача 6 (проектирование). Топ-10 URL в логе Nginx

**Эталон.**

```shell
$ awk '{print $7}' access.log | sort | uniq -c | sort -rn | head -10
```

В combined-формате Nginx 7-е поле — URL запроса. `sort | uniq -c` считает повторы; `sort -rn` сортирует по числу в обратном порядке; `head -10` берёт топ.

**Альтернативы.** Если нужно ограничить день: `awk -v day="$(date +%d/%b/%Y)" '$0 ~ day {print $7}' ...` (но надёжнее регулярка для даты). Если log-файл сжатый — `zcat access.log.gz | awk ...`.

### Задача 7 (отладка). `awk -F: '$3>=1000 {print $1}' /etc/passwd` — пустой вывод

**Эталон.** Две возможные причины:

1. **Файл не тот.** Если awk запущен от ограниченного пользователя в неправильной директории — может быть открыт пустой `passwd` (например, из chroot-окружения).
2. **Локаль.** В некоторых локалях `>=` работает иначе с числами, представленными как строки. Решение: `$3+0>=1000` — явное приведение к числу. Это редкий, но реальный случай.

**На что обращать внимание.** Прежде всего — `wc -l /etc/passwd`: убедиться, что файл вообще не пустой. Это первая проверка, прежде чем грешить на регулярки.

## Модуль 12. Shell-скриптинг

### Задача 4 (аналитическая). `set -euo pipefail`

**Эталон.** `-e` (`errexit`) останавливает скрипт при первой команде, вернувшей ненулевой код. `-u` (`nounset`) — ошибка при обращении к необъявленной переменной (защищает от опечаток в именах). `-o pipefail` — код возврата конвейера равен коду последней команды с ненулевым кодом (без этого `false | true` возвращает 0, и ошибка теряется).

**На что обращать внимание.** «Это для безопасности» — поверхностный ответ. Глубже: «для того, чтобы ошибки не маскировались — скрипт падает там, где сломалось, а не молча идёт дальше с неверным состоянием».

### Задача 6 (проектирование). `backup-home.sh`

**Эталон.**

```shell
#!/usr/bin/env bash
set -euo pipefail

user="${1:-}"
[[ -z "$user" ]] && { echo "usage: $0 USERNAME" >&2; exit 1; }

if ! id "$user" >/dev/null 2>&1; then
  echo "user not found: $user" >&2
  exit 2
fi

home="/home/$user"
dst="/backup/${user}-$(date +%F).tar.gz"
log="/var/log/backup-home.log"

trap 'echo "interrupted, partial archive at $dst" >&2' INT TERM

tar -czf "$dst" "$home" 2>>"$log"
echo "ok: $dst"
```

**На что обращать внимание.** Если студент не сделал `trap` — задача всё равно решена, но грязная: при `Ctrl+C` остаётся недописанный архив. Если не разделил `id`-проверку и `tar` — менее ясно, но рабоче.

### Задача 7 (отладка). `for f in $(ls *.txt)` ломается на пробелах

**Эталон.** Конструкция `$(ls *.txt)` подставляется как строка, которую `for` разбивает по пробелам — и `My Notes.txt` превращается в два «файла»: `My` и `Notes.txt`. Решение: использовать glob прямо в `for` — он сам корректно работает с пробелами:

```shell
for f in *.txt; do
  echo "<$f>"
done
```

**На что обращать внимание.** Это классический подвох. Хороший наводящий: «А что вернёт `ls`, если файлов нет вообще? И что `for` с этим сделает?»

## Модуль 13. Процессы и ресурсы

### Задача 4 (аналитическая). `kill -15` vs `kill -9`

**Эталон.** `SIGTERM` (15) — «вежливая» просьба завершиться. Программа получает сигнал, выполняет обработчик: сохраняет файлы, закрывает соединения, освобождает ресурсы, завершается с кодом, понятным родителю. Это корректное завершение. `SIGKILL` (9) — принудительное завершение силами ядра. Процесс не получает сигнал в обработке; ядро просто удаляет его из таблицы процессов. Данные в памяти теряются, открытые файлы могут остаться в полу-записанном виде, дочерние процессы получают `init` в родители и могут осиротеть.

Поэтому правило: всегда сначала `SIGTERM`, и только если процесс не реагирует за 5–10 секунд — `SIGKILL`. Многие сервисы вообще не имеют обработчика `SIGTERM` и завершаются мгновенно; но если он есть — это шанс корректно сохраниться.

**На что обращать внимание.** «`-9` сильнее, поэтому надёжнее» — частая ошибка мышления. Хорошее уточнение: «А что произойдёт с базой данных, если её прибить через `-9` во время транзакции?»

### Задача 7 (отладка). Процесс «висит» и не реагирует ни на `-15`, ни на `-9`

**Эталон.** Состояние `D` (uninterruptible sleep) — процесс ждёт ответа от ядра, обычно от устройства ввода-вывода: сломанный диск, отвалившееся NFS-хранилище, повисший USB. `SIGKILL` не помогает, потому что ядро не возвращает управление пользовательскому пространству — там и сигнал не обработать.

Решение:

1. `ps -o pid,stat,wchan,comm -p 5432` — посмотреть, на чём именно процесс висит (поле `WCHAN`);
2. чинить *причину* — отмонтировать сломанный диск, перезагрузить сетевое хранилище;
3. в крайнем случае — `reboot` системы.

**На что обращать внимание.** Студент часто пробует `-9` несколько раз и сдаётся. Глубже: «Если `-9` не работает, значит, ядро держит процесс. Что именно ядру могло понадобиться?»

## Модуль 14. Удалённая работа

### Задача 4 (аналитическая). `rsync` vs `scp` для повторной синхронизации

**Эталон.** При повторной синхронизации большого каталога:

1. **Дельта-передача.** `rsync` копирует только изменившиеся файлы — и внутри файлов передаёт только изменившиеся блоки. `scp` всегда копирует всё целиком.
2. **Возобновление.** `rsync` с `--partial` дописывает прерванную передачу. `scp` начинает заново.
3. **Атрибуты.** `rsync -a` сохраняет права, владельцев, временные метки. `scp -p` — частично; некоторые атрибуты теряются.
4. **Прогресс.** `rsync --progress` информативен; `scp` показывает только скорость текущего файла.

**На что обращать внимание.** «`rsync` быстрее» — упрощение. Точнее: «`rsync` быстрее при повторной передаче того же набора; для первой передачи скорости близкие».

### Задача 7 (отладка). Ключ добавлен, SSH всё ещё просит пароль

**Эталон.** Три причины и три проверки:

1. **Права на `~/.ssh/`.** Каталог должен быть `700`, `authorized_keys` — `600`. Иначе SSH из соображений безопасности игнорирует ключи: `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys`.
2. **Имя файла ключа.** На клиенте SSH ищет `~/.ssh/id_rsa`, `id_ed25519`, `id_ecdsa`. Если ключ имеет другое имя — нужно либо переименовать, либо указать в `~/.ssh/config` через `IdentityFile`.
3. **SELinux/AppArmor.** На RHEL/CentOS SELinux может блокировать доступ к `~/.ssh/`. Проверка: `sudo ausearch -m AVC -ts recent`. Решение: `restorecon -Rv ~/.ssh/`.

Куда смотреть в первую очередь: `ssh -v user@host` (verbose) — клиент покажет, какой ключ предлагает и что отвечает сервер. На сервере: `sudo journalctl -u ssh --since "5 minutes ago"` — там обычно конкретная причина.

**На что обращать внимание.** Хороший признак — студент начинает с `ssh -v`. Плохой — сразу копирует ключ ещё раз «вдруг не дошёл».

## Модуль 15. Git

### Задача 7 (отладка). Случайно закоммитили пароль

**Эталон.** Простого «удалить файл в следующем коммите» недостаточно — старая версия с паролем остаётся в истории. Шаги:

1. **Сменить пароль/токен немедленно.** Это безусловно. Утечка может быть уже произошедшей.
2. **Удалить файл из истории.** Современный инструмент — `git filter-repo`:
   ```shell
   $ git filter-repo --path secret.txt --invert-paths
   ```
   Старый `git filter-branch` тоже работает, но Git его помечает как deprecated.
3. **Принудительно отправить переписанную историю:** `git push --force-with-lease`. Внимание: это переписывает историю у всех; команду нужно скоординировать.
4. **Уведомить команду.** Все, у кого был клон, должны заново клонировать репозиторий или сделать `git fetch && git reset --hard origin/main`.

Если репозиторий уже на GitHub/GitLab публично — пароль считается скомпрометированным, шаг 1 (смена) обязателен.

**На что обращать внимание.** Если студент пропускает шаг 1 («сейчас исправлю историю и норм») — это критическая ошибка мышления. Пароль уже мог быть скачан ботом; единственная гарантия — его смена.

## Модуль 17. Современная сеть

### Задача 4 (аналитическая). `iptables` vs `nftables` vs `ufw`

**Эталон.**

- **Ноутбук под Ubuntu** — `ufw`. Простой синтаксис (`ufw allow 22/tcp`), хватает для базовой защиты, понятен новичку.
- **Стандартный сервер Ubuntu** — `ufw` или `nftables`. Зависит от того, насколько сложные правила нужны. Если простые «открой порты X, Y, Z» — `ufw`; если нужны зоны, NAT, marking — `nftables`.
- **Сложный маршрутизатор** — `nftables`. Современный синтаксис, читаемые наборы правил, атомарные обновления, ставка отрасли. `iptables` всё ещё работает, но для нового кода — не стоит.

**На что обращать внимание.** Студент часто называет `iptables` для всех трёх — это «как в учебнике». Хороший знак — понимает, что `iptables` устарел в роли *основного* инструмента.

### Задача 7 (отладка). Python-сервер работает, `curl localhost:8000` — connection refused

**Эталон.** Три причины:

1. **Сервер слушает только loopback на нужном интерфейсе.** Python `http.server` по умолчанию слушает `0.0.0.0` — но если запущен с `-b 127.0.0.1`, доступен только с локалхоста. Проверка: `ss -tlnp | grep 8000` — если `Local Address` показывает `127.0.0.1:8000`, это причина.
2. **Файрвол.** `ufw status` или `sudo iptables -L -n` — есть ли правило, блокирующее порт 8000?
3. **Не на том сетевом пространстве (если в контейнере).** Если сервер запущен в Docker без `-p 8000:8000`, порт не пробрасывается на хост.

**На что обращать внимание.** Студент должен начинать с `ss -tlnp | grep 8000`. Если этого нет в его плане — значит, не понимает, как проверить, *что вообще слушается*.

## Модуль 18. Контейнеры

### Задача 4 (аналитическая). Image vs container

**Эталон.** Image — неизменяемый шаблон: «снимок» файловой системы и настроек, описывающий, что должно быть внутри. Container — запущенный экземпляр образа, живой процесс с собственным состоянием. Один образ можно запустить много раз — получится много независимых контейнеров. Аналогия: класс и объект в ООП. Файлы образа неизменны; файлы контейнера живут отдельно (в его слое поверх образа) и исчезают вместе с ним, если не подключён том.

**На что обращать внимание.** Студент часто говорит «контейнер — это запущенный образ». Это верно, но мало. Уточнение: «А если я запущу один и тот же образ дважды — у меня будет два контейнера или один и тот же?»

### Задача 6 (проектирование). `Dockerfile` для Python-приложения

**Эталон.**

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000
CMD ["python", "-m", "app"]
```

Ключевые решения: образ `-slim` (меньше); `requirements.txt` копируется отдельно от кода, чтобы `pip install` кэшировался; `--no-cache-dir` не оставляет кэш wheel-файлов в образе; `CMD` в exec-форме `["..."]` не запускает оболочку между Docker и Python.

**Альтернативы.** Multi-stage build (один образ для сборки, другой для запуска) — даёт меньший финальный образ; для базового курса избыточно.

### Задача 7 (отладка). Контейнер запускается и сразу выходит

**Эталон.** Три типичные причины:

1. **Основная команда завершилась.** Контейнер живёт, пока живёт PID 1. Если `CMD` — `echo hello`, контейнер выйдет сразу. Это норма для одноразовых команд (`docker run --rm alpine ls`).
2. **Команда упала.** `docker logs <container>` покажет, что именно она вывела перед выходом. Часто — Python error, отсутствие конфига, неверный путь.
3. **Нет PID 1 в foreground.** Если CMD запускает процесс в фоне (через `&`), оболочка завершается сразу. Решение — запускать процесс в foreground или использовать `tini`/`dumb-init` как PID 1.

**На что обращать внимание.** Студент часто перезапускает контейнер несколько раз, не глядя в логи. Правильный первый шаг — `docker logs`.

## Модуль 19. Диагностика

### Задача 4 (аналитическая). План диагностики «сайт не работает»

**Эталон.** От близкого к далёкому:

1. **Сервис на сервере жив?** `systemctl status nginx` (или другой) — сервис вообще работает?
2. **Порт слушается?** `ss -tlnp | grep :80` — процесс действительно слушает на ожидаемом порту?
3. **Локально страница отдаётся?** `curl -I localhost:80` — отвечает ли nginx прямо с локалхоста?
4. **Сеть наружу?** `curl -I http://<external-ip>:80` — отвечает ли сервис через внешний IP?
5. **DNS?** `dig +short example.com` — указывает ли домен на правильный IP?
6. **HTTPS-сертификат?** Если сайт по HTTPS — `curl -vI https://example.com` покажет проблемы с сертификатом.
7. **Внешний файрвол / провайдер?** Если всё локально работает, но снаружи не доступно — порт закрыт на уровне cloud-провайдера или ISP.

**На что обращать внимание.** Хороший план — идёт по слоям. Плохой — «попробую разные curl». Если у студента нет 5+ шагов — заслуживает дополнительного разбора.

### Задача 5 (аналитическая). Почему «перезагрузить сервис» — плохой первый шаг

**Эталон.** Перезапуск может *временно* скрыть проблему. Это даёт обманчивое ощущение «починилось», и проблема возвращается через час/день/неделю в самый неудобный момент — потому что причина не устранена.

Правильный порядок: сначала *диагностика* (что именно сломалось), потом *понимание* (почему сломалось), и только потом *действие* (перезапуск, правка, починка). Иначе вы не узнаете причину и не сможете её устранить.

**На что обращать внимание.** Студент часто оправдывает «быстрый перезапуск» — «время дорого». Глубже: «если время дороже причины, ты не починил систему, ты её перенастроил, и она снова сломается».

### Задача 7 (отладка). `df -h` показывает 100%, но `du -sh /var/*` даёт меньше

**Эталон.** Три объяснения:

1. **Удалённый файл, открытый процессом.** Если процесс открыл большой файл, потом файл удалили — место не освобождается до закрытия процесса. `du` не видит его (имени нет), а `df` видит занятые блоки. Проверка: `sudo lsof | grep deleted | grep /var`. Решение: перезапустить процесс, удерживающий удалённый файл.
2. **Inode заняты.** Файловая система может быть заполнена *по inode*, а не по объёму. Проверка: `df -i`. Это бывает, когда много мелких файлов (например, `/var/spool/postfix/`). Решение: удалить или переразметить ФС с большим числом inode.
3. **`du` не видит часть данных.** Например, символические ссылки на другие точки монтирования, или скрытые каталоги, к которым у пользователя нет прав. Решение: `sudo du -sh /var/*` — увидеть всё.

**На что обращать внимание.** Студент должен знать про «удалённые открытые файлы» — это самый частый случай и самый коварный. Если он этого не знает — пропуск важной темы.

## Приложение. Регулярные выражения

### Задача 4 (аналитическая). Жадные vs ленивые квантификаторы

**Эталон.** Жадный `.*` захватывает максимально длинный возможный фрагмент. Ленивый `.*?` — минимально короткий. Пример: в строке `<b>bold</b> and <i>italic</i>` шаблон `<.*>` найдёт от первого `<` до *последнего* `>` — то есть всю строку с тегами. `<.*?>` найдёт минимум — отдельный тег `<b>`.

В классических `grep -E` и `sed -E` ленивых квантификаторов нет. Их обходят запретом нежелательного символа в классе: `<[^>]*>` — то же, что ленивая форма, работает везде.

**На что обращать внимание.** «Всегда используй ленивые» — упрощение. Иногда жадные именно нужны: «возьми всё до конца строки».

### Задача 5 (аналитическая). Почему регэкспы — плохо для HTML/JSON

**Эталон.** Регулярные выражения описывают *регулярные языки* — строго ограниченный класс. HTML, XML, JSON — это *контекстно-свободные* языки с произвольной вложенностью. Регулярки не умеют считать парные скобки до произвольной глубины, не умеют запоминать произвольное число открытых тегов. Пример: регэксп не может надёжно отличить `<b>текст</b>` от `<b><i>текст</i></b>`.

Технически — потому, что регулярки реализуются конечными автоматами с ограниченной памятью; парсеры структурированных языков требуют стека (то есть pushdown-автомата).

**На что обращать внимание.** Студент-математик уровня 2 курса должен помнить иерархию Хомского (регулярные → контекстно-свободные → контекстно-зависимые → рекурсивно-перечислимые). Если не помнит — это повод его освежить и заодно понять, *почему* регэкспы не для HTML.

### Задача 7 (отладка). Регулярка работает в Python, не работает в `grep`

**Эталон.** В Python (модуль `re`) используется PCRE-подобный диалект: `\d`, `\s`, `\w`, `{n,m}` работают без экранирования. В классическом `grep` (POSIX BRE) `\d` — это просто буква `d`, `{n,m}` нужно экранировать как `\{n,m\}`. Решения:

1. Использовать `grep -E` (ERE): `{n,m}` работает, но `\d` всё равно не понимается — пишите `[0-9]`.
2. Использовать `grep -P` (Perl-compatible regex): полный PCRE, `\d` работает. Не стандарт, но на GNU grep есть.

```shell
$ grep -P '^\d{3}-\d{3}-\d{4}$' file.txt
$ grep -E '^[0-9]{3}-[0-9]{3}-[0-9]{4}$' file.txt
```

**На что обращать внимание.** Хороший студент понимает, что «регулярка» — это не один язык, а семейство диалектов. Плохой — пишет одно и то же синтаксис везде и удивляется, почему не работает.
