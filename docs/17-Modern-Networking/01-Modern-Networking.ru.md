# Современная сеть в Linux

Раздел продолжает тему сетей из [модуля 07. Сети](../07-Networking) и фокусируется на современном инструментарии. В модуле 07 мы пользовались набором команд из пакета `net-tools` — `ifconfig`, `route`, `netstat`. На современных дистрибутивах их часто нет «из коробки», а вместо них стоит пакет `iproute2`: команды `ip` и `ss`. Здесь мы разберём, что эти команды дают вместо устаревших, как настроить постоянную сеть через Netplan (отдельный раздел этого модуля) и какие у современного Linux есть варианты межсетевого экрана.

## Цели раздела

После раздела студент должен уметь:

- смотреть IP-адреса интерфейсов;
- смотреть маршруты;
- проверять открытые порты и соединения;
- тестировать HTTP-сервис через `curl`;
- понимать место `iptables`, `nftables`, `ufw` и `firewalld`.

## IP-адреса

Первый вопрос при разборе сетевой проблемы — «а какой у машины адрес и работает ли вообще интерфейс». Отвечает на него `ip addr` (есть и короткая форма `ip a`). Команда перечисляет все сетевые интерфейсы и присвоенные им адреса:

```shell
$ ip addr
$ ip a
```

Если интерфейсов много, вывод можно сузить до одного — например, посмотреть только проводной `eth0`:

```shell
$ ip addr show eth0
```

## Маршруты

Адрес у машины есть — но как пакеты находят дорогу в другие сети? За это отвечает таблица маршрутизации, и показывает её `ip route`:

```shell
$ ip route
```

В выводе ищите строку с `default` — это маршрут по умолчанию. Он указывает шлюз (gateway): адрес, на который система отправляет всё, для чего нет более конкретного маршрута. Нет маршрута по умолчанию — машина не выйдет в интернет, даже если локальная сеть работает.

## Соединения и порты

Чтобы узнать, какие порты на машине открыты и кто к ней подключён, служит `ss` (socket statistics) — современная замена старого `netstat`:

```shell
$ ss -tuln
$ ss -tunap
```

Флаги `ss` удобно комбинировать, и каждый сужает картину:

- `-t` — показывать TCP-соединения;
- `-u` — показывать UDP;
- `-l` — только слушающие сокеты, то есть порты, которые чего-то ждут (именно это нужно для проверки «запустился ли сервер»);
- `-n` — не преобразовывать номера в имена, показывать «как есть» — так быстрее и нагляднее;
- `-p` — показать, какой процесс держит сокет.

Связку `ss -tuln` стоит запомнить как ответ на вопрос «какие порты сейчас слушаются».

> **Типичные ошибки.**
>
> - **`ss -tuln` ничего не показывает — а сервис должен слушать.** Часто сервис привязан только к `127.0.0.1`, и снаружи к нему действительно не достучаться. В `ss -tuln` это видно по столбцу `Local Address:Port`: `127.0.0.1:8080` против `0.0.0.0:8080` или `[::]:8080`. Нужно править конфиг сервиса, а не firewall.
> - **`ss -p` без `sudo` молчит.** Чтобы увидеть, какой процесс держит сокет, ядру нужны привилегии. Без `sudo` ss покажет порты, но столбец «процесс» останется пустым.
> - **`ip addr` показывает интерфейс «DOWN».** Адрес может быть прописан, но интерфейс не поднят. Нужно `sudo ip link set eth0 up`, и проверить состояние снова. На постоянной конфигурации эту проблему решают через Netplan (см. следующий раздел).
> - **Маршрут по умолчанию есть, а наружу не проходит.** `ip route` показывает `default via ...`, `ping шлюза` отвечает, а `ping 8.8.8.8` — нет. Обычно дело в firewall или в NAT провайдера; следующий шаг — `traceroute 8.8.8.8` и проверка правил `iptables -L` / `nft list ruleset`.

## Проверка сервиса

Когда сетевой сервис запущен, его нужно проверить — и для этого почти всегда берут `curl`. Самый простой вызов отправляет HTTP-запрос и печатает ответ:

```shell
$ curl http://localhost:8000
$ curl -I https://example.com
$ curl -v http://localhost:8000
```

Два флага меняют поведение под разные задачи. `curl -I` запрашивает только заголовки ответа — удобно быстро проверить, жив ли сервис и какой код он возвращает, не выкачивая всю страницу. А `curl -v` (verbose) показывает каждый шаг соединения — установку TCP, отправленные и полученные заголовки; это главный инструмент, когда запрос не работает и непонятно почему.

## Разобранный пример: «приложение не отвечает снаружи»

**Задача.** На учебном сервере запустили веб-приложение, которое по словам разработчика «слушает порт 8080». С самого сервера `curl http://localhost:8080` отвечает, а с вашего ноутбука `curl http://<ip-сервера>:8080` зависает. Найти, на каком слое проблема.

**Решение.**

1. Проверить, что сервис вообще слушает — и на каком адресе. `127.0.0.1:8080` означает «только локально», `0.0.0.0:8080` или `[::]:8080` — «на всех интерфейсах»:
   ```shell
   $ ss -tuln | grep 8080
   ```
2. Если в `Local Address` стоит `127.0.0.1` — это и есть ответ: сервис нужно перенастроить, чтобы слушал на внешнем интерфейсе. Если стоит `0.0.0.0` — идём дальше.
3. Посмотреть, какой процесс держит порт (`-p` требует `sudo`):
   ```shell
   $ sudo ss -tulnp | grep 8080
   ```
4. Проверить маршруты и наличие адреса на внешнем интерфейсе — клиент должен «доходить» до сервера:
   ```shell
   $ ip addr show
   $ ip route
   ```
5. С ноутбука пингануть IP сервера. Если `ping` идёт — сеть в порядке, проблема в портах или firewall:
   ```shell
   $ ping <ip-сервера>
   ```
6. Проверить firewall на сервере. Часто `ufw` или `iptables` режут всё, кроме `22`:
   ```shell
   $ sudo ufw status
   $ sudo iptables -L INPUT -n
   ```
7. Подробный запрос с ноутбука — `curl -v` покажет, на каком шаге всё встаёт: TCP не открывается (порт закрыт firewall), открывается, но HTTP не отвечает (сервис слушает не там), или возвращает ошибку (сервис жив, но падает на запросе):
   ```shell
   $ curl -v http://<ip-сервера>:8080
   ```

**Проверка.** Вы должны уметь назвать слой, на котором проблема: bind-адрес сервиса, маршрутизация, firewall или сам сервис. В отчёт идёт *один* конкретный шаг, который её устранит.

## Межсетевой экран: краткий контекст

То, какие соединения вообще разрешены, решает не сам сервис, а межсетевой экран (firewall) — он управляет тем, какой сетевой трафик пропускать, а какой блокировать. В Linux вы встретите несколько инструментов для работы с ним, и важно понимать их соотношение:

- `iptables` — исторический инструмент, всё ещё встречается на многих серверах;
- `nftables` — современная подсистема фильтрации пакетов, пришедшая на смену `iptables`;
- `ufw` (uncomplicated firewall) — упрощённая надстройка, типичная для Ubuntu;
- `firewalld` — распространён в RHEL, Fedora и CentOS.

Глубоко настраивать firewall в рамках базового курса не нужно — достаточно усвоить саму идею и узнавать эти названия, встретив их в чужих инструкциях.

## Практическое задание

1. Выполните `ip addr` и найдите активный интерфейс.
2. Выполните `ip route` и найдите маршрут по умолчанию.
3. Запустите простой HTTP-сервер: `python3 -m http.server 8000` (если `python3` не установлен, поставьте: `sudo apt install python3`).
4. В другом терминале проверьте его через `curl http://localhost:8000`.
5. Найдите слушающий порт `8000` через `ss -tuln`.
