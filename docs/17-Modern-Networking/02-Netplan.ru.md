# Постоянная сеть: Netplan и NetworkManager

Команды `ip addr add` и `ip route add` из модулей 07 и 17 удобны для экспериментов: вы добавили адрес, проверили, что всё работает, — и забыли. Но эти изменения живут только до перезагрузки. Чтобы сеть поднималась автоматически каждый раз при старте машины, её конфигурацию записывают в файлы. В современных Ubuntu (с версии 17.10) и Debian (с 12) этим занимается Netplan; в Ubuntu Desktop и в большинстве дистрибутивов с графической средой — NetworkManager. Разберём, как устроены оба и когда какой использовать.

## Цели раздела

После раздела студент должен уметь:

- читать YAML-конфигурацию Netplan;
- настраивать статический IP-адрес и маршруты через Netplan;
- применять и проверять конфигурацию командой `netplan`;
- понимать роль NetworkManager в сравнении с Netplan;
- управлять соединениями через `nmcli`.

## Зачем нужна постоянная конфигурация

Когда сервер загружается, в его сетевых интерфейсах ничего нет. Чтобы они получили адреса и подключились к сети, при загрузке должна сработать какая-то служба, которая возьмёт конфигурацию из файла и применит её через те же системные вызовы, что и `ip addr add`. Раньше эту роль играли `/etc/network/interfaces` (Debian), скрипты `ifup`/`ifdown`, а в RHEL — `/etc/sysconfig/network-scripts/`. В современных системах их сменили два инструмента:

- **Netplan** — YAML-описание сети в `/etc/netplan/*.yaml`. Сам ничего не настраивает: при `netplan apply` он генерирует конфигурацию для нижележащего бэкенда (backend, «реальный исполнитель» — `systemd-networkd` или `NetworkManager`) и просит тот применить её. По умолчанию в Ubuntu Server бэкенд — `networkd`, в Ubuntu Desktop — `NetworkManager`.
- **NetworkManager (NM)** — служба, активно управляющая соединениями: знает про Wi-Fi, VPN, мобильные сети, переключения между ними. Хранит профили в `/etc/NetworkManager/system-connections/`. Управляется командой `nmcli` (CLI) или графической оболочкой.

Грубо говоря, Netplan — это «опиши, какая должна быть сеть, я применю», а NetworkManager — это «я слежу за сетью в реальном времени, помогу с переключениями и Wi-Fi». На серверах чаще встречается первый, на ноутбуках — второй.

## Netplan: статический IP

Файлы Netplan лежат в `/etc/netplan/` и имеют расширение `.yaml`. Минимальная конфигурация для одного интерфейса с DHCP:

```yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s3:
      dhcp4: true
```

Эта конфигурация говорит: «версия формата 2, бэкенд — `networkd`, интерфейс `enp0s3` получает адрес по DHCP». Поле `renderer` обычно ставят `networkd` для серверов и `NetworkManager` для рабочих станций.

Статический адрес выглядит чуть подробнее:

```yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s3:
      dhcp4: no
      addresses:
        - 192.168.1.50/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1
```

Разбор:

- `dhcp4: no` — отказываемся от DHCP;
- `addresses` — список IPv4-адресов на интерфейсе. Запись `192.168.1.50/24` сразу несёт и сам адрес, и маску подсети;
- `routes` — маршруты. `to: default` — это «маршрут по умолчанию», `via: 192.168.1.1` — шлюз;
- `nameservers.addresses` — DNS-серверы, к которым обращается система за разрешением имён.

## Применение и проверка

После правки YAML-файла Netplan должен «собрать» из него реальную конфигурацию и применить:

```shell
$ sudo netplan apply
```

Если конфигурация некорректна, команда ругнётся ещё до применения и не сломает сеть. Но при работе с удалённой машиной (особенно по SSH) опаснее другое: правильный YAML может тем не менее оборвать ваше соединение, если, например, новый адрес не совпадает с подсетью. Для таких случаев есть страховка — `netplan try`:

```shell
$ sudo netplan try
```

Она применяет конфигурацию на пробу. Если в течение двух минут вы не подтвердите её клавишей Enter, Netplan автоматически откатит изменения. То есть SSH не оборвётся «навсегда»: даже если вы напортачили, через пару минут сеть вернётся к прежним настройкам, и вы сможете подключиться снова и поправить файл.

Проверить, что новые настройки применились:

```shell
$ ip addr show enp0s3
$ ip route
$ resolvectl status
```

`resolvectl` (часть systemd-resolved) покажет, какие DNS-серверы и зоны сейчас активны.

## Типичные ошибки

- **Отступы YAML.** YAML принимает только пробелы — табуляция приведёт к ошибке парсинга. Стандарт — два пробела на уровень.
- **Несколько файлов в `/etc/netplan/`.** Netplan сливает все `.yaml`-файлы каталога в порядке имени. Если в системе один файл задаёт DHCP, а другой — статический адрес, итог зависит от порядка имён. На сервере держите один файл и понимайте, какой именно.
- **Забытое `routes/default`.** Адрес есть, маска подходит, но интернет не работает: проверьте, есть ли маршрут по умолчанию.
- **`netplan apply` по SSH без `try`.** Один неправильный YAML — и сервер недоступен до следующего физического визита. Привычка делать `netplan try` экономит часы.

## NetworkManager: `nmcli`

На рабочей станции вы обычно работаете с NetworkManager через значок в трее. На сервере с NetworkManager — через `nmcli`. Самые частые команды:

```shell
$ nmcli device              # список интерфейсов и их статус
$ nmcli connection show     # список профилей соединений
$ nmcli connection up wifi-home   # активировать профиль
$ nmcli connection modify wifi-home ipv4.method manual ipv4.addresses 192.168.1.50/24
$ nmcli connection up wifi-home   # перезапустить с новыми настройками
```

`nmcli` многословен, но прозрачен: каждое поле имеет своё имя (`ipv4.method`, `ipv4.gateway`, `802-11-wireless.ssid`), и список можно посмотреть командой `nmcli connection show <имя>`.

Когда Netplan настроен с `renderer: NetworkManager`, профили из его YAML-файлов появляются в `nmcli connection show` — это удобный способ объединить «декларативность Netplan» и «гибкость NM».

## Когда что использовать

- **Серверы Ubuntu/Debian без графики** — Netplan + `networkd`. Декларативный YAML, проще аудит, меньше движущихся частей.
- **Ubuntu Desktop, Fedora, Mint** — NetworkManager. Графическая оболочка, Wi-Fi, VPN, быстрые переключения.
- **Серверы с Wi-Fi или мобильным интернетом** (нетипично, но бывает) — Netplan + `NetworkManager` или NM напрямую: его модель «соединение» лучше ложится на радиосеть, чем «постоянный интерфейс» в `networkd`.
- **Старые системы (Ubuntu 16.04, CentOS 7)** — старая модель `/etc/network/interfaces` или `/etc/sysconfig/network-scripts/`. Здесь Netplan не работает.

## Разобранный пример: перевод сервера со статикой

**Задача.** На сервере Ubuntu Server 22.04 интерфейс `enp1s0` получает адрес по DHCP. Нужно перевести его на постоянный `10.0.0.50/24` со шлюзом `10.0.0.1` и DNS `1.1.1.1`, не сломав SSH.

**Решение.**

1. Зайдите по SSH на сервер. Узнайте имя интерфейса:
   ```shell
   $ ip -br link
   ```
2. Откройте файл Netplan на правку:
   ```shell
   $ sudo nano /etc/netplan/00-installer-config.yaml
   ```
3. Замените содержимое:
   ```yaml
   network:
     version: 2
     renderer: networkd
     ethernets:
       enp1s0:
         dhcp4: no
         addresses: [10.0.0.50/24]
         routes:
           - to: default
             via: 10.0.0.1
         nameservers:
           addresses: [1.1.1.1, 8.8.8.8]
   ```
4. Примените на пробу с откатом по таймауту:
   ```shell
   $ sudo netplan try
   ```
5. Если SSH-сессия не отвалилась — подтвердите Enter. Если отвалилась — Netplan откатится сам, переподключитесь и поправьте файл.
6. Проверьте результат:
   ```shell
   $ ip addr show enp1s0
   $ ip route
   ```

**Проверка.** `ip addr` показывает `10.0.0.50/24`, `ip route` — строку `default via 10.0.0.1 dev enp1s0`. `ping 1.1.1.1` отвечает; `ping example.com` тоже отвечает (значит, DNS тоже работает).
