# Контейнеры в Linux

Все темы курса до этого момента — пользователи, права, процессы, сеть, сервисы — мы изучали на «единой системе». Но реальная работа последних лет всё чаще выглядит иначе: на одном сервере крутится десяток приложений, и каждое — в своём изолированном окружении. Эти изоляции и называют контейнерами. Контейнер позволяет запустить процесс так, чтобы он не видел соседей по серверу, имел собственную файловую систему и сеть, но при этом не превращался в полноценную виртуальную машину со своим ядром. Для учебных целей контейнеры удобны вдвойне: они дают воспроизводимое окружение, которое легко уничтожить и поднять заново, не повредив основную систему.

## Цели раздела

После раздела студент должен уметь:

- объяснить, чем контейнер отличается от виртуальной машины;
- понимать базовые идеи namespaces и cgroups;
- запускать контейнер;
- подключать каталог через bind mount;
- пробрасывать порт;
- смотреть логи и заходить внутрь контейнера.

## Идея контейнера

Вопреки расхожему образу «лёгкой виртуальной машины», контейнер — это не отдельная система. Это обычный процесс Linux, вокруг которого ядро выстроило иллюзию: процесс ведёт себя так, будто работает в своей системе один.

Эту иллюзию создают два механизма ядра, и на обзорном уровне достаточно понимать их роли:

- namespaces (пространства имён) отвечают за изоляцию видимости: процесс в контейнере видит только свои процессы, свою сеть, свою файловую систему, своих пользователей — остального для него как бы не существует;
- cgroups (control groups) отвечают за изоляцию потребления: они ограничивают и учитывают ресурсы — сколько контейнеру можно занять CPU, памяти, ввода-вывода.

Именно поэтому контейнер легче виртуальной машины: у него нет собственного ядра и собственной загрузки, он пользуется ядром хостовой системы. Запуск контейнера — это, по сути, запуск процесса, а не старт целой ОС.

## Образы и контейнеры

Два слова, которые легко спутать, но различать их важно. Образ (image) — это неизменяемый шаблон: файловая система и настройки, «слепок» того, что должно быть внутри. Контейнер (container) — это запущенный экземпляр образа, живой процесс. Один образ можно запустить много раз и получить много независимых контейнеров — примерно как класс и объекты в программировании.

Сначала образ нужно получить — скачать из реестра:

```shell
$ docker pull ubuntu:22.04
```

Запуск контейнера — это `docker run`: укажите образ и команду, которую внутри выполнить. Флаг `--rm` говорит «удали контейнер сразу после завершения», чтобы не копить мусор:

```shell
$ docker run --rm ubuntu:22.04 echo "hello"
```

Чаще, однако, нужен не разовый запуск команды, а интерактивная работа внутри контейнера. Флаги `-it` (интерактивный режим с терминалом) дают вам приглашение командной строки прямо внутри:

```shell
$ docker run --rm -it ubuntu:22.04 bash
```

Docker — не единственный движок. Podman — популярная альтернатива, и базовые команды у него те же; в большинстве примеров достаточно заменить одно слово:

```shell
$ podman run --rm -it ubuntu:22.04 bash
```

## Bind mount

По умолчанию файловая система контейнера изолирована и исчезает вместе с ним. Но часто нужно, чтобы контейнер работал с вашими файлами на хосте — например, прогонял обработку данных из текущего каталога. Для этого служит bind mount: вы «прокидываете» каталог хоста внутрь контейнера флагом `-v`, а `-w` сразу делает его рабочим:

```shell
$ docker run --rm -it -v "$PWD":/work -w /work ubuntu:22.04 bash
```

Теперь `/work` внутри контейнера — это и есть ваш текущий каталог: файлы, созданные там изнутри, останутся на хосте и после остановки контейнера.

## Порты

Сеть контейнера тоже изолирована, поэтому сервис, запущенный внутри, снаружи по умолчанию недоступен. Чтобы до него достучаться, порт нужно явно пробросить флагом `-p`:

```shell
$ docker run --rm -p 8080:80 nginx
```

Запись `8080:80` читается как «порт на хосте : порт в контейнере»: обращения к `8080` на вашей машине перенаправляются на `80` внутри контейнера, где слушает nginx.

## Логи и вход в контейнер

Когда контейнеры уже работают, ими нужно управлять. `docker ps` показывает список запущенных контейнеров — с их идентификаторами и именами:

```shell
$ docker ps
```

Если сервис в контейнере ведёт себя не так, первым делом смотрят его вывод — `docker logs` показывает всё, что контейнер напечатал в stdout и stderr:

```shell
$ docker logs <container>
```

А когда логов недостаточно и нужно осмотреться изнутри, `docker exec` запускает дополнительный процесс — например, оболочку — внутри уже работающего контейнера:

```shell
$ docker exec -it <container> bash
```

## Свой образ: Dockerfile

До сих пор мы пользовались готовыми образами с Docker Hub. Но рано или поздно появляется задача: «упаковать в контейнер мой собственный проект». Для этого есть Dockerfile — текстовый файл с инструкциями, по которому Docker собирает образ.

Минимальный Dockerfile для веб-приложения на Python:

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["python", "-m", "app"]
```

Разберём по строкам:

- `FROM` — базовый образ, на котором всё строится. `python:3.12-slim` — официальный Python 3.12 в облегчённой версии Debian.
- `WORKDIR` — рабочий каталог внутри образа. Все последующие команды выполняются относительно него.
- `COPY src dst` — копирует файлы с хоста (из контекста сборки) в образ. Сначала копируем только `requirements.txt` и устанавливаем зависимости — это позволяет Docker кэшировать слой и не пересобирать его, пока файл зависимостей не меняется.
- `RUN команда` — выполняет команду на этапе сборки. Каждый `RUN` создаёт отдельный слой в образе.
- `COPY . .` — копирует весь остальной код приложения.
- `EXPOSE 8000` — декларация: «контейнер слушает порт 8000». На сам проброс наружу не влияет (нужен `-p` при `docker run`), но документирует намерение.
- `CMD ["...", "..."]` — команда, которая запустится при `docker run`, если не указано иное. Форма с массивом — рекомендованная: она не запускает оболочку между Docker и вашей программой.

Собрать образ из этого файла:

```shell
$ docker build -t my-app:dev .
```

Точка в конце — это «контекст сборки»: каталог, файлы из которого Docker отправит демону для сборки. Опция `-t` (tag) даёт образу имя и версию.

Запустить только что собранный образ:

```shell
$ docker run --rm -p 8000:8000 my-app:dev
```

### Типичные ошибки в Dockerfile

- **`COPY . .` в самом начале.** При любой правке кода Docker заново выполнит все следующие шаги, включая `pip install`. Сначала копируйте файл зависимостей и устанавливайте их, потом — весь остальной код. Кэш будет работать.
- **Запуск от root.** По умолчанию процесс в контейнере работает под root. Для боевых приложений добавляют `RUN useradd -r app && USER app`, чтобы при компрометации контейнера злоумышленник не получил root внутри.
- **Тяжёлый базовый образ.** `python:3.12` без `-slim` весит примерно 1 ГБ. Версия `-slim` — меньше 200 МБ; `-alpine` — около 60 МБ (но в alpine нет glibc, что иногда ломает зависимости).
- **`RUN` без `--no-cache-dir`.** `pip` оставляет в образе кэш скачанных wheel-файлов (упакованных Python-пакетов формата wheel) — это +50–200 МБ ненужного веса.

Дальнейшее изучение — Docker Compose (для запуска нескольких контейнеров вместе), реестры (хранение и распространение собственных образов), build-кэш. Это уже за рамками базового курса, но Dockerfile из этого раздела — то, с чего реально начинается «доставка приложения через Docker».

## Разобранный пример: упаковать учебное Flask-приложение

**Задача.** В каталоге `~/work/hello-web/` лежат два файла: `app.py` (минимальное Flask-приложение, слушает на `0.0.0.0:8000`) и `requirements.txt` со строкой `flask`. Нужно собрать Docker-образ, запустить его так, чтобы приложение было доступно на `http://localhost:8080`, и убедиться, что код в образ попал, а кэш сборки используется при повторном запуске.

**Решение.**

1. Создать `Dockerfile` рядом с кодом. Порядок строк — не косметика: сначала зависимости, потом код. Тогда при правке `app.py` Docker не будет переустанавливать Flask заново:
   ```dockerfile
   FROM python:3.12-slim
   WORKDIR /app
   COPY requirements.txt ./
   RUN pip install --no-cache-dir -r requirements.txt
   COPY . .
   EXPOSE 8000
   CMD ["python", "app.py"]
   ```
2. Добавить `.dockerignore` — иначе в контекст сборки уедут `__pycache__/`, `.venv/` и `.git/`:
   ```text
   .venv/
   __pycache__/
   .git/
   *.pyc
   ```
3. Собрать образ. Точка в конце — это контекст сборки (текущий каталог); `-t` задаёт имя и тег:
   ```shell
   $ docker build -t hello-web:dev .
   ```
4. Запустить контейнер с пробросом порта. Запись `8080:8000` читается «8080 на хосте → 8000 в контейнере»:
   ```shell
   $ docker run --rm -p 8080:8000 hello-web:dev
   ```
5. В отдельном терминале проверить, что приложение отвечает:
   ```shell
   $ curl http://localhost:8080
   ```
6. Поменять одну строку в `app.py` и пересобрать. Заметьте, какие шаги Docker возьмёт из кэша (`CACHED`), а какие выполнит заново — должен пересобраться только последний `COPY . .` и слои после него:
   ```shell
   $ docker build -t hello-web:dev .
   ```

**Проверка.** `docker images | grep hello-web` показывает собранный образ. `curl http://localhost:8080` возвращает HTML вашего приложения. При повторной сборке после правки `app.py` шаг `RUN pip install` отмечен как `CACHED` — порядок инструкций в Dockerfile работает.

4. Создайте файл внутри `/work` и проверьте, появился ли он на хосте.
5. Запустите `nginx` с пробросом порта.
6. Проверьте сервис через `curl localhost:8080`.
