# Контейнеры в Linux

Контейнеры позволяют запускать процессы в изолированном окружении. Для студентов это полезно как способ воспроизводимо запускать сервисы, инструменты и вычислительные окружения без изменения всей системы.

## Цели раздела

После раздела студент должен уметь:

- объяснить, чем контейнер отличается от виртуальной машины;
- понимать базовые идеи namespaces и cgroups;
- запускать контейнер;
- подключать каталог через bind mount;
- пробрасывать порт;
- смотреть логи и заходить внутрь контейнера.

## Идея контейнера

Контейнер — это обычный процесс Linux, которому система создаёт изолированное окружение.

На обзорном уровне важны два механизма:

- namespaces изолируют видимость ресурсов: процессы, сеть, файловую систему, пользователей;
- cgroups ограничивают и учитывают ресурсы: CPU, память, ввод-вывод.

Контейнер обычно легче виртуальной машины, потому что использует ядро хостовой системы.

## Образы и контейнеры

Образ (image) — шаблон файловой системы и настроек.

Контейнер (container) — запущенный экземпляр образа.

Скачать образ:

```shell
$ docker pull ubuntu:22.04
```

Запустить контейнер:

```shell
$ docker run --rm ubuntu:22.04 echo "hello"
```

Интерактивный запуск:

```shell
$ docker run --rm -it ubuntu:22.04 bash
```

Если вместо Docker используется Podman, базовые команды похожи:

```shell
$ podman run --rm -it ubuntu:22.04 bash
```

## Bind mount

Подключить текущий каталог внутрь контейнера:

```shell
$ docker run --rm -it -v "$PWD":/work -w /work ubuntu:22.04 bash
```

Теперь файлы текущего каталога доступны внутри контейнера в `/work`.

## Порты

Пробросить порт:

```shell
$ docker run --rm -p 8080:80 nginx
```

Здесь порт `8080` на хосте перенаправляется на порт `80` в контейнере.

## Логи и вход в контейнер

Список контейнеров:

```shell
$ docker ps
```

Логи:

```shell
$ docker logs <container>
```

Запустить shell внутри работающего контейнера:

```shell
$ docker exec -it <container> bash
```

## Практическое задание

1. Запустите контейнер Ubuntu интерактивно.
2. Выполните внутри `cat /etc/os-release`.
3. Подключите текущий каталог как `/work`.
4. Создайте файл внутри `/work` и проверьте, появился ли он на хосте.
5. Запустите `nginx` с пробросом порта.
6. Проверьте сервис через `curl localhost:8080`.
