# Удалённая работа

Большая часть реальной работы с Linux происходит на удалённых серверах: учебных машинах, вычислительных узлах, облачных VM. Поэтому студент должен уверенно подключаться по SSH, передавать файлы и запускать задачи так, чтобы они продолжали работать после отключения.

## Цели раздела

После раздела студент должен уметь:

- подключаться к серверу по SSH;
- создавать и использовать SSH-ключи;
- настраивать `~/.ssh/config`;
- копировать файлы через `scp` и `rsync`;
- запускать долгие задачи на удалённой машине.

## Подключение по SSH

```shell
$ ssh user@example.com
$ ssh user@192.168.1.10
```

Если сервер использует нестандартный порт:

```shell
$ ssh -p 2222 user@example.com
```

## SSH-ключи

Создать ключ:

```shell
$ ssh-keygen -t ed25519 -C "student@example.com"
```

Скопировать публичный ключ на сервер:

```shell
$ ssh-copy-id user@example.com
```

Публичный ключ можно передавать, приватный ключ нельзя показывать и отправлять.

## `~/.ssh/config`

Чтобы не вводить длинную команду каждый раз, можно создать конфигурацию:

```text
Host lab
  HostName example.com
  User student
  Port 22
  IdentityFile ~/.ssh/id_ed25519
```

После этого подключение выглядит так:

```shell
$ ssh lab
```

## Передача файлов

Скопировать файл на сервер:

```shell
$ scp report.txt user@example.com:/home/user/
```

Скопировать файл с сервера:

```shell
$ scp user@example.com:/home/user/result.txt .
```

`rsync` удобен для каталогов и повторной синхронизации:

```shell
$ rsync -av data/ user@example.com:/home/user/data/
$ rsync -av user@example.com:/home/user/results/ results/
```

## Долгая задача на сервере

```shell
$ ssh lab
$ tmux
$ ./run-experiment.sh > out.log 2> err.log
```

Отключиться от `tmux`: `Ctrl+b`, затем `d`.

Проверить позже:

```shell
$ ssh lab
$ tmux attach
```

## Практическое задание

1. Создайте SSH-ключ.
2. Настройте подключение через `~/.ssh/config`.
3. Скопируйте каталог с данными на сервер через `rsync`.
4. Запустите задачу в `tmux`.
5. Отключитесь и подключитесь снова.
6. Заберите результаты обратно через `rsync`.

## Контрольные вопросы

1. Чем публичный SSH-ключ отличается от приватного?
2. Для чего нужен `~/.ssh/config`?
3. Когда удобнее `rsync`, а не `scp`?
4. Почему долгую задачу лучше запускать в `tmux`?
