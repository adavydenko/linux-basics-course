# Удалённая работа

С SSH вы уже знакомы по [модулю 06](../06-Security-and-File-Permissions/06-SSH-and-SCP.ru.md): мы видели, как подключаться, как генерировать ключи `ed25519` и копировать файлы через `scp`. Здесь мы доходим до повседневного использования удалённого доступа: `~/.ssh/config` для коротких имён, `rsync` для синхронизации каталогов, `tmux` для долгих сессий, переживающих обрыв связи. Большая часть реальной работы с Linux происходит на удалённых серверах — учебных машинах, вычислительных узлах, облачных VM, — и без этого набора инструментов любой выход в дорогу превращается в потерю прогресса.

## Цели раздела

После раздела студент должен уметь:

- подключаться к серверу по SSH;
- создавать и использовать SSH-ключи;
- настраивать `~/.ssh/config`;
- копировать файлы через `scp` и `rsync`;
- запускать долгие задачи на удалённой машине.

## Подключение по SSH

Основной способ попасть на удалённую машину — команда `ssh` (Secure Shell). Ей нужно сказать, под каким пользователем и на какой адрес подключаться, в формате `пользователь@хост`. Адрес может быть как доменным именем, так и IP:

```shell
$ ssh user@example.com
$ ssh user@192.168.1.10
```

По умолчанию SSH стучится на порт 22. Если администратор сервера сменил порт, его указывают флагом `-p`:

```shell
$ ssh -p 2222 user@example.com
```

## SSH-ключи

Вводить пароль при каждом подключении неудобно и небезопасно. Альтернатива — ключи: пара из приватного и публичного ключа. Создаёт её `ssh-keygen`; тип `ed25519` — современный и рекомендуемый, а комментарий после `-C` просто помогает потом опознать ключ:

```shell
$ ssh-keygen -t ed25519 -C "student@example.com"
```

Чтобы сервер начал пускать вас по ключу, на него нужно положить публичную часть пары. Это автоматизирует `ssh-copy-id`:

```shell
$ ssh-copy-id user@example.com
```

Запомните железное правило: публичный ключ можно отдавать кому угодно и копировать на любые серверы, а приватный ключ не покидает вашу машину — его нельзя показывать, пересылать или коммитить в репозиторий. Скомпрометированный приватный ключ — это как отданный пароль от всех серверов сразу.

> **Типичные ошибки.**
>
> - **Слишком открытые права на приватный ключ.** Если `~/.ssh/id_ed25519` доступен для чтения группе или всем (`chmod 644`), SSH откажется его использовать с предупреждением «UNPROTECTED PRIVATE KEY FILE». Правильно — `chmod 600 ~/.ssh/id_ed25519`.
> - **Скопировали приватный ключ, а не публичный.** Файл без расширения (`id_ed25519`) — *приватный*; рядом лежит `id_ed25519.pub` — *публичный*. На сервер кладётся именно `.pub`. Перепутали — приходится сначала отзывать ключ, а потом разбираться, кому он мог утечь.
> - **`ssh-copy-id` не помог.** Либо на сервере запрещён вход по паролю, и тогда ключ нужно класть руками через консоль провайдера, либо `~/.ssh/authorized_keys` существует, но с правами `666` — `sshd` отказывается читать «слишком открытый» файл. Решение: `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys` на сервере.
> - **Один ключ на десятке серверов.** Технически работает, но утечка одного файла открывает все серверы сразу. Удобнее держать пару ключей под разные роли (учебные, рабочие) и описывать их в `~/.ssh/config` через `IdentityFile`.

## `~/.ssh/config`

Когда серверов становится несколько, набирать каждый раз длинную строку с пользователем, портом и путём к ключу утомительно. Файл `~/.ssh/config` позволяет описать подключение один раз и дать ему короткое имя:

```text
Host lab
  HostName example.com
  User student
  Port 22
  IdentityFile ~/.ssh/id_ed25519
```

Теперь все эти параметры скрыты за псевдонимом `lab`, и подключение сокращается до:

```shell
$ ssh lab
```

## Передача файлов

Подключиться к серверу мало — обычно туда нужно отправить данные, а обратно забрать результаты. Для разовой передачи отдельных файлов есть `scp` (secure copy). Работает он как `cp`, только одна из сторон — удалённая, и записывается через двоеточие:

```shell
$ scp report.txt user@example.com:/home/user/
$ scp user@example.com:/home/user/result.txt .
```

Для каталогов и тем более для повторной синхронизации лучше подходит `rsync`. Его главное преимущество — он передаёт только то, что изменилось, поэтому второй и последующие запуски проходят быстро. Флаги `-av` означают «архивный режим» (сохранить права, время, структуру) и «подробный вывод»:

```shell
$ rsync -av data/ user@example.com:/home/user/data/
$ rsync -av user@example.com:/home/user/results/ results/
```

## Разобранный пример: рабочий цикл «правка на ноуте — запуск на сервере»

**Задача.** У вас есть каталог `~/work/experiment/` на ноутбуке. Нужно настроить так, чтобы один короткой командой код уезжал на учебный сервер `lab`, там запускался расчёт в `tmux`, а после — результаты возвращались обратно.

**Решение.**

1. Создать SSH-ключ и положить публичную часть на сервер:
   ```shell
   $ ssh-keygen -t ed25519 -C "student@laptop"
   $ ssh-copy-id user@example.com
   ```
2. Добавить псевдоним в `~/.ssh/config`, чтобы дальше работать с коротким именем `lab`:
   ```text
   Host lab
     HostName example.com
     User student
     IdentityFile ~/.ssh/id_ed25519
   ```
3. Отправить код на сервер через `rsync`. Завершающий слэш у `experiment/` важен: с ним копируется *содержимое* каталога, без него — сам каталог:
   ```shell
   $ rsync -av --delete ~/work/experiment/ lab:/home/student/experiment/
   ```
   Флаг `--delete` удаляет на сервере файлы, которых уже нет у вас, — иначе старые «черновики» будут копиться.
4. Зайти на сервер и запустить расчёт в `tmux`, чтобы он пережил обрыв SSH:
   ```shell
   $ ssh lab
   $ cd ~/experiment
   $ tmux new -s run
   $ ./run.sh > out.log 2> err.log
   ```
   Отключиться от сессии: `Ctrl+b`, затем `d` — расчёт продолжит работать.
5. Через час забрать результаты на ноутбук. Та же `rsync`, но в обратную сторону:
   ```shell
   $ rsync -av lab:/home/student/experiment/results/ ~/work/experiment/results/
   ```

**Проверка.** После шага 3 на сервере появляется свежая копия каталога — сверьте `ls -la` на ноутбуке и на сервере. После шага 5 в локальном `results/` лежат файлы с серверными временными метками. Если повторить `rsync` сразу же — он пробежит за секунду и ничего не передаст: значит, синхронизация работает «инкрементно».

## Долгая задача на сервере

Типичная ошибка новичка — запустить на сервере многочасовой расчёт прямо в SSH-сессии, а потом потерять его вместе с оборвавшимся соединением. Правильный приём — запускать долгие задачи внутри `tmux`. Вы заходите на сервер, открываете `tmux`, запускаете задачу — и сессия продолжает жить на сервере, даже когда вы отключитесь:

```shell
$ ssh lab
$ tmux
$ ./run-experiment.sh > out.log 2> err.log
```

Отключиться от сессии, не прерывая её: нажмите `Ctrl+b`, затем `d`. После этого можно спокойно закрыть ноутбук.

Когда захотите проверить, как идут дела, просто подключитесь снова и вернитесь в ту же сессию:

```shell
$ ssh lab
$ tmux attach
```

## Практическое задание

1. Создайте SSH-ключ.
2. Настройте подключение через `~/.ssh/config`.
3. Скопируйте каталог с данными на сервер через `rsync`.
4. Запустите задачу в `tmux`.
5. Отключитесь и подключитесь снова.
6. Заберите результаты обратно через `rsync`.
