# DNS

- [Доп. видео](https://kodekloud.com/topic/dns/)

Компьютеры в сети находят друг друга по числовым IP-адресам, но человеку запоминать `172.217.0.132` неудобно — мы мыслим именами вроде `google.com`. Кто-то должен переводить одно в другое. Этим занимается DNS (Domain Name System, система доменных имён) — распределённая «телефонная книга» интернета, которая сопоставляет имена хостов с IP-адресами.

Можно, конечно, хранить такую книгу на каждой машине вручную — но тогда при смене любого адреса пришлось бы обходить все компьютеры и править записи. DNS решает это иначе: сервер имён публикует адрес для домена в одном месте, и, если адрес меняется, обновить нужно только эту единственную запись.

## Цели раздела

После раздела студент должен уметь:

- объяснить, зачем нужен DNS;
- использовать `ping` для проверки доступности;
- понимать роль `/etc/hosts`, `/etc/resolv.conf`, `/etc/nsswitch.conf`;
- назвать базовые типы DNS-записей;
- проверять разрешение имён через `nslookup` и `dig`.

## `ping`

Прежде чем разбираться с именами, нужен инструмент, который покажет, отвечает ли машина вообще. Это `ping`: он отправляет узлу короткие пакеты и ждёт ответа:

```shell
$ ping 192.168.1.11
PING 192.168.1.11 (192.168.1.11) 56(84) bytes of data.
64 bytes from 192.168.1.11: icmp_seq=1 ttl=64 time=4.12 ms
64 bytes from 192.168.1.11: icmp_seq=2 ttl=64 time=4.05 ms
```

Каждая строка `64 bytes from ...` — это один полученный ответ. Поле `time` показывает, сколько миллисекунд шёл пакет туда и обратно: чем меньше, тем быстрее связь. Прервать команду — `Ctrl+C`: в Linux `ping` по умолчанию работает, пока его не остановят (в отличие от Windows, где он шлёт фиксированное число пакетов).

Аргументом `ping` может быть и IP-адрес, и имя хоста. Но откуда система возьмёт адрес по имени? Самый простой источник — локальный файл `/etc/hosts`. Добавив туда запись, вы сможете обращаться к машине по своему короткому имени. Файл принадлежит администратору, поэтому редактор открывают через `sudo`:

```shell
$ sudo nano /etc/hosts
```

Допишите строку «адрес — имя»:

```text
192.168.1.11 db
```

Теперь имя `db` работает наравне с адресом:

```shell
$ ping db
PING db (192.168.1.11) 56(84) bytes of data.
64 bytes from db (192.168.1.11): icmp_seq=1 ttl=64 time=0.052 ms
64 bytes from db (192.168.1.11): icmp_seq=2 ttl=64 time=0.079 ms
```

В скобках `ping` показывает, в какой адрес он развернул имя `db`, — это удобный способ убедиться, что запись подхватилась. В `/etc/hosts` можно описать сколько угодно имён, по одному на строку:

```text
192.168.1.10 web
192.168.1.11 db
192.168.1.12 nfs
192.168.1.20 web-1
192.168.1.21 db-1
192.168.1.22 nfs-1
```

## Файлы DNS-настроек

`/etc/hosts` хорош для пары машин, но не для всего интернета — для остального система обращается к DNS-серверу. Где его искать, записано в файле `/etc/resolv.conf`:

```shell
$ cat /etc/resolv.conf
nameserver 192.168.1.100
```

Строка `nameserver` и задаёт адрес DNS-сервера, которому система отправляет запросы на разрешение имён.

Остаётся вопрос: что система проверяет первым — локальный файл или DNS-сервер? Это решает `/etc/nsswitch.conf` — файл, который задаёт порядок источников для разных видов данных: имён хостов, пользователей, групп:

```shell
$ cat /etc/nsswitch.conf
# ...
hosts: files dns
# ...
```

Нас интересует строка `hosts`. Значение `files dns` читается по порядку: сначала заглянуть в локальные файлы (`files`, то есть `/etc/hosts`), и только если там имени нет — обратиться к `dns`.

## Доменные имена

Доменные имена не плоские, а иерархические — справа налево, от общего к частному. Самая правая часть — доменная зона верхнего уровня, и по ней часто можно угадать тип организации:

- `.com` — коммерческие или общие домены;
- `.net` — сетевые или общие домены;
- `.edu` — образовательные организации;
- `.org` — организации, часто некоммерческие.

## Типы DNS-записей

DNS хранит не просто пары «имя — адрес». Для разных задач есть разные типы записей:

- `A` — сопоставляет имя с IPv4-адресом (самый распространённый случай);
- `AAAA` — то же, но для IPv6-адреса;
- `CNAME` — псевдоним: одно имя указывает не на адрес, а на другое имя.

Проверить, как именно разрешается имя, проще всего командой `nslookup`:

```shell
$ nslookup www.google.com
Server: 8.8.8.8
Address: 8.8.8.8#53
Non-authoritative answer:
Name: www.google.com
Address: 172.217.0.132
```

Здесь `Server` — это DNS-сервер, который ответил на запрос, а `Address` в конце — найденный IP-адрес имени. Пометка `Non-authoritative answer` означает, что ответ пришёл из кэша сервера, а не напрямую от того, кто отвечает за этот домен.

Когда нужно больше деталей, на помощь приходит `dig` (domain information groper):

```shell
$ dig www.google.com
; <<>> DiG 9.10.3-P4-Ubuntu <<>> www.google.com
;; global options: +cmd
;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 28065
;; flags: qr rd ra; QUERY: 1, ANSWER: 6, AUTHORITY: 0, ADDITIONAL: 1
;; QUESTION SECTION:
;www.google.com. IN A
;; ANSWER SECTION:
www.google.com. 245 IN A 64.233.177.103
;; Query time: 5 msec
;; SERVER: 8.8.8.8#53(8.8.8.8)
```

Вывод `dig` подробный, но читать его несложно. Главное — секция `ANSWER SECTION`: в ней и лежит результат. Строка `www.google.com. 245 IN A 64.233.177.103` говорит, что у имени запись типа `A` с адресом `64.233.177.103`, а число `245` — это TTL (Time To Live, «время жизни»), сколько секунд этот ответ можно держать в кэше, прежде чем перепросить заново. Ниже `Query time` показывает, сколько занял запрос, а `SERVER` — кто на него ответил.

## Разобранный пример: «почему `ping db` ведёт не туда»

**Задача.** На машине администратора `ping db` отвечает с адреса `10.0.0.5`, хотя в DNS компании `db.local.lan` указывает на `192.168.5.5`. Нужно понять, откуда берётся «старый» адрес, и проверить порядок источников имён.

**Решение.**

1. Смотрим, кто отвечает первым. В `/etc/nsswitch.conf` находим строку `hosts: files dns`. Порядок читается слева направо: сначала локальные файлы, потом DNS. Значит, если имя `db` есть в `/etc/hosts`, до DNS дело даже не дойдёт.
2. Заглядываем в `/etc/hosts`: `grep db /etc/hosts`. Находим строку `10.0.0.5 db` — она и срабатывает. Это «застрявший» след от прошлой настройки.
3. Удаляем или комментируем строку: `sudo nano /etc/hosts` → ставим `#` в начале нужной строки → `Ctrl+O`, `Ctrl+X`. Перезапуска не требуется — система перечитывает файл при каждом обращении.
4. Проверяем через `getent hosts db` — это правильный способ задать вопрос «куда система разрешит имя», потому что `getent` обходит источники в том же порядке, что и приложения.
5. Если ответ всё равно «не тот», смотрим, не «закешировал» ли резолвер старый адрес: `resolvectl flush-caches` (в системах с systemd-resolved). После этого `ping db` должен пойти уже на `192.168.5.5`.

**Проверка.** `getent hosts db` показывает ожидаемый адрес из DNS. `ping db` в скобках выводит тот же адрес — значит, имя действительно разрешается через DNS, а не из локального файла.

> **Типичные ошибки.**
>
> - **«Поправил `/etc/hosts`, а ничего не изменилось».** Проверьте порядок в `/etc/nsswitch.conf`: если там `hosts: dns files`, локальный файл смотрится после DNS и его записи могут «затенять» нужные. И второе — некоторые приложения (например, браузеры) кешируют адреса сами; перезапустите процесс или сбросьте кеш.
> - **`ping` показывает «name or service not known» только для одного имени.** Часто оказывается, что в `/etc/hosts` оно записано с лишним пробелом, табом или невидимым символом из скопированного куска. Откройте файл в `cat -A /etc/hosts` — невидимые символы станут видны.
> - **`nslookup` отвечает быстро, а приложение всё равно «не находит» хост.** `nslookup` обращается напрямую к DNS, минуя `/etc/hosts` и `nsswitch.conf`. Чтобы повторить тот же путь, что у приложения, используйте `getent hosts <имя>`.
> - **`dig` без `+short` — это «стена текста».** Если нужен только адрес, добавляйте `+short`: `dig +short google.com` вернёт ровно одну строку с IP-адресом.

## Практическое задание

1. Добавьте тестовую запись в `/etc/hosts`.
2. Проверьте её через `ping`.

4. Проверьте порядок разрешения имён в `/etc/nsswitch.conf`.
5. Выполните `nslookup` и `dig` для одного домена.
