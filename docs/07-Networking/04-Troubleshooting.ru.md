# Диагностика сетевых проблем

- [Доп. видео](https://kodekloud.com/topic/troubleshooting/)

Вернёмся к проблеме, с которой начинался раздел: Боб не может подключиться к серверу репозитория. Соблазн велик — начать перезагружать всё подряд в надежде, что само починится. Но грамотная диагностика устроена иначе: вы идёте по цепочке связи шаг за шагом, от ближнего к дальнему, и на каждом шаге задаёте один простой вопрос. Первый же шаг, на котором ответ окажется «нет», и укажет на поломку. Пройдём эту цепочку целиком.

## Цели раздела

После раздела студент должен уметь:

- проверять состояние сетевого интерфейса;
- проверять разрешение имени в IP-адрес;
- проверять доступность через `ping`;
- смотреть маршрут через `traceroute`;
- проверять, слушает ли сервис нужный порт;
- поднимать сетевой интерфейс.

## Проверить интерфейсы

Начинаем с самого ближнего к нам звена — сетевого интерфейса своей машины. Вопрос простой: он вообще включён? Это покажет `ip link`:

```shell
$ ip link
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN mode DEFAULT group default qlen 1000
2: enp1s0f1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP mode DEFAULT group default qlen 1000
```

Первый интерфейс, `lo`, — это loopback, внутренняя «петля» машины; нас интересует второй, `enp1s0f1`. У него `state UP` — значит, интерфейс поднят, и на этом шаге всё в порядке. Если бы здесь стояло `DOWN`, поломка нашлась бы сразу.

## Проверить DNS

Интерфейс работает — идём дальше. Боб обращается к серверу по имени `caleston-repo-01`, а сеть работает с адресами. Значит, следующий вопрос: превращается ли имя в IP-адрес? Проверим командой `nslookup`:

```shell
$ nslookup caleston-repo-01
Server:      192.168.1.100
Address:     192.168.1.100#53

Non-authoritative answer:
Name: caleston-repo-01
Address: 192.168.2.5
```

Имя разрешилось в адрес `192.168.2.5` — DNS свою работу сделал. Теперь мы знаем, куда именно стучаться.

## Проверить связность

Адрес есть — отвечает ли по нему хоть кто-нибудь? Это и проверяет `ping`:

```shell
$ ping caleston-repo-01
PING caleston-repo-01 (192.168.2.5) 56(84) bytes of data.

--- caleston-repo-01 ping statistics ---
3 packets transmitted, 0 received, 100% packet loss, time 2034ms
```

А вот и первое «нет». Строка `3 packets transmitted, 0 received, 100% packet loss` означает: отправлено три пакета, не вернулся ни один. Узел не отвечает. Важный вывод: раз DNS отработал, а `ping` — нет, проблема не в именах. Дело в чём-то между нами и сервером: в маршрутизации, в межсетевом экране или в самом удалённом хосте. Сужаем поиск дальше.

## Проверить маршрут

`ping` сказал, что пакеты не доходят, — но где именно они теряются? Это покажет `traceroute`: он прослеживает путь до цели и выводит каждый промежуточный узел (маршрутизатор), который пакет проходит по дороге. На Ubuntu/Debian этой команды может не быть из коробки — тогда её ставят пакетом `traceroute` (`sudo apt install traceroute`).

```shell
$ traceroute 192.168.2.5
traceroute to 192.168.2.5 (192.168.2.5), 30 hops max, 60 byte packets
 1  192.168.1.1 (192.168.1.1)  0.521 ms  0.498 ms  0.472 ms
 2  192.168.2.1 (192.168.2.1)  1.245 ms  1.198 ms  1.156 ms
 3  * * *
```

Каждая пронумерованная строка — это очередной узел на пути к цели, а три значения справа — время отклика для трёх пробных пакетов. Здесь видна важная картина: первые два узла ответили нормально, а на третьем пошли звёздочки. Строка `* * *` означает, что узел не ответил в отведённое время. То есть пакеты Боба уходят, доходят до второго маршрутизатора — и дальше теряются. Поломка где-то за ним.

## Проверить порт

Допустим, до сервера пакеты всё-таки доходят. Остаётся последний вопрос цепочки: а слушает ли нужный сервис свой порт? Если служба не запущена, машина будет отвечать на `ping`, но соединение на порт всё равно не примет. В старых материалах для этого используют `netstat` из пакета `net-tools`:

```shell
$ netstat -an | grep 80 | grep -i LISTEN
```

В современных дистрибутивах (Ubuntu 22.04, Debian 11+, RHEL 9+) пакет `net-tools` не ставится по умолчанию, и команда `netstat` просто не найдётся. Её современная замена — `ss` (socket statistics) из пакета `iproute2`. Опции `-tuln` означают: TCP- и UDP-сокеты, в режиме «слушающих» (`-l`), с числовыми портами без перевода в имена (`-n`):

```shell
$ ss -tuln | grep ':80'
```

Если в выводе есть строка с `:80`, значит, на этом порту кто-то слушает.

## Поднять интерфейс

Самый частый «виновник» — выключенный интерфейс из первого шага. Поднять его обратно можно командой `ip link set` (нужны права администратора):

```shell
$ sudo ip link set dev enp1s0f1 up
```

## Контрольный порядок диагностики

1. `ip link` — интерфейс включён?
2. `ip addr` — есть IP-адрес?
3. `ip route` — есть маршрут?
4. `nslookup` или `dig` — работает DNS?
5. `ping` — есть связность?
6. `ss` или `netstat` — слушает ли сервис порт?

## Возврат к Бобу

> **Возврат к Бобу.** Боб развернул внутренний сайт на сервере приложений: служба слушает на `0.0.0.0:8080`, со своего ноутбука Боб открывает `http://app-server:8080` — браузер показывает «не удаётся подключиться». При этом `ssh app-server` и `ping app-server` работают, а с самого сервера `curl http://localhost:8080` отдаёт корректный ответ. Опишите план диагностики: какие команды Боб выполнит по порядку и на каком шаге вы ожидаете найти проблему? Подсказка: задействуйте `ss`, проверку фаервола, `curl -v` и логи службы из модуля 09.
