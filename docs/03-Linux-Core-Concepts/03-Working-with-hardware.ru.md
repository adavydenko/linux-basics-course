# Работа с оборудованием

- [Доп. видео](https://kodekloud.com/topic/working-with-hardware/)

Вы подключаете USB-диск — и через секунду он уже доступен системе. Кто его заметил, кто решил, под каким именем он появится? За этим стоит цепочка из ядра, драйверов и служб, и понимать её полезно: рано или поздно устройство «не подхватится», и придётся разбираться, на каком звене сломалось. В этом разделе посмотрим, как Linux обнаруживает оборудование и как расспросить систему о её устройствах прямо из командной строки.

## Цели раздела

После раздела студент должен уметь:

- объяснить роль драйверов устройств;
- понимать, зачем нужен `udev`;
- читать сообщения ядра через `dmesg`;
- получать информацию о PCI-устройствах, блочных устройствах, CPU и памяти;
- понимать, когда команде нужны права суперпользователя.

## Как Linux видит устройства

Проследим всю цепочку на примере USB-диска. Как только вы подключаете устройство, его замечает драйвер в пространстве ядра: он видит изменение состояния шины и формирует уведомление об этом — событие `uevent`.

Само ядро файлы устройств не создаёт — оно лишь сообщает о событии наружу, в пространство пользователя, диспетчеру устройств `udev`. А вот `udev` уже действует: он динамически создаёт для нового устройства узел в каталоге `/dev/`. Поэтому, когда процесс завершится, подключённый диск появится среди файлов `/dev/` — и с этого момента система может с ним работать.

## Сообщения ядра: `dmesg`

Ядро не молчит о том, что делает: по ходу работы оно записывает сообщения в кольцевой буфер (ring buffer) — небольшую область памяти, где новые записи постепенно вытесняют старые. Прочитать этот буфер позволяет команда `dmesg`. Особенно много записей появляется при загрузке системы — об обнаруженном оборудовании, загруженных драйверах и ошибках настройки.

```shell
$ dmesg
$ dmesg | grep -i usb
```

Первая команда выводит весь буфер целиком, и в нём легко утонуть. Вторая полезнее на практике: она пропускает вывод через `grep` и оставляет только строки со словом `usb` (флаг `-i` делает поиск нечувствительным к регистру). Так `dmesg` превращается в инструмент диагностики — подключили устройство, оно «не видно», заглянули в сообщения ядра и узнали, что пошло не так.

## Информация от `udev`

Раз `udev` ведёт учёт устройств, у него можно расспросить подробности. За это отвечает утилита `udevadm`. В режиме `info` она достаёт сведения о конкретном устройстве из базы — например, его путь в системе:

```shell
$ udevadm info --query=path --name=/dev/sda5
```

А режим `monitor` работает иначе — он не отвечает на разовый вопрос, а слушает события ядра в реальном времени и печатает их по мере появления: путь и имя устройства. Запустите его, подключите или отключите оборудование — и увидите, что происходит, прямо в момент события:

```shell
$ udevadm monitor
```

## Просмотр оборудования

Кроме событий, систему можно просто спросить: «что у тебя за железо?». Для этого есть целое семейство команд с приставкой `ls` (list) — каждая отвечает за свой вид оборудования. Запомнить их легко по этому общему корню.

Начнём с `lspci`. Она выводит PCI-устройства, подключённые к системе, — это, например, Ethernet-карты, RAID-контроллеры, видеокарты и беспроводные адаптеры:

```shell
$ lspci
```

`lsblk` показывает блочные устройства — диски и их разделы — в виде наглядного дерева:

```shell
$ lsblk
```

В колонке `MAJ:MIN` каждое устройство получает пару чисел — старший и младший номера. Старший номер (`MAJ`) кодирует *класс* устройства; по нему ядро понимает, какой драйвер должен с ним работать. Самые распространённые классы:

| Старший номер | Класс устройства |
|---:|---|
| 1 | оперативная память (RAM) |
| 3 | жёсткий диск или CD-ROM |
| 6 | параллельные принтеры |
| 8 | SCSI-диски |

Младший номер (`MIN`) различает уже конкретные экземпляры одного класса: разделы одного диска, отдельные принтеры и так далее.

`lscpu` рассказывает о процессоре: архитектуру, число ядер, режимы работы. Эти сведения важны, когда вы планируете запускать вычислительные задачи и хотите понять, на что рассчитывать:

```shell
$ lscpu
```

`lsmem --summary` выводит краткую сводку по доступной памяти:

```shell
$ lsmem --summary
```

Похожую информацию, но в разрезе «занято / свободно», даёт `free`; флаг `-m` показывает значения в мегабайтах:

```shell
$ free -m
```

Наконец, `lshw` (list hardware) собирает подробный отчёт сразу обо всём оборудовании машины:

```shell
$ lshw
```

С `lshw` связан важный момент. Часть сведений об оборудовании доступна только администратору, поэтому без прав суперпользователя `lshw` покажет неполную картину. Чтобы получить всё, команду запускают через `sudo`:

```shell
$ sudo lshw
```

Это общее правило: если команда «недоговаривает» или жалуется на доступ, скорее всего, ей нужны права суперпользователя.

> **Типичные ошибки.**
>
> - **`dmesg` без `grep` — стена текста.** Запустив `dmesg` без фильтра, легко не заметить нужное сообщение среди сотен строк загрузки. Привыкайте сразу сужать поиск: `dmesg | grep -i usb`, `dmesg | tail -50`, `dmesg --since "10 min ago"`.
> - **`sudo` забыт перед `lshw`.** Без прав суперпользователя `lshw` молча выдаёт неполную информацию — не ругается, не подсказывает. Если в выводе странно мало деталей, перезапустите с `sudo`.
> - **«Устройство не появилось в `/dev`».** Чаще всего дело не в железе, а в драйвере: ядро увидело событие, но `udev` не смог сопоставить ему правило. Заглядывайте в `dmesg` сразу после подключения — там будут подсказки, какого модуля не хватило.

## Практическое задание

1. Выполните `lsblk` и найдите блочные устройства типа `disk`.
2. Выполните `lscpu` и определите количество CPU или ядер.
3. Выполните `free -m` и запишите общий объём памяти.
4. Выполните `dmesg | grep -i usb` и объясните, есть ли в выводе сообщения об USB.
