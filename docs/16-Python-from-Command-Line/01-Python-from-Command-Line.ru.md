# Python из командной строки

После shell-скриптов (модуль 12) Python — это естественный следующий шаг, когда «склейки команд» становится мало: нужны структуры данных, парсинг JSON, сетевые запросы, более сложная логика. Этот раздел не является курсом Python; он рассчитан на студентов, у которых Python уже есть в активе. Его цель — научить запускать Python-скрипты в Linux воспроизводимо: из терминала, с аргументами, логами и изолированным окружением, чтобы код, написанный на ноутбуке, без проблем работал на сервере.

## Цели раздела

После раздела студент должен уметь:

- проверить, какой Python запускается;
- запускать Python-файлы из терминала;
- передавать аргументы;
- использовать `stdin`, `stdout`, `stderr`;
- создавать виртуальное окружение через `venv`;
- устанавливать зависимости из `requirements.txt`;
- запускать долгую Python-задачу с логами.

## Какой Python запускается

Первый источник путаницы для новичка — то, что слово `python` в системе может означать не то, что вы думаете. Поэтому начните не с запуска скрипта, а с вопроса «а какой вообще интерпретатор у меня под рукой»:

```shell
$ python --version
$ python3 --version
$ which python
$ which python3
```

`--version` показывает версию, а `which` — путь к исполняемому файлу. В разных системах `python` и `python3` могут указывать на разные интерпретаторы, а где-то `python` отсутствует вовсе. Чтобы не зависеть от случайностей конкретной машины, в учебных инструкциях лучше всегда писать `python3` явно.

## Запуск файла

Запустить скрипт — значит передать его файл интерпретатору:

```shell
$ python3 script.py
```

Скрипт становится по-настоящему полезным, когда работает с разными входными данными. Всё, что вы допишете после имени файла, передаётся ему как аргументы:

```shell
$ python3 script.py input.csv output.txt
```

Внутри программы эти аргументы видны через список `sys.argv`, а для скриптов посложнее их разбирают библиотекой `argparse`.

## Перенаправления и коды завершения

Когда скрипт что-то печатает, по умолчанию это уходит на экран. Но вывод можно перенаправить в файл символом `>` — удобно, чтобы сохранить результат расчёта:

```shell
$ python3 script.py > out.log
```

Здесь важная тонкость: у программы два отдельных потока вывода — обычный (stdout) и поток ошибок (stderr). Запись `2>` направляет именно ошибки, и разводить их по разным файлам — хорошая привычка: в `out.log` остаётся чистый результат, а всё, что пошло не так, собирается в `err.log`:

```shell
$ python3 script.py > out.log 2> err.log
```

А завершился ли скрипт успешно, подскажет код завершения в переменной `$?`: `0` — всё хорошо, ненулевое значение — была ошибка:

```shell
$ echo $?
```

## Виртуальное окружение `venv`

Теперь о главном инструменте воспроизводимости. Велик соблазн ставить пакеты глобально через `sudo pip`, но делать так не стоит: можно затронуть пакеты, от которых зависит сама система, а результат станет невоспроизводимым — на другой машине набор библиотек окажется иным. Решение — `venv` (virtual environment): изолированная папка с собственным Python и собственным набором пакетов, отдельным для каждого проекта.

Создаётся окружение одной командой; по соглашению папку называют `.venv`:

```shell
$ python3 -m venv .venv
```

Само по себе окружение пассивно — его нужно активировать, чтобы оно «подхватилось» текущей оболочкой:

```shell
$ source .venv/bin/activate
```

После активации стоит убедиться, что `python` теперь указывает именно внутрь `.venv`, а не на системный интерпретатор:

```shell
$ which python
$ python --version
```

Внутри активного окружения `pip` ставит пакеты локально, в саму папку `.venv`. Список нужных пакетов проекта обычно лежит в файле `requirements.txt`, и установить их все разом можно так:

```shell
$ pip install -r requirements.txt
```

Обратное действие — записать текущий набор пакетов в этот файл, чтобы другой человек смог воспроизвести ваше окружение:

```shell
$ pip freeze > requirements.txt
```

Когда работа закончена, окружение отключают командой `deactivate` — оболочка возвращается к системному Python:

```shell
$ deactivate
```

> **Типичные ошибки.**
>
> - **`sudo pip install` в системный Python.** Кажется коротким путём, но ломает пакеты, на которые опирается сама система: следующий `apt upgrade` может конфликтовать с тем, что вы поставили. Всегда работайте через `venv`.
> - **`.venv` есть, но `pip` ставит куда-то ещё.** Признак — забытое `source .venv/bin/activate`. Проверка одной командой: `which python` после активации должно указывать *внутрь* `.venv`. Если нет — окружение не активно.
> - **Активированное `.venv` коммитят в Git.** В нём сотни мегабайт скачанных пакетов, и у другого человека всё равно понадобится своя. Решение — добавить `.venv/` в `.gitignore`, а зависимости фиксировать через `pip freeze > requirements.txt`.
> - **Разные версии Python на ноутбуке и сервере.** Скрипт на 3.12 спокойно использует `match`, а на сервере с 3.9 это синтаксическая ошибка. Привычка фиксировать минимальную версию в `README.md` и при необходимости в `pyproject.toml` снимает половину «у меня же работало».

## Долгий запуск

Долгие Python-задачи запускают по тем же правилам, что и любые другие. Чтобы расчёт пережил закрытие сессии, используют `nohup`, перенаправляя оба потока вывода в файлы. Обратите внимание: интерпретатор берётся прямо из `.venv` по полному пути — так задача точно увидит установленные пакеты, даже без активации окружения:

```shell
$ nohup .venv/bin/python script.py data.csv > out.log 2> err.log &
```

Для интерактивной долгой работы удобнее `tmux`: запускаете задачу внутри сессии, отключаетесь, при желании возвращаетесь:

```shell
$ tmux
$ source .venv/bin/activate
$ python script.py data.csv
```

## Разобранный пример: воспроизводимое окружение нового проекта

**Задача.** Коллега прислал короткий Python-скрипт `analyze.py`, который читает CSV и считает статистику с помощью `pandas`. На вашей машине Python — системный, лишние пакеты в нём ставить нельзя. Нужно подготовить изолированное окружение, поставить зависимости и убедиться, что скрипт запускается одинаково у вас и у коллеги.

**Решение.**

1. Создать каталог проекта и зайти внутрь:
   ```shell
   $ mkdir ~/work/analyze && cd ~/work/analyze
   ```
2. Создать виртуальное окружение и активировать его. Папку называем `.venv` — это соглашение, которое потом узнают и редактор, и `.gitignore`:
   ```shell
   $ python3 -m venv .venv
   $ source .venv/bin/activate
   ```
3. Убедиться, что `python` теперь указывает внутрь `.venv`. Это самый частый источник «у меня pandas не ставится — а его и не должно быть в системе»:
   ```shell
   $ which python
   $ python --version
   ```
4. Установить нужные пакеты и сразу зафиксировать список — иначе через неделю забудете, *какие* версии работали:
   ```shell
   $ pip install pandas
   $ pip freeze > requirements.txt
   ```
5. Положить скрипт коллеги и запустить, разводя нормальный вывод и ошибки по разным файлам:
   ```shell
   $ python analyze.py data.csv > out.log 2> err.log
   $ echo $?
   ```
6. Закрыть окружение, когда работа окончена:
   ```shell
   $ deactivate
   ```

**Проверка.** `cat requirements.txt` показывает `pandas==...` (с конкретной версией). Файл `out.log` содержит результат, `err.log` пуст, `echo $?` вернул `0`. У коллеги тот же скрипт после `python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt` даст идентичный результат — это и есть воспроизводимость.

## Практическое задание

1. Создайте каталог проекта.
2. Создайте `.venv`.
3. Активируйте окружение.
4. Установите один пакет и сохраните `requirements.txt`.
5. Напишите короткий скрипт, который принимает имя входного файла.
6. Запустите скрипт с перенаправлением stdout и stderr.
7. Запустите скрипт через `nohup`.
