# Права доступа к файлам

- [Доп. видео](https://kodekloud.com/topic/file-permissions-and-ownership/)

В Linux у каждого файла и каталога есть владелец и набор правил, которые определяют, кто и что может с ним делать. Эти правила — права доступа — лежат в основе безопасности системы: благодаря им обычный пользователь не может изменить системные файлы, а чужие данные остаются недоступными для посторонних. В этом разделе разберёмся, как читать права в выводе `ls -l` и как менять их под свои задачи.

## Цели раздела

После раздела студент должен уметь:

- читать права доступа в выводе `ls -l`;
- понимать права `read`, `write`, `execute`;
- изменять права через `chmod`;
- менять владельца и группу через `chown` и `chgrp`;
- понимать числовую форму прав.

## Как читать права

Когда вы выполняете `ls -l`, первый столбец вывода выглядит примерно как `-rwxrw-r--`. Это и есть права доступа в компактной записи.

![perm](../images/ru_perm.svg)

Первый символ говорит о типе объекта: `-` — обычный файл, `d` — каталог, `l` — символическая ссылка и так далее.

Оставшиеся девять символов разбиты на три тройки — для владельца файла, для его группы и для всех остальных. В каждой тройке три права: `r` (read) — чтение, `w` (write) — запись, `x` (execute) — выполнение. Если право есть, на его месте стоит буква; если нет — прочерк. Так, `rwxrw-r--` означает: владельцу разрешено всё, группе — чтение и запись, остальным — только чтение.

## Права каталогов

У каталога права читаются немного иначе, чем у файла, поэтому начнём с них. Посмотреть права конкретного каталога можно той же командой `ls`, но с флагом `-d` — иначе она покажет содержимое каталога, а не его самого:

```shell
$ ls -ld /home/bob/random_dir
```

Чтобы понять, какая из трёх троек прав относится к вам, нужно знать, под каким пользователем вы работаете. Это подскажет `whoami`:

```shell
$ whoami
```

Главное отличие — в праве `x`. Для каталога оно означает возможность войти в него и проходить через него по пути. Без `x` каталог нельзя «открыть», даже если на него есть право на чтение:

```shell
$ cd /home/bob/random_dir
-bash: cd: /home/bob/random_dir: Permission denied
```

Хитрость с правами каталога в том, что `r` и `x` отвечают за разные задачи: `r` разрешает посмотреть **список** содержимого через `ls`, а `x` — войти внутрь и обращаться к файлам по имени. Это видно, если оставить только одно из этих прав:

```shell
$ ls /home/bob/random_dir            # с x, но без r — Permission denied
$ cat /home/bob/random_dir/known.txt # тот же случай — но файл открывается, если его точное имя известно
```

То есть каталог с правами `--x` ведёт себя как закрытая комната, в которую можно войти по точному адресу, но нельзя посмотреть, что в ней лежит.

## Права файлов

![filep](../images/ru_filep.svg)

Для обычного файла права означают примерно то, чего и ожидаешь: `r` — можно прочитать содержимое, `w` — можно изменить файл, `x` — файл можно запустить как программу или скрипт.

## Изменение прав: `chmod`

Когда понятно, что показывает `ls -l`, остаётся научиться эти права менять. За это отвечает одна команда — `chmod` (change mode). Она умеет работать в двух форматах, и оба удобно держать в голове: символьный нагляднее («владельцу добавить выполнение»), а числовой компактнее (`chmod 750`). Разберём их по очереди.

**Символьная форма** читается почти как фраза: «кому» — `u` (владелец), `g` (группа), `o` (остальные); «что сделать» — `+` добавить право, `-` убрать; «какое право» — `r`, `w`, `x`.

Дать владельцу полный доступ к файлу:

```shell
$ chmod u+rwx test-file
```

Одной командой можно затронуть сразу несколько категорий — например, дать всем троим чтение и одновременно убрать у них выполнение:

```shell
$ chmod ugo+r-x test-file
```

Убрать у остальных все права разом:

```shell
$ chmod o-rwx test-file
```

А ещё в одной команде можно описать разные изменения для разных категорий, разделив их запятыми — владельцу полный доступ, группе только чтение, остальным ничего:

```shell
$ chmod u+rwx,g+r-x,o-rwx test-file
```

**Числовая форма** короче. Каждое право — это число: `r` = 4, `w` = 2, `x` = 1. Сумма этих чисел задаёт набор прав для одной категории, а три цифры подряд описывают владельца, группу и остальных.

Так, `777` — полный доступ для всех (4 + 2 + 1 в каждой позиции):

```shell
$ chmod 777 test-file
```

`555` — чтение и выполнение для всех, без права записи (4 + 1):

```shell
$ chmod 555 test-file
```

`660` — чтение и запись владельцу и группе, остальным ничего (4 + 2, затем 0):

```shell
$ chmod 660 test-file
```

`750` — полный доступ владельцу (4 + 2 + 1), чтение и выполнение группе (4 + 1), остальным ничего:

```shell
$ chmod 750 test-file
```

Теперь, увидев в чужой инструкции `chmod 644`, вы сможете прочитать это число сами: владельцу — чтение и запись, группе и остальным — только чтение.

## Разобранный пример: «закройте скрипт от посторонних, оставьте группе только запуск»

**Задача.** В каталоге `~/bin` лежит скрипт `deploy.sh`, который содержит пароли в открытом виде. Нужно сделать так, чтобы владелец мог его читать, изменять и запускать, члены группы — только запускать (без чтения исходника), а остальные — вообще ничего.

**Решение.**

1. Смотрим, что сейчас: `ls -l ~/bin/deploy.sh`. Допустим, права `-rw-r--r--` — это `644`: владельцу чтение и запись, остальным только чтение. Подходит не очень — скрипт могут прочитать чужие, а группа даже запустить его не может.
2. Расписываем требуемые права по тройкам: владелец — `rwx` (4 + 2 + 1 = 7), группа — `--x` (только 1), остальные — `---` (0). Итог: `710`.
3. Применяем: `chmod 710 ~/bin/deploy.sh`. Можно и символьной формой — `chmod u=rwx,g=x,o= ~/bin/deploy.sh`, — но числовая короче, когда требуется задать сразу все три тройки.
4. Проверяем результат: `ls -l ~/bin/deploy.sh` должен показать `-rwx--x---`. Первая тройка — `rwx`, вторая — `--x`, третья — пусто.
5. Если у файла «не та» группа (например, личная группа `bob`, а нужна `developers`), меняем её одной командой: `sudo chgrp developers ~/bin/deploy.sh`. Иначе право `g+x` дано группе, в которой больше никого нет — настройка работает, но никто, кроме владельца, ничего не увидит.

**Проверка.** Член группы `developers` выполняет `bash ~bob/bin/deploy.sh` (или `~bob/bin/deploy.sh`, если файл доступен через `x` на пути) — скрипт запускается, но `cat ~bob/bin/deploy.sh` отвечает «Permission denied». Это и есть нужное поведение: запускать можно, заглянуть внутрь — нельзя.

## Изменение владельца

Права отвечают на вопрос «что можно делать», а вот «кому» — это владелец и группа файла. Менять их умеют команды `chown` (change owner) и `chgrp` (change group); обычно для этого нужны права администратора.

`chown` может изменить сразу и владельца, и группу — они указываются через двоеточие:

```shell
$ chown bob:developer test-file
```

Если указать только имя пользователя, сменится лишь владелец, а группа останется прежней:

```shell
$ chown bob android.apk
```

Когда нужно поменять только группу, удобнее `chgrp`:

```shell
$ chgrp android test-file
```

## Специальные биты прав

Кроме «обычных» `rwx`-прав в Linux есть три дополнительных бита, которые меняют поведение файла или каталога. Их видно в выводе `ls -l` — они занимают места `x`-битов, но изображаются разными буквами.

**SUID (Set User ID, бит «выполнять от имени владельца»).** Если у исполняемого файла стоит SUID и владелец — `root`, то при запуске обычным пользователем процесс получает права владельца. Классический пример — `/usr/bin/passwd`: пароль меняет каждый, но запись идёт в `/etc/shadow`, доступный только `root`. SUID показывается буквой `s` на месте `x` владельца:

```shell
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 May 12 02:30 /usr/bin/passwd
```

Поставить и снять SUID:

```shell
$ sudo chmod u+s /usr/local/bin/my-tool
$ sudo chmod u-s /usr/local/bin/my-tool
```

В числовой форме SUID — это четвёртая цифра: `chmod 4755 my-tool` означает «`755` + SUID».

**SGID (Set Group ID).** Аналог SUID, но для группы. На исполняемом файле даёт «выполнять с правами группы». На каталоге работает иначе и полезнее: все новые файлы внутри каталога автоматически получают группу самого каталога, а не основную группу создающего. Это спасает в общих каталогах вроде `/srv/project/`:

```shell
$ sudo chmod g+s /srv/project
$ sudo chgrp developers /srv/project
```

Теперь всё, что любой член `developers` создаёт в `/srv/project`, получает группу `developers` — без неё файлы оказались бы личными по группе.

**Sticky bit (бит липкости, `t`).** Имеет смысл только на каталоге: запретить пользователям удалять чужие файлы, даже если у каталога права `rwxrwxrwx`. Удалять файл может либо его владелец, либо `root`. Самый известный пример — `/tmp`:

```shell
$ ls -ld /tmp
drwxrwxrwt 18 root root 12288 May 19 14:30 /tmp
```

Буква `t` в самом конце — это sticky. В числовой форме — `chmod 1777 /tmp`.

Все три бита можно ставить одновременно, передавая `chmod` четырёхзначное число: `chmod 6755` — это `755` + SUID + SGID, `chmod 7777` — все три бита поверх `777`.

## `umask`: маска прав по умолчанию

Когда вы создаёте новый файл командой `touch` или каталог через `mkdir`, права им не выдаются «с потолка». Система берёт стандартный набор (`666` для файлов, `777` для каталогов) и снимает с него биты, перечисленные в `umask` (от user mask). Так получаются итоговые права.

Посмотреть текущую маску:

```shell
$ umask
0022
```

Значение `022` означает: снять у группы и у остальных бит `w`. Поэтому новые файлы создаются с правами `644` (`666 − 022`), а каталоги — с `755` (`777 − 022`).

Сменить маску в текущей оболочке (например, чтобы все новые файлы были закрыты от чужих):

```shell
$ umask 077
$ touch test
$ ls -l test
-rw------- 1 bob bob 0 May 19 14:35 test
```

Маска `077` снимает все права у группы и остальных — файл доступен только владельцу. Эту команду можно положить в `~/.profile`, чтобы маска применялась во всех новых сессиях.

## Практическое задание

1. Создайте файл и посмотрите его права через `ls -l`.
2. Дайте владельцу выполнение через `chmod u+x`.
3. Уберите все права у остальных через `chmod o-rwx`.
4. Установите права `750` и объясните каждую цифру.
5. Измените группу файла через `chgrp`.
