# IPtables

- [Доп. видео](https://kodekloud.com/topic/iptables-introduction/)

Права доступа защищают файлы внутри системы. Но сервер ещё и подключён к сети — и через сеть к нему может постучаться кто угодно. Межсетевой экран (firewall) решает, какие сетевые пакеты пропустить, а какие отбросить. В Linux классический инструмент для этого — `iptables`. Разберём, как он устроен и как составлять простые правила фильтрации.

> В современных дистрибутивах всё чаще используются `nftables`, `ufw` или `firewalld`. Но `iptables` никуда не делся: он по-прежнему встречается в учебных материалах, на старых серверах и в документации, а его логика лежит в основе более новых инструментов — поэтому начать стоит именно с него.

## Цели раздела

После раздела студент должен уметь:

- понимать таблицы и цепочки `iptables`;
- смотреть текущие правила;
- добавлять разрешающие и запрещающие правила;
- отличать `DROP` от `REJECT`;
- удалять правило по номеру строки;
- понимать назначение `INPUT`, `OUTPUT`, `FORWARD`.

## Таблицы и цепочки

У `iptables` есть своя внутренняя иерархия, и, не разобравшись в ней, легко запутаться. Самый верхний уровень — **таблицы**: каждая отвечает за свой вид обработки пакетов. Внутри таблицы лежат **цепочки** — это списки правил, привязанные к определённому моменту в жизни пакета. А уже сами **правила** в цепочке описывают, как поступить с конкретным пакетом.

Двух таблиц достаточно, чтобы понять картину:

1. `FILTER` — таблица по умолчанию, та самая, что фильтрует трафик. В ней три цепочки, и они отличаются направлением пакета:
   - `INPUT` — пакеты, адресованные самой этой системе;
   - `FORWARD` — пакеты, которые лишь проходят через систему транзитом (актуально для маршрутизаторов);
   - `OUTPUT` — пакеты, рождённые на самой системе и уходящие наружу.
2. `NAT` — таблица для подмены адресов при установке нового соединения. Её цепочки:
   - `PREROUTING` — правка пакета сразу после получения, до маршрутизации;
   - `OUTPUT` — правка локально созданных пакетов;
   - `POSTROUTING` — правка пакетов перед самой отправкой наружу.

В Ubuntu 22.04 и большинстве современных дистрибутивов пакет `iptables` уже установлен по умолчанию. Если по какой-то причине его нет — поставьте:

```shell
$ sudo apt install iptables
```

Прежде чем что-то менять, всегда стоит посмотреть, что уже есть. Опция `-L` (list) выводит текущие правила:

```shell
$ sudo iptables -L
```

На чистой системе вывод будет таким:

```text
Chain INPUT (policy ACCEPT)
target     prot opt source               destination

Chain FORWARD (policy ACCEPT)
target     prot opt source               destination

Chain OUTPUT (policy ACCEPT)
target     prot opt source               destination
```

Все три цепочки пусты, а в скобках указана политика по умолчанию — `ACCEPT`. Это значит: пока правил нет, пропускается вообще всё.

## Разрешающие правила

Начнём с того, что разрешим только нужное. Пусть к нашему серверу должен иметь доступ компьютер с адресом `172.16.238.187` — и только к двум портам: `22` (SSH) и `80` (веб). Опишем это двумя правилами:

```shell
$ sudo iptables -A INPUT -p tcp -s 172.16.238.187 --dport 22 -j ACCEPT
$ sudo iptables -A INPUT -p tcp -s 172.16.238.187 --dport 80 -j ACCEPT
```

Прочитаем опции:

- `-A` (`--append`) добавляет правило в конец цепочки — здесь в `INPUT`;
- `-p` (`--protocol`) задаёт протокол, здесь `tcp`;
- `-s` (`--source`) задаёт адрес источника;
- `--dport` задаёт порт назначения (`--sport` — порт источника);
- `-j` (`--jump`) задаёт действие — что сделать с подходящим пакетом, здесь `ACCEPT` (пропустить).

Проверим, что правила появились:

```shell
$ sudo iptables -L
```

## Запрещающие правила

Сами по себе разрешающие правила пока ничего не дают: политика по умолчанию всё ещё `ACCEPT`, то есть проходит и всё остальное. Закроем дверь — запретим любые входящие пакеты, не подошедшие под правила выше:

```shell
$ sudo iptables -A INPUT -j DROP
```

И вот здесь критически важен порядок. `iptables` проверяет правила сверху вниз и останавливается на первом подходящем. Поэтому запрет «всё остальное» должен стоять **после** разрешающих правил — иначе он отбросит пакеты раньше, чем дело дойдёт до разрешений, и в систему не пройдёт никто.

## `DROP` и `REJECT`

Запретить пакет можно двумя действиями, и разница между ними не косметическая.

- `DROP` молча отбрасывает пакет — отправителю не приходит вообще никакого ответа. Со стороны это выглядит так, будто сервера не существует: соединение просто «висит» до таймаута.
- `REJECT` тоже не пропускает пакет, но вежливо отвечает отправителю сообщением об ошибке — «соединение отклонено». Отправитель сразу понимает, что порт закрыт.

`DROP` чаще выбирают для внешнего периметра (меньше подсказок потенциальному злоумышленнику), `REJECT` — внутри сети, где важнее быстрая и понятная диагностика.

## Исходящий трафик

Фильтровать можно не только входящий трафик, но и исходящий — для этого есть цепочка `OUTPUT`. Например, запретим серверу самому ходить на порт `80`:

```shell
$ sudo iptables -A OUTPUT -p tcp --dport 80 -j DROP
```

А вот обратная задача — разрешить серверу HTTPS-соединения с `google.com`:

```shell
$ sudo iptables -I OUTPUT -p tcp -d google.com --dport 443 -j ACCEPT
```

Обратите внимание на `-I` вместо `-A`. `-I` (insert) вставляет правило не в конец, а в начало цепочки. Это нужно как раз из-за порядка проверки: если ниже уже стоит запрещающее правило, добавленное в конец разрешение до пакета просто не доберётся.

## Удаление правила

Чтобы убрать правило, на него нужно как-то сослаться. Удобнее всего — по номеру строки, а чтобы их увидеть, добавьте к `-L` опцию `--line-numbers`:

```shell
$ sudo iptables -L --line-numbers
```

Теперь у каждого правила есть номер. Удалим, например, третье правило из цепочки `INPUT` опцией `-D` (delete):

```shell
$ sudo iptables -D INPUT 3
```

Имейте в виду: после удаления номера сдвигаются, поэтому перед каждым удалением список лучше смотреть заново.

## Несколько портов

Писать по отдельному правилу на каждый порт утомительно. Модуль `multiport` (подключается опцией `-m`) позволяет перечислить порты списком:

```shell
$ sudo iptables -A INPUT -p tcp -m multiport --dports 22,80,443 -j ACCEPT
$ sudo iptables -A OUTPUT -p tcp -m multiport --sports 22,80,443 -j ACCEPT
```

## ICMP и MAC

Фильтровать можно не только по портам. ICMP (Internet Control Message Protocol, протокол управляющих сообщений сети) — это служебный протокол, на котором работают `ping` и `traceroute`. Чтобы система перестала отвечать на `ping`, заблокируйте протокол `icmp` на нужном интерфейсе (`-i` задаёт входящий интерфейс):

```shell
$ sudo iptables -A INPUT -p icmp -i eth0 -j DROP
```

А с помощью модуля `mac` правило можно привязать к конкретному устройству по его MAC-адресу (Media Access Control — аппаратный адрес сетевого интерфейса, шесть пар шестнадцатеричных цифр):

```shell
$ sudo iptables -A INPUT -m mac --mac-source 0e:d5:8a:af:00:de -j DROP
```
