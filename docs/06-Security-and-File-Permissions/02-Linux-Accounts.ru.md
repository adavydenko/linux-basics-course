# Учётные записи Linux

- [Доп. видео](https://kodekloud.com/topic/linux-accounts/)

Прежде чем говорить о правах доступа к файлам, нужно ответить на более простой вопрос: а кто вообще работает в системе? Linux — многопользовательская система, и каждое действие выполняется от чьего-то имени. Разберёмся, где система хранит сведения о пользователях и группах, как узнать, кто есть кто, и как на время «стать» другим пользователем.

## Цели раздела

После раздела студент должен уметь:

- понимать, где хранятся сведения о пользователях и группах;
- читать базовую информацию из `/etc/passwd` и `/etc/group`;
- узнавать UID и GID пользователя;
- смотреть текущие входы в систему;
- переключаться между пользователями;
- понимать разницу между `su` и `sudo`.

## Учётные записи пользователей

Система не держит сведения о пользователях в какой-то скрытой базе данных — это обычные текстовые файлы, которые можно открыть и прочитать. Список всех учётных записей лежит в `/etc/passwd`:

```shell
$ cat /etc/passwd
```

А список групп — в `/etc/group`:

```shell
$ cat /etc/group
```

![user](../images/ru_user.svg)

У каждого пользователя есть имя и уникальный числовой идентификатор — UID (user ID). Имя удобно человеку, а ядро системы внутри различает пользователей именно по числу. Кроме того, у пользователя есть основная группа со своим идентификатором — GID (group ID). Быстрее всего посмотреть эти данные командой `id`:

```shell
$ id michael
uid=1001(michael) gid=1001(michael) groups=1001(michael),1003(developers)
```

В выводе видно сразу всё: UID пользователя, его основной GID и список всех групп, в которых он состоит. Здесь `michael` входит и в свою личную группу, и в группу `developers`.

Чтобы узнать домашний каталог и оболочку (shell), которая запускается при входе, загляните в `/etc/passwd` — но не перелистывая весь файл, а выбрав нужную строку через `grep`:

```shell
$ grep -i michael /etc/passwd
michael:x:1001:1001::/home/michael:/bin/sh
```

Поля в строке разделены двоеточиями: имя, метка пароля, UID, GID, комментарий, домашний каталог и оболочка. Здесь домашний каталог — `/home/michael`, а оболочка — `/bin/sh`. Подробный разбор этой структуры ждёт нас в разделе о файлах управления доступом.

Файлы говорят, кто вообще существует в системе. Но кто работает в ней прямо сейчас? Это покажет `who`:

```shell
$ who
bob pts/2 Apr 28 06:48 (172.16.238.187)
```

Каждая строка — это одна активная сессия: имя пользователя, терминал, время входа и адрес, с которого он подключился. Здесь `bob` зашёл по сети с адреса `172.16.238.187`.

Если же нужно заглянуть в прошлое — кто и когда входил раньше, когда система перезагружалась, — поможет `last`:

```shell
$ last
michael :1 :1 Tue May 12 20:00 still logged in
sarah :1 :1 Tue May 12 12:00 still running
reboot system boot 5.3.0-758-gen Mon May 11 13:00 - 19:00 (06:00)
```

Эта команда читает журнал входов и показывает историю целиком. Строки `still logged in` означают сессии, которые ещё не закрыты, а строка `reboot` отмечает момент загрузки системы. При расследовании инцидента именно `last` помогает понять, кто и когда был в системе.

## Переключение пользователей

Иногда нужно ненадолго стать другим пользователем — например, проверить, что видит он, или выполнить задачу от его имени. Самый прямой способ — команда `su` (substitute user, «подменить пользователя»). С дефисом она не просто меняет личность, но и подгружает окружение целевого пользователя, как при обычном входе:

```shell
$ su -
Password:
root ~#
```

Если нужно выполнить всего одну команду от чужого имени, не открывая целую сессию, пригодится `su -c`:

```shell
$ su -c "whoami"
Password:
root
```

Но для административных задач этот путь не лучший. `su` требует пароль того пользователя, в которого вы превращаетесь, — а значит, всем, кому нужны права администратора, пришлось бы знать пароль `root`. Поэтому рекомендуемый способ запускать команды с правами администратора — `sudo` (superuser do). Она просит ваш собственный пароль и выполняет лишь одну указанную команду:

```shell
$ sudo apt-get install nginx
[sudo] password for michael:
```

Кому именно разрешено пользоваться `sudo` и какие команды им доступны — описано в файле `/etc/sudoers` и связанных с ним файлах:

```shell
$ sudo cat /etc/sudoers
```

В файле каждая разрешающая строка устроена из четырёх полей, разделённых пробелами:

| Поле | Значение | Пример |
|:---:|---|---|
| 1 | пользователь или группа | `bob`, `%sudo` |
| 2 | хост, где правило действует | `localhost`, `ALL` |
| 3 | от чьего имени запускать команду | `(ALL)` |
| 4 | разрешённая команда | `/bin/ls`, `ALL` |

И ещё один штрих про учётную запись `root`. В типичном Ubuntu/Debian `root` в `/etc/passwd` сохраняет обычную оболочку `/bin/bash`, а интерактивный вход под ним блокируют по-другому: либо пустым паролем (`!` в `/etc/shadow`, проверяется командой `sudo passwd -S root`), либо настройкой `PermitRootLogin no` в `/etc/ssh/sshd_config`, либо комбинацией обоих. Назначить root оболочку `/usr/sbin/nologin` в `/etc/passwd` тоже технически можно, но в реальных дистрибутивах так почти не делают — это сломает часть служебных скриптов, ожидающих рабочую оболочку. Но получать права root рекомендованным способом — через `sudo` — при любом из этих вариантов всё равно можно:

```shell
$ grep -i ^root /etc/passwd
root:x:0:0:root:/root:/bin/bash
$ sudo passwd -S root
root L 05/12/2026 0 99999 7 -1
```

В выводе `passwd -S` буква `L` (locked) подтверждает, что прямой вход под `root` заблокирован.
