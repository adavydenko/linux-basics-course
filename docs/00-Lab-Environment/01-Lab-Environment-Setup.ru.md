# Настройка окружения для лабораторных работ

Лабораторные работы курса нужно выполнять руками в терминале. При этом многие задания меняют систему: создают и удаляют пользователей, правят права доступа, настраивают службы, работают с дисками и сетью. Выполнять такое на своём рабочем ноутбуке не стоит — ошибка может повредить систему или настройки.

Поэтому для лабораторных работ нужно отдельное, изолированное окружение, которое не жалко сломать. Ниже описаны два варианта: учебный контейнер и виртуальная машина.

## Зачем нужно изолированное окружение

- Ошибочная команда в учебной среде не затронет рабочую систему.
- Окружение можно в любой момент пересоздать «с нуля» и начать задание заново.
- У всех студентов одинаковый набор инструментов и версия дистрибутива, поэтому команды из пособия ведут себя предсказуемо.

Весь курс ориентирован на **Ubuntu 22.04 LTS**. Учебное окружение собрано на этой же версии.

## Вариант 1. Учебный контейнер (рекомендуется)

Контейнер подходит для большинства лабораторных работ: командная строка, файлы и каталоги, текстовая обработка, пакеты, пользователи и права, поиск, архивация, сеть, скрипты, Git и Python. Он запускается за секунды и легко пересоздаётся.

### Что понадобится

Установленный Docker. Проверить, что он работает:

```shell
$ docker --version
```

### Сборка образа

В каталоге `lab-environment` репозитория лежит файл `Dockerfile`. Соберите из него образ:

```shell
$ cd lab-environment
$ docker build -t linux-basics-lab .
```

### Запуск окружения

```shell
$ docker run -it --rm linux-basics-lab
```

После запуска вы окажетесь в оболочке Bash под учебным пользователем `student`. У этого пользователя есть права `sudo`, а сама среда одноразовая: с флагом `--rm` контейнер удаляется при выходе, и в следующий раз вы получаете чистую систему.

Чтобы сохранять файлы между запусками, подключите каталог с хост-машины:

```shell
$ docker run -it --rm -v "$PWD/lab-work:/home/student/lab-work" linux-basics-lab
```

Тогда всё, что вы положите в `~/lab-work` внутри контейнера, останется в каталоге `lab-work` на вашем компьютере.

### Ограничения контейнера

В контейнере **не получится** полноценно выполнить часть заданий, потому что они требуют доступа к ядру и оборудованию хоста:

- работа с дисками и разделами, LVM (модуль «Хранилища в Linux»);
- управление службами через `systemd` и `systemctl` (модуль «Управление службами»);
- настройка уровней запуска и загрузки системы.

Эти темы выполняйте в виртуальной машине (вариант 2).

## Вариант 2. Виртуальная машина

Виртуальная машина — это полноценная система с собственным ядром. Она нужна для лабораторных работ с дисками, LVM, загрузкой системы и службами `systemd`, но подходит и для всех остальных тем.

### Как подготовить

1. Установите гипервизор: VirtualBox, UTM (для macOS на Apple Silicon) или `multipass`.
2. Создайте виртуальную машину с **Ubuntu 22.04 LTS** (Server или Desktop).
3. Для лабораторных по хранилищам добавьте машине **второй, отдельный диск** — все эксперименты с разделами и LVM проводите только на нём, не трогая системный диск.
4. Сделайте снимок (snapshot) чистой системы сразу после установки. Если задание что-то сломает, машину можно вернуть к снимку и начать заново.

### Самый быстрый способ — multipass

Если нужна именно командная строка без графического интерфейса, удобнее всего `multipass`:

```shell
$ multipass launch 22.04 --name linux-lab
$ multipass shell linux-lab
```

Для лабораторных по хранилищам создайте машину с дополнительным диском (см. документацию `multipass`).

## Когда команды требуют `sudo`

Многие учебные команды читают и меняют системные файлы или настройки, доступные только администратору. Если команда из пособия завершается с ошибкой вида `Permission denied` или `Operation not permitted`, почти всегда причина в недостатке прав — выполните ту же команду с `sudo` в начале:

```shell
$ sudo <команда>
```

В пособии `sudo` указывается явно там, где он нужен почти всегда (например, установка пакетов или правка `/etc/hosts`). В остальных случаях исходите из правила: команды, которые только читают данные в вашем домашнем каталоге, прав администратора не требуют; команды, которые меняют систему целиком, — требуют.

## Формат отчёта по лабораторной работе

Чтобы преподаватель мог проверить работу, по каждой лабораторной сдаётся отчёт единого вида. Отчёт должен содержать:

1. **Цель работы** — одна-две строки своими словами.
2. **Ход работы** — список выполненных команд по порядку.
3. **Пояснения** — краткое объяснение, что делает каждая ключевая команда.
4. **Результат** — вывод терминала или скриншоты, подтверждающие, что задание выполнено.
5. **Ответы на контрольные вопросы** из соответствующего раздела.

Отчёт удобно вести как текстовый файл прямо в учебном окружении и сохранять его в подключённом каталоге `lab-work`.
