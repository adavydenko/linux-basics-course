# Диагностика среды разработки

- [Доп. лабораторная](https://kodekloud.com/topic/lab-troubleshoot-the-development-environment/)

> **Пререквизиты модуля.** Эта лабораторная задумана как интеграционный кейс — она использует темы сразу из нескольких модулей. Прежде чем браться, убедитесь, что прошли: модуль 02 (Shell I), модуль 05 (Shell II), модуль 06 (SSH), модуль 08 (хранилища), модуль 09 (systemd), модуль 15 (Git, как уметь сдавать решение) и модуль 16 (Python из CLI — `venv`, `requirements.txt`). Модули 15 и 16 в книге идут позже, но темы из них в этой лабораторной нужны — пройдите их заранее или просмотрите по диагонали.

Это та самая лабораторная, ради которой и затевался весь курс: здесь нет новой теории, зато есть сломанная среда, которую нужно довести до рабочего состояния. Вы пройдёте полный путь — перенесёте приложение на сервер, поднимете базу данных, найдёте и исправите ошибку в конфигурации, запустите приложение вручную, а потом оформите его как systemd-сервис, чтобы оно поднималось само. По дороге пригодятся `scp`, `ssh`, права доступа, `systemctl`, чтение логов и работа с виртуальным окружением Python — то есть почти всё, что вы уже изучили.

Действуйте по шагам и не торопитесь: на каждом этапе проверяйте результат, прежде чем идти дальше. Большая часть ошибок в таких сценариях ловится именно проверкой, а не догадками.

## Цели раздела

После раздела студент должен уметь:

- копировать архив на удалённый сервер;
- распаковывать приложение в `/opt`;
- проверять состояние PostgreSQL;
- редактировать конфигурационные файлы;
- диагностировать ошибку подключения приложения к БД;
- менять владельца файлов;
- активировать Python virtualenv;
- создавать systemd unit для приложения.

## Ход работы

1. Приложение лежит архивом на ноутбуке Боба — сначала его нужно доставить на сервер. Скопируйте `caleston-code.tar.gz` в домашний каталог Боба на web-сервере `devapp01` командой `scp` (secure copy):

   ```shell
   scp caleston-code.tar.gz devapp01:~/
   ```

2. Теперь распакуйте архив прямо на `devapp01`. Приложения принято размещать в `/opt`, но этот каталог принадлежит `root`, поэтому без `sudo` распаковать туда не получится:

   ```shell
   ssh devapp01
   sudo tar -zxf caleston-code.tar.gz -C /opt
   ```

3. Архив своё дело сделал — уберите его, чтобы не занимал место и не путался под руками:

   ```shell
   rm caleston-code.tar.gz
   ```

4. Прежде чем двигаться дальше, убедитесь, что распаковка прошла как надо и каталог приложения действительно на месте:

   ```shell
   ls -l /opt/caleston-code/mercuryProject/
   ```

5. Приложение есть, но без базы данных оно работать не будет. Перейдите на сервер БД `devdb01` и проверьте, в каком состоянии PostgreSQL:

   ```shell
   exit
   ssh devdb01
   systemctl status postgresql.service
   ```

   Смотрите на строку `Active`. Если там `Active: inactive (dead)`, база данных не запущена — это первая поломка, которую предстоит устранить.

6. Прежде чем запускать базу, разрешите приложению к ней подключаться. За правила доступа клиентов в PostgreSQL отвечает файл `pg_hba.conf`; добавьте в его конец строку, которая пускает подключения с любого адреса:

   ```text
   host all all 0.0.0.0/0 md5
   ```

   Файл системный, так что открывать его придётся через `sudo`:

   ```shell
   sudo vi /etc/postgresql/10/main/pg_hba.conf
   ```

7. Теперь, когда правила доступа на месте, можно поднимать саму базу:

   ```shell
   sudo systemctl start postgresql.service
   ```

8. База запущена — но приложению нужно знать, на каком порту её искать. Посмотрите, какой порт слушает PostgreSQL:

   ```shell
   sudo netstat -ptean
   ```

   `netstat` есть не во всех современных системах; там, где его нет, ту же информацию даст `ss` (socket statistics), а `grep` отфильтрует только нужные строки:

   ```shell
   sudo ss -tulpen | grep postgres
   ```

   Запомните найденный номер порта — он понадобится через пару шагов.

9. База готова, пора проверить само приложение. Вернитесь на `devapp01`, зайдите в каталог проекта и попробуйте запустить web-сервер вручную:

   ```shell
   exit
   ssh devapp01
   cd /opt/caleston-code/mercuryProject
   python3 manage.py runserver 0.0.0.0:8000
   ```

   Скорее всего, приложение не поднимется: оно выведет длинный stack trace и завершится. Это ожидаемо — поломка ещё не найдена. Остановите его через `Ctrl+C`.

10. Найдите причину ошибки.

    Читать stack trace нужно с конца — там самое конкретное сообщение. В нашем случае приложение жалуется, что не может подключиться к базе данных по указанному адресу и порту. А правильный порт вы уже видели на шаге 8 — сравните его с тем, что прописано в настройках.

11. Осталось понять, где эти настройки лежат. Параметры подключения к БД в Django хранятся в блоке `DATABASES`; найдите файл, в котором он встречается:

    ```shell
    find . -type f -exec grep -l 'DATABASES = {' "{}" \;
    ```

    Откройте найденный файл:

    ```shell
    vi ./mercury/settings.py
    ```

    В секции `DATABASES` исправьте `host` и `port` на правильные — те, на которых реально слушает PostgreSQL.

12. Приложение распаковывали под `root`, поэтому сейчас все его файлы принадлежат не тому пользователю. Передайте весь каталог приложения пользователю `mercury` — флаг `-R` (recursive) делает это для всех вложенных файлов и каталогов разом:

    ```shell
    sudo chown -R mercury /opt/caleston-code
    ```

13. Прежде чем запускать приложение, подготовьте окружение Python и обновите схему базы:

    ```shell
    source ../venv/bin/activate
    python3 manage.py migrate
    ```

    `venv` (virtual environment) — это виртуальное окружение Python: изолированный набор библиотек, отдельный от системного Python. Благодаря ему зависимости проекта не конфликтуют с системными пакетами, а команда `migrate` приводит структуру таблиц в базе к тому виду, который ожидает приложение.

14. Запускать приложение вручную каждый раз неудобно — оформите его как systemd-сервис, чтобы система поднимала его сама.

    Сначала остановите запущенное приложение через `Ctrl+C`. Затем узнайте полный путь к интерпретатору `python3` — он понадобится в unit-файле:

    ```shell
    which python3
    ```

    Создайте unit-файл:

    ```shell
    sudo vi /etc/systemd/system/mercury.service
    ```

    Содержимое. Если на предыдущем шаге `which python3` показал путь к Python внутри `venv`, используйте именно его:

    ```ini
    [Unit]
    Description=Project Mercury Web Application

    [Service]
    ExecStart=/opt/caleston-code/venv/bin/python3 manage.py runserver 0.0.0.0:8000
    Restart=on-failure
    WorkingDirectory=/opt/caleston-code/mercuryProject/
    User=mercury

    [Install]
    WantedBy=multi-user.target
    ```

15. Unit-файл написан, но systemd о нём ещё не знает. Перечитайте конфигурацию, включите сервис в автозапуск и запустите его:

    ```shell
    sudo systemctl daemon-reload
    sudo systemctl enable mercury
    sudo systemctl start mercury
    ```

16. И последнее — убедитесь, что всё действительно работает. `status` покажет, поднялся ли сервис, а `journalctl` — что он записал в журнал:

    ```shell
    systemctl status mercury
    journalctl -u mercury
    ```

    Если в строке `Active` стоит `active (running)` — среда восстановлена и готова к демонстрации.

## Технические замечания

- В unit-файлах лучше использовать абсолютные пути, потому что окружение systemd отличается от интерактивного shell.
- Для Python-приложений важно запускать интерпретатор из `venv`, иначе сервис может не увидеть зависимости проекта.
- После создания или изменения unit-файла нужен `systemctl daemon-reload`.
- Если сервис не запускается, сначала смотрите `systemctl status`, затем `journalctl -u`.
- `netstat` может отсутствовать в современных системах; используйте `ss`.
