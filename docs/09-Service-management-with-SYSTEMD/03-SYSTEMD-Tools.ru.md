# Инструменты systemd

- [Доп. видео](https://kodekloud.com/topic/systemd-tools/)

Unit-файл вы уже написать умеете — но управлять службой каждый день вы будете не через файлы, а через две команды. `systemctl` (system control) отдаёт службам приказы: запусти, останови, покажи статус. `journalctl` (journal control) показывает, что эти службы записали в журнал. Первая отвечает на вопрос «работает ли?», вторая — «а что именно произошло?». Эта пара покрывает почти всю повседневную работу с systemd.

## Цели раздела

После раздела студент должен уметь:

- запускать, останавливать и перезапускать сервисы;
- включать и отключать автозапуск;
- смотреть статус сервиса;
- смотреть и редактировать unit-файлы;
- читать логи через `journalctl`.

## `systemctl`

`systemctl` — главная команда администратора на сервере с systemd. Через неё проходит почти всё управление службами, и круг её задач удобно держать в голове как четыре группы:

- управление жизненным циклом службы прямо сейчас: `start`, `stop`, `restart`, `reload`;
- управление автозапуском при загрузке: `enable`, `disable`;
- просмотр и управление unit'ами — какие службы вообще есть в системе и в каком они состоянии;
- просмотр и смена targets — целей загрузки, то есть режима, в котором работает система.

## Частые команды `systemctl`

Самое частое действие — запустить службу. Здесь и дальше в примерах используется `docker`, но на его месте может стоять имя любого сервиса:

```shell
$ sudo systemctl start docker
```

Зеркальная команда останавливает службу — процесс завершается, порты освобождаются:

```shell
$ sudo systemctl stop docker
```

`restart` — это `stop` и `start` одной командой; так перечитывают конфигурацию или выводят службу из зависшего состояния:

```shell
$ sudo systemctl restart docker
```

Иногда полный перезапуск избыточен: некоторые службы умеют перечитать конфигурацию на лету, не разрывая соединений. За это отвечает `reload` — но только если служба его поддерживает:

```shell
$ sudo systemctl reload docker
```

Все команды выше действуют только до перезагрузки. Чтобы служба поднималась при каждой загрузке системы, её нужно включить:

```shell
$ sudo systemctl enable docker
```

И обратное действие — убрать службу из автозапуска (при этом она продолжит работать, просто не стартует после перезагрузки):

```shell
$ sudo systemctl disable docker
```

Прежде чем что-то менять, полезно узнать, в каком состоянии служба сейчас. `status` показывает это подробно — и, в отличие от команд выше, не требует `sudo`:

```shell
$ systemctl status docker
```

В выводе обращайте внимание на строку `Active`. У исправно работающей службы там стоит `active (running)`; `inactive (dead)` означает, что служба остановлена, а `failed` — что она пыталась запуститься, но упала. Ниже status показывает несколько последних строк журнала — часто этого уже достаточно, чтобы понять причину.

Сводно по основным состояниям:

| Состояние   | Что это значит                                          |
|-------------|---------------------------------------------------------|
| `active`    | Служба работает                                         |
| `inactive`  | Служба остановлена                                      |
| `failed`    | Сбой: упала, не запустилась, истёк тайм-аут             |

Отдельно стоит запомнить: если вы правили unit-файл, systemctl продолжит работать со старой версией, пока вы не попросите перечитать конфигурацию:

```shell
$ sudo systemctl daemon-reload
```

Открыть unit-файл на редактирование можно прямо через systemctl — флаг `--full` показывает файл целиком, а не пустую заготовку для переопределений:

```shell
$ sudo systemctl edit project-mercury.service --full
```

Перейдём к targets — целям загрузки. Узнать, в каком режиме система стартует по умолчанию:

```shell
$ systemctl get-default
```

И сменить этот режим — например, перевести сервер из графического в текстовый многопользовательский, чтобы не тратить ресурсы на ненужный рабочий стол:

```shell
$ sudo systemctl set-default multi-user.target
```

Наконец, две команды для обзора. Эта показывает вообще все unit'ы, которые система загрузила, включая остановленные:

```shell
$ systemctl list-units --all
```

А без `--all` список сокращается до активных — обычно с него и стоит начинать:

```shell
$ systemctl list-units
```

Если нужно быстро посмотреть, что написано в unit-файле службы и где он лежит, не открывая редактор:

```shell
$ systemctl cat project-mercury.service
```

## `journalctl`

`systemctl status` отвечает на вопрос «работает ли служба», но когда что-то сломалось, нужен другой вопрос — «что именно произошло». На него отвечает журнал.

В системах с systemd логи собирает специальная служба — `systemd-journald`. Она складывает в единый журнал сообщения ядра, вывод всех служб systemd и другие источники. Поэтому вместо того, чтобы искать нужный лог-файл по всему `/var/log`, вы обращаетесь к одному инструменту — `journalctl`.

Запущенный без аргументов, `journalctl` вываливает весь журнал — от самых старых записей к свежим. Полезно знать, но на практике так почти не делают: записей слишком много.

```shell
$ journalctl
```

Флаг `-b` (boot) сужает вывод до текущей загрузки — всё, что произошло с момента включения машины. Удобно, когда нужно понять, что пошло не так при старте системы:

```shell
$ journalctl -b
```

Чаще всего вас интересует конкретная служба. Флаг `-u` (unit) оставляет в выводе только её записи:

```shell
$ journalctl -u docker.service
```

А если проблема случилась в известный момент, журнал можно обрезать по времени флагом `--since` — так вы не будете листать историю за недели:

```shell
$ journalctl -u docker.service --since "2022-01-01 13:45:00"
```

> **Типичные ошибки.**
>
> - **Поправил unit-файл, перезапустил — изменения не подхватываются.** systemctl держит в памяти разобранную копию конфигурации. После любой правки `.service` или `.timer` сначала `sudo systemctl daemon-reload`, и только потом `restart`. Это самый частый «странный баг» с systemd.
> - **`start` без `enable`.** Команда `start` поднимает службу здесь и сейчас, но после перезагрузки её надо поднимать заново. Если служба должна работать всегда — добавляйте `enable` (или используйте `enable --now`, что делает обе операции одной командой).
> - **Подсмотрел `journalctl -u` и не увидел свежих записей.** По умолчанию `journalctl` показывает «от старых к новым»; на длинных журналах нужное часто внизу. Полезные флаги: `-e` (открыть в конце), `-n 50` (последние 50 строк), `-f` (следить за новыми записями в реальном времени).
> - **`systemctl status` показывает `active (exited)` — это сломалось?** Нет, это нормальное состояние для unit'ов с `Type=oneshot`: служба сделала своё дело и завершилась. «Сломанное» состояние — `failed`.

## Самопроверка

- [ ] могу одной командой запустить службу и одновременно включить её в автозапуск;
- [ ] прочту вывод `systemctl status` и скажу, в каком состоянии служба и почему;
- [ ] объясню, чем `restart` отличается от `reload`, и приведу пример, где `reload` уместнее;
- [ ] найду в журнале конкретной службы записи за последний час;
- [ ] прочту незнакомый unit-файл и пойму, какую команду запустит `ExecStart` и под каким пользователем;
- [ ] объясню, зачем нужен `daemon-reload`, и в каких случаях без него ничего не сработает.

## Практическое задание

1. Посмотрите статус любого сервиса.
2. Посмотрите его unit-файл через `systemctl cat`.
3. Посмотрите логи через `journalctl -u`.
4. Найдите target по умолчанию.
5. Выведите список активных units.

## Возврат к Бобу

> **Возврат к Бобу.** В понедельник утром Боб обнаруживает, что внутренний веб-сервис не отвечает. На сервере он запускает `systemctl status mercury.service` и видит состояние `failed`. В выводе мелькает строка «Start request repeated too quickly». В пятницу коллега правил unit-файл — добавлял переменные окружения. Опишите план диагностики: какие команды Боб выполнит по порядку, чтобы понять, что именно сломалось, и где он найдёт первую причину? Подсказка: задействуйте `systemctl status`, `systemctl cat`, `journalctl -u` и `daemon-reload`.
