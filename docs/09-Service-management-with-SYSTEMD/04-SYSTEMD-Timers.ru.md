# Таймеры systemd

В разделе про `cron` (модуль 06) мы видели, как запускать задачу по расписанию: добавить строку в crontab — и служба `cron` сама вызовет её в нужное время. В системах с systemd есть собственный механизм для того же — `.timer` юниты. Он мощнее cron, лучше интегрируется с журналом systemd и постепенно становится стандартом для повторяющихся задач на современных серверах.

## Цели раздела

После раздела студент должен уметь:

- объяснить, чем `.timer` отличается от cron;
- читать `.timer`-юнит;
- написать пару «`.service` + `.timer`» для повторяющейся задачи;
- проверять, что таймер запланирован, и смотреть его выполнение в журнале.

## Зачем нужны таймеры

Cron делает свою работу хорошо, но у него есть слабые места:

- логи cron-задач никуда не пишутся «из коробки» — нужно явно перенаправлять в файлы;
- если задача не запустилась (например, машина была выключена), cron её не повторит — расписание двигается дальше;
- условия «после старта системы», «после загрузки сети», «только если другой юнит активен» — за пределами возможностей cron;
- ресурсы (CPU, память) cron-задачам не ограничить.

Таймеры systemd закрывают все эти точки. Кроме того, у них есть одно концептуальное удобство: расписание (`.timer`) и команда (`.service`) — это два отдельных юнита. Можно один и тот же сервис запускать вручную и по расписанию, а в логах будет видна каждая попытка отдельно.

## Анатомия `.timer`-юнита

`.timer`-юнит — это файл с тремя секциями, две из которых вам уже знакомы по `.service`:

```ini
[Unit]
Description=Ежедневная очистка временных файлов

[Timer]
OnCalendar=daily
Persistent=true
Unit=cleanup.service

[Install]
WantedBy=timers.target
```

Разберём поля секции `[Timer]`, ради которой всё и затевалось:

- `OnCalendar` — расписание. Принимает либо «человеческие» ключевые слова (`daily`, `weekly`, `monthly`, `hourly`), либо календарные выражения в собственном формате systemd (см. ниже).
- `Persistent=true` — «догнать пропущенный запуск». Если на момент срабатывания таймера машина была выключена, после её загрузки systemd сразу выполнит команду один раз. Без этой опции пропущенный запуск просто теряется.
- `Unit` — какой `.service` запускать. Если опустить, systemd возьмёт `.service` с тем же именем, что и таймер: для `cleanup.timer` — `cleanup.service`. Указывать поле явно стоит, когда из таймера запускается другой по имени сервис.

Секция `[Install]` определяет, к какой цели «подключить» таймер при `systemctl enable`. Каноническая цель — `timers.target`, она входит в обычную загрузку.

## Формат `OnCalendar`

Календарные выражения systemd — это шаблон вида:

```text
ДЕНЬ_НЕДЕЛИ ГГГГ-ММ-ДД ЧЧ:ММ:СС
```

Любое поле можно опустить (тогда подразумевается «любой») или указать диапазон/список. Несколько примеров:

| Выражение | Когда срабатывает |
|-----------|-------------------|
| `*-*-* 03:00:00` | каждый день в 03:00 |
| `Mon..Fri 09:00` | будни, в 09:00 |
| `Sun *-*-* 04:00` | каждое воскресенье в 04:00 |
| `*-*-01 00:00:00` | первого числа каждого месяца, в полночь |
| `*-*-* *:00,30:00` | каждый час и каждые полчаса (`:00`, `:30`) |
| `2026-01-01 00:00:00` | один раз — в новогоднюю полночь 2026 года |

Проверить, что выражение разбирается правильно и когда оно сработает ближайшие N раз — командой `systemd-analyze`:

```shell
$ systemd-analyze calendar 'Mon..Fri 09:00'
Original form: Mon..Fri 09:00
Normalized form: Mon..Fri *-*-* 09:00:00
Nearest elapse: Wed 2026-05-20 09:00:00 MSK
       From now: 18h left
```

Эта же команда — самый быстрый способ отладки расписания.

## Разобранный пример: ежедневная резервная копия

Соберём всё вместе. Задача: каждую ночь в `02:30` запускать архивирование `/var/www` в `/backup/`.

**Шаг 1. Service-юнит.** Создаём `/etc/systemd/system/www-backup.service`:

```ini
[Unit]
Description=Backup /var/www to /backup/

[Service]
Type=oneshot
ExecStart=/usr/bin/tar -czf /backup/www-%Y-%m-%d.tar.gz /var/www/
User=root
```

`Type=oneshot` отличает «однократную задачу с завершением» от долгоживущей службы. После выполнения systemd корректно зафиксирует, что юнит «отработал», и его `systemctl status` будет показывать `inactive (dead)` до следующего запуска. Подстановки `%Y-%m-%d` системой не интерпретируются — это сложнее, чем в cron, поэтому на практике подстановку даты обычно делают внутри отдельного скрипта (`/usr/local/bin/backup.sh`), который и вызывается из `ExecStart`.

**Шаг 2. Timer-юнит.** Создаём `/etc/systemd/system/www-backup.timer`:

```ini
[Unit]
Description=Daily timer for /var/www backup

[Timer]
OnCalendar=*-*-* 02:30:00
Persistent=true

[Install]
WantedBy=timers.target
```

**Шаг 3. Подключаем.** После создания файлов перечитываем конфигурацию systemd и включаем таймер:

```shell
$ sudo systemctl daemon-reload
$ sudo systemctl enable --now www-backup.timer
```

Флаг `--now` сразу же активирует таймер, без него `enable` лишь поставит его в автозапуск.

**Проверка.** Должны увидеть свой таймер в общем списке и время ближайшего срабатывания:

```shell
$ systemctl list-timers
NEXT                        LEFT     LAST                        PASSED  UNIT                 ACTIVATES
Tue 2026-05-20 02:30:00 MSK 11h left Mon 2026-05-19 02:30:00 MSK 12h ago www-backup.timer     www-backup.service
```

Когда таймер сработает, в журнале появится запись:

```shell
$ journalctl -u www-backup.service --since today
```

## `OnBootSec` и `OnUnitActiveSec`: интервалы вместо календаря

Кроме `OnCalendar` есть «относительные» триггеры:

- `OnBootSec=15min` — через 15 минут после загрузки системы;
- `OnStartupSec=1h` — через час после запуска самого systemd;
- `OnUnitActiveSec=30min` — каждые 30 минут с момента последнего запуска связанного сервиса;
- `OnUnitInactiveSec=10min` — через 10 минут после завершения предыдущего запуска.

`OnUnitActiveSec` особенно удобен для задач «каждые N минут от старта системы»: cron такого формата не предлагает.

## Когда выбрать таймер, а когда cron

Cron всё ещё уместен, когда:

- задача простая и независимая, лежит в crontab пользователя;
- нет нужды в журнале (например, скрипт сам пишет логи);
- расписание укладывается в стандартные пять полей.

Systemd-таймер уместнее, когда:

- задача — часть сервиса, описанного юнитом;
- важно «догнать» пропущенный запуск (`Persistent=true`);
- хочется ограничить ресурсы (`CPUQuota=`, `MemoryMax=` в `[Service]`);
- задача должна стартовать после конкретного события (`After=` другие юниты);
- расписание сложнее, чем умеет cron.

## Типичные ошибки

- **Забыли `daemon-reload` после правки.** Systemd видит старую версию юнита, новые поля не подхватываются.
- **`Type=oneshot` упущен.** Без него `tar`/скрипт завершается, и systemd считает службу упавшей, потому что «процесс ушёл» — статус `failed`.
- **Подстановки `%Y-%m-%d` в `ExecStart`.** Systemd их не понимает; кладите команду в отдельный скрипт.
- **`Persistent=true` без `OnCalendar`.** Опция работает только с календарными триггерами, не с интервальными.
- **Запуск таймера, но не сервиса.** Если попытались `systemctl start www-backup.service` руками и получили ошибку — таймер тут ни при чём, исправляйте сервис.
