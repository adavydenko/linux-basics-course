# Создание собственного systemd-сервиса

- [Доп. видео](https://kodekloud.com/topic/creating-a-systemd-service/)

Чтобы система взяла приложение под присмотр, ей нужно объяснить три вещи: что это за служба, какую команду запускать и когда поднимать её автоматически. Всё это описывается в одном текстовом файле — service unit. В этом разделе мы напишем такой файл с нуля и запустим собственный сервис.

## Цели раздела

После раздела студент должен уметь:

- объяснить, что такое systemd;
- понимать структуру `.service` unit-файла;
- описать секции `[Unit]`, `[Service]`, `[Install]`;
- перезагружать конфигурацию systemd;
- запускать собственный сервис.

Прежде чем писать unit-файл, стоит понять, кому мы его адресуем. Systemd (system daemon) — это система инициализации и менеджер сервисов Linux: первый процесс, который ядро запускает после загрузки, и он же дальше отвечает за все остальные службы. Большинство крупных дистрибутивов — RHEL, CentOS, Fedora, Ubuntu, Debian, Arch Linux — давно перешли на systemd, так что навык работы с ним переносится почти на любой современный сервер.

Systemd занимается не только запуском демонов по требованию. Он управляет точками монтирования и автомонтирования, собирает логи, отслеживает зависимости между службами и решает массу других задач администрирования. Для нас сейчас важна одна его роль — менеджер сервисов.

## Что такое service unit

Unit — это любой объект, которым управляет systemd: служба, точка монтирования, таймер, цель загрузки. Нас интересует один вид unit'ов — служба, и описывается она файлом с суффиксом `.service`. По сути это анкета приложения: systemd читает её и узнаёт всё, что нужно для управления процессом. Файл состоит из трёх основных секций, и дальше мы разберём каждую.

## `[Unit]`

Секция `[Unit]` отвечает на вопрос «что это за служба и с чем она связана»: здесь задаётся человекочитаемое описание, ссылка на документацию и зависимости от других unit'ов.

Пример:

```ini
[Unit]
Description=Python Django for Project Mercury
Documentation=http://wiki.caleston-dev.ca/mercury
After=postgresql.service
```

Разберём поля:

- `Description` — краткое описание сервиса. Именно эту строку вы увидите потом в выводе `systemctl status`, так что писать стоит понятно;
- `Documentation` — ссылка на документацию: куда заглянуть, если со службой что-то не так;
- `After` — указывает, что этот unit нужно запускать только после перечисленных. В примере приложение зависит от базы данных, поэтому `After=postgresql.service` гарантирует, что Django стартует, когда PostgreSQL уже поднят.

## `[Service]`

Если `[Unit]` отвечает на вопрос «что это», то `[Service]` — на вопрос «как это запускать»: какую именно команду выполнить, от чьего имени и что делать, если процесс упадёт.

```ini
[Service]
ExecStart=/usr/bin/project-mercury.sh
User=project_mercury
Restart=on-failure
RestartSec=10
```

Разберём поля:

- `ExecStart` — команда, которую systemd выполнит при запуске службы. Путь к исполняемому файлу лучше указывать абсолютным: окружение systemd отличается от вашего интерактивного shell, и привычный `PATH` тут может не сработать;
- `User` — от имени какого пользователя работает сервис. Отдельный непривилегированный пользователь — это страховка: даже если приложение взломают, злоумышленник не получит права `root`;
- `Restart` — политика перезапуска. Значение `on-failure` означает «подними службу заново, если она завершилась с ошибкой» — то самое автоматическое восстановление, ради которого мы и затевали unit-файл;
- `RestartSec` — пауза перед повторным запуском. Без неё systemd при постоянных падениях будет перезапускать процесс в бесконечном цикле; здесь между попытками проходит 10 секунд.

## `[Install]`

Последняя секция, `[Install]`, нужна только для одного — чтобы служба запускалась сама при загрузке системы. Она вступает в дело, когда вы выполняете `systemctl enable`.

```ini
[Install]
WantedBy=multi-user.target
```

Поле `WantedBy` указывает, к какой цели загрузки «подвесить» службу. На сервере типичный выбор — `multi-user.target` (состояние «система загружена, многопользовательский режим, сеть работает»); для рабочей станции с графической оболочкой берут `graphical.target`. В нашем учебном сценарии — серверное приложение Project Mercury — подходит как раз `multi-user.target`.

## Запуск сервиса

Файл написан, но systemd о нём ещё не знает — конфигурацию он читает только при старте и по явной команде. Поэтому после создания или правки unit-файла его нужно перечитать:

```shell
$ sudo systemctl daemon-reload
```

Теперь можно запустить сервис вручную и проверить, что он вообще поднимается:

```shell
$ sudo systemctl start project-mercury.service
```

Команда `start` запускает службу здесь и сейчас, но после перезагрузки её придётся поднимать заново. Чтобы systemd делал это сам при каждой загрузке, службу нужно «включить» — и вот тут срабатывает секция `[Install]`:

```shell
$ sudo systemctl enable project-mercury.service
```

Запомните эту пару: `start` — «запусти сейчас», `enable` — «запускай всегда». Обычно для боевого сервиса нужно и то, и другое.

## Разобранный пример: оборачиваем Python-скрипт в systemd-сервис

**Задача.** Команда написала маленький скрипт `/usr/local/bin/metrics.py`, который раз в N секунд опрашивает базу и пишет метрики в файл. Скрипт нужно превратить в системную службу: запускать от непривилегированного пользователя, автоматически перезапускать при сбое и поднимать при загрузке системы.

**Решение.**

1. Заводим отдельного пользователя без оболочки и без домашнего каталога — на случай взлома: `sudo useradd --system --no-create-home --shell /usr/sbin/nologin metrics`. Файлы скрипта оставляем доступными для чтения и выполнения этому пользователю.
2. Создаём unit-файл `/etc/systemd/system/metrics.service`:

   ```ini
   [Unit]
   Description=Metrics collector for Project Mercury
   After=network-online.target

   [Service]
   ExecStart=/usr/bin/python3 /usr/local/bin/metrics.py
   User=metrics
   Restart=on-failure
   RestartSec=5

   [Install]
   WantedBy=multi-user.target
   ```

   Путь к `python3` — абсолютный: окружение systemd скудное, `PATH` в нём другой, и просто `python3` может «не найтись».
3. Сообщаем systemd, что появился новый unit: `sudo systemctl daemon-reload`. Без этого шага следующая команда ничего не найдёт.
4. Запускаем сервис прямо сейчас: `sudo systemctl start metrics.service`. Сразу же смотрим статус: `systemctl status metrics.service` — строка `Active` должна быть `active (running)`.
5. Включаем автозапуск при следующих загрузках: `sudo systemctl enable metrics.service`. Можно объединить «запустить + включить» одной командой: `sudo systemctl enable --now metrics.service`.

**Проверка.** `systemctl is-active metrics.service` отвечает `active`, `systemctl is-enabled metrics.service` — `enabled`. После `sudo reboot` (если это учебная машина, не рабочая!) сервис должен подняться сам. Журнал — через `journalctl -u metrics.service`.
