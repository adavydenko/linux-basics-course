# Итоговый проект

Весь курс вы осваивали инструменты по отдельности: файлы, права, процессы, сеть, скрипты, контейнеры. Итоговый проект существует, чтобы свести их в одно целое — реальная работа не складывается из изолированных команд, она складывается из связок, которые решают задачу от начала до конца. Здесь вы пройдёте такой путь самостоятельно: подготовите Linux-среду, запустите в ней задачу, сохраните результаты и оформите всё так, чтобы запуск можно было воспроизвести другому человеку без вашего вмешательства.

## Цель проекта

Показать главное: что вы воспринимаете Linux не как набор команд, которые нужно вспоминать поодиночке, а как рабочую среду — место, где учебная или вычислительная задача живёт целиком, от подготовки до результата.

## Сценарий

Вам нужно подготовить проект на Linux-сервере или виртуальной машине. Шаги ниже — это не жёсткий рецепт, а каркас: каждый из них опирается на отдельный модуль курса, и вместе они складываются в законченный рабочий цикл.

1. Создать Git-репозиторий проекта.
2. Подготовить структуру каталогов.
3. Создать или получить небольшой набор текстовых данных.
4. Написать shell-скрипт обработки данных.
5. При необходимости запустить Python-скрипт из командной строки в `.venv`.
6. Запустить задачу так, чтобы stdout и stderr сохранялись в логи.
7. Проверить процессы и ресурсы.
8. Сохранить результаты.
9. Подготовить диагностический отчёт.
10. Описать, как повторить запуск с нуля.

## Минимальная структура проекта

Аккуратная структура каталогов — это не косметика, а часть воспроизводимости: проверяющий должен с первого взгляда понимать, где код, где данные, а где результаты. Ниже — минимальный каркас, от которого стоит отталкиваться:

```text
project/
  README.md
  scripts/
  data/
  results/
  logs/
  requirements.txt
```

`requirements.txt` нужен только если в проекте используется Python; в чисто shell-проекте его можно не создавать.

## Обязательные требования

Эти пункты — не пожелания, а условия зачёта. Все они служат одной идее: проект должен запускаться не только у вас и не только сегодня.

- Все команды запуска описаны в `README.md`.
- В проекте есть `.gitignore`.
- Временные файлы, `.venv`, логи и большие результаты не коммитятся без необходимости.
- Скрипт можно запустить повторно.
- stdout и stderr сохраняются отдельно или явно описано, почему это не требуется.
- В отчёте есть команды диагностики: `df`, `free`, `ps` или `top`, `journalctl`/`dmesg` при необходимости.

## Варианты проекта

Жёстко заданной темы нет — выберите тот сценарий, который вам ближе. Любой из них даёт достаточно материала, чтобы показать все требуемые навыки; важна не экзотичность задачи, а аккуратность её исполнения.

1. Обработка CSV-файла через пайплайн команд оболочки и `awk`.
2. Анализ логов: частые ошибки, топ IP-адресов, динамика по времени.
3. Запуск Python-скрипта обработки данных в `.venv`.
4. Запуск простого сервиса в контейнере и проверка через `curl`.
5. Сравнение двух наборов результатов и оформление отчёта.

## Критерии оценки

- Воспроизводимость: другой студент может повторить шаги по README.
- Аккуратность Git-истории и `.gitignore`.
- Корректное использование команд Linux.
- Наличие логов и результатов.
- Понимание ресурсов и диагностики.
- Краткое объяснение выбранных команд.

## Что нужно сдать

- Ссылку на Git-репозиторий или архив проекта.
- `README.md` с инструкцией запуска.
- Скрипты в `scripts/`.
- Небольшой пример входных данных.
- Результаты работы.
- Диагностический отчёт.
