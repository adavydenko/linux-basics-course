# Git из командной строки

В предыдущем модуле мы научились заходить на удалённый сервер и работать там. Но рано или поздно появится вопрос: «а как поделиться кодом и историей изменений?» — отправлять ZIP-архивы по почте на третий день становится мучением. Здесь и нужен Git: распределённая система контроля версий, которая помнит каждую правку, умеет переключаться между состояниями проекта и синхронизировать его между разными машинами. В этом курсе достаточно базового набора действий из командной строки — именно его и разберём; полноценное освоение Git (ветки, конфликты, rebase) — отдельный курс.

## Цели раздела

После раздела студент должен уметь:

- клонировать репозиторий;
- смотреть состояние рабочего каталога;
- добавлять файлы в индекс;
- создавать коммиты;
- получать и отправлять изменения;
- использовать `.gitignore`.

## Базовый набор действий

Большинство учебных задач укладывается в один и тот же цикл: получить репозиторий, поработать, зафиксировать изменения, отправить их обратно. Разберём этот цикл по шагам.

Чаще всего вы начинаете не с пустого места, а с уже существующего репозитория. Команда `git clone` скачивает его целиком — вместе со всей историей — в новый каталог:

```shell
$ git clone https://example.com/course/lab.git
$ cd lab
```

Дальше — главная команда, к которой стоит обращаться постоянно. `git status` показывает, что сейчас происходит в рабочем каталоге: какие файлы изменены, какие готовы к коммиту, какие Git ещё не отслеживает.

```shell
$ git status
```

`status` говорит, *какие* файлы изменились, а `git diff` показывает, *что именно* в них изменилось — построчно, старое против нового:

```shell
$ git diff
```

Git не коммитит изменения автоматически — сначала вы явно отбираете то, что войдёт в следующий коммит. Это делает `git add`, добавляя файл в так называемый индекс (staging area, «область подготовки»):

```shell
$ git add report.md
```

Когда нужные файлы отобраны, `git commit` записывает их в историю как одну точку, к которой потом можно вернуться. Флаг `-m` задаёт сопроводительное сообщение — короткое описание, что и зачем вы сделали:

```shell
$ git commit -m "Add lab report"
```

Коммиты пока живут только на вашей машине. Чтобы синхронизироваться с общим репозиторием, есть пара зеркальных команд: `git pull` забирает чужие изменения к вам, `git push` отправляет ваши коммиты наружу. Перед началом работы обычно делают `pull`, после — `push`:

```shell
$ git pull
$ git push
```

> **Типичные ошибки.**
>
> - **`git add .` после долгой работы.** В индекс заодно попадают временные файлы, кэш редактора, секреты в `.env`. Привычка делать `git status` *перед* `add` и `add` отдельными файлами экономит часы на чистке истории.
> - **Коммит без сообщения или с сообщением `fix`.** Через месяц вы не вспомните, что именно «fix». В сообщении полезно отвечать на вопрос «зачем», а не «что» — `diff` и так покажет, *что* поменялось.
> - **`git push --force` в общий репозиторий.** Перезаписывает чужую историю. Сначала `git pull --rebase`, потом обычный `push`. `--force` оставьте для своих личных веток, и то аккуратно.
> - **Секреты в коммите.** Пароль, токен или приватный ключ, попавший хоть в один коммит, считается утечкой — даже если вы потом удалили файл. История Git помнит всё. Решение — `.gitignore` *до* первого `add` и регулярная проверка `git status` на «что-то лишнее».

## `.gitignore`

Не всё, что лежит в каталоге проекта, должно попадать в репозиторий. Временные файлы, многомегабайтные результаты вычислений, локальные виртуальные окружения и — особенно — секреты вроде паролей и ключей в историю Git попадать не должны. Чтобы Git их просто не замечал, в проект кладут файл `.gitignore` со списком того, что игнорировать.

Пример `.gitignore`:

```gitignore
.venv/
__pycache__/
*.log
*.tmp
results/
.DS_Store
```

## Разобранный пример: первый коммит учебного проекта

**Задача.** Вы начинаете лабораторную: каталог пуст, нужно создать репозиторий, добавить `README.md` и `.gitignore`, сделать осмысленный первый коммит и отправить его в удалённый репозиторий, который преподаватель уже создал на учебном GitLab.

**Решение.**

1. Инициализировать репозиторий в каталоге проекта:
   ```shell
   $ cd ~/work/lab-01
   $ git init
   ```
2. *До* первого `add` создать `.gitignore`. Это важный порядок: если временные файлы попадут в первый коммит, потом их придётся выкорчёвывать из истории:
   ```shell
   $ cat > .gitignore <<'EOF'
   .venv/
   __pycache__/
   *.log
   results/
   EOF
   ```
3. Добавить `README.md` с описанием задачи и одной командой запуска:
   ```shell
   $ echo "# Lab 01: log analysis" > README.md
   ```
4. Проверить состояние — должно быть видно ровно два файла, ничего лишнего:
   ```shell
   $ git status
   ```
5. Добавить файлы в индекс по одному и зафиксировать. Перечисление имён, а не `git add .`, защищает от случайного попадания мусора:
   ```shell
   $ git add README.md .gitignore
   $ git commit -m "Initialize lab 01: README, gitignore"
   ```
6. Привязать удалённый репозиторий и отправить коммит. Имя `origin` — стандартное соглашение для «основного» удалённого репозитория:
   ```shell
   $ git remote add origin git@lab.example.com:student/lab-01.git
   $ git push -u origin main
   ```
   Флаг `-u` запоминает связь между локальной веткой и удалённой, чтобы дальше работали короткие `git pull` и `git push`.

**Проверка.** `git log --oneline` показывает один коммит с осмысленным сообщением. `git status` сообщает `working tree clean`. На странице репозитория в GitLab видны те же два файла; временных файлов и `.venv/` там нет.

## Хороший коммит

История репозитория полезна ровно настолько, насколько аккуратно сделаны коммиты. Через полгода — или для проверяющего вашу лабораторную — история должна читаться как осмысленный рассказ, а не свалка. Хороший коммит:

- относится к одной задаче — не «много всякого за день», а одно завершённое изменение;
- имеет понятное сообщение, по которому видно, что и зачем сделано;
- не содержит случайных временных файлов — за это как раз отвечает `.gitignore`;
- может быть проверен через `git diff` — то есть содержит ровно те изменения, которые вы намеревались внести.

## Практическое задание

1. Создайте новый репозиторий.
2. Добавьте `README.md`.
3. Создайте `.gitignore`.
4. Сделайте первый коммит.
5. Измените `README.md`, посмотрите `git diff`.
6. Сделайте второй коммит.
