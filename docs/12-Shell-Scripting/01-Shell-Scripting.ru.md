# Скрипты для оболочки

В предыдущем модуле мы научились собирать конвейеры из команд: `cat | grep | sort | uniq -c`. Это уже мини-программа, только живущая внутри одной командной строки. Рано или поздно вы поймаете себя на том, что вводите одну и ту же цепочку по третьему разу: создать каталоги, разложить файлы, прогнать обработку, проверить лог. Как только последовательность устоялась, её стоит записать в файл — и запускать одной командой. Такой файл с командами оболочки (shell), который выполняется как программа, и называется shell-скриптом. Скрипты избавляют от рутины, исключают опечатки и, что важно для учёбы, делают вашу работу воспроизводимой: тот же скрипт у другого человека даст тот же результат.

## Цели раздела

После раздела студент должен уметь:

- создавать простой shell-скрипт;
- передавать аргументы в скрипт;
- использовать переменные, условия и циклы;
- проверять код завершения команды;
- писать скрипт, который можно повторно запускать и читать через неделю.

## Первый скрипт

Начнём с самого маленького работающего скрипта. Создайте файл `hello.sh`:

```shell
#!/usr/bin/env bash

echo "Hello, Linux"
```

Сам по себе текстовый файл программой ещё не является — системе нужно разрешить его выполнять. Флаг `+x` (execute) у `chmod` делает файл исполняемым, после чего его можно запустить, указав путь `./`:

```shell
$ chmod +x hello.sh
$ ./hello.sh
```

Первая строка — `#!/usr/bin/env bash` — называется shebang (от «sharp» и «bang» — `#` и `!`). Она сообщает системе, каким интерпретатором выполнять файл; без неё система не знала бы, что перед ней именно bash-скрипт, а не текст или код на другом языке.

## Переменные

Скрипту почти всегда нужно что-то запоминать — путь, имя, промежуточный результат. Для этого служат переменные:

```shell
#!/usr/bin/env bash

course="linux"
echo "Course: $course"
```

Здесь есть две детали, на которых спотыкаются почти все новички. Первая: при присваивании вокруг `=` не должно быть пробелов — `course = "linux"` уже не присваивание, а попытка запустить команду `course`. Вторая: чтобы прочитать значение переменной, перед именем ставят `$` — `$course` подставит «linux».

## Аргументы командной строки

Скрипт становится по-настоящему полезным, когда работает не с жёстко зашитыми значениями, а с тем, что ему передали при запуске. Эти переданные значения — аргументы — доступны под специальными именами:

```shell
#!/usr/bin/env bash

echo "Script name: $0"
echo "First argument: $1"
echo "Second argument: $2"
echo "All arguments: $@"
```

`$0` — это имя самого скрипта, `$1` и `$2` — первый и второй аргументы, а `$@` — сразу все аргументы. Если запустить скрипт так:

```shell
$ ./args.sh input.csv output.txt
```

то `$1` окажется `input.csv`, а `$2` — `output.txt`.

## Условия

Аргументам нельзя доверять вслепую: пользователь может указать несуществующий файл или вообще ничего не передать. Поэтому скрипты проверяют свои входные данные с помощью условий — конструкции `if`:

```shell
#!/usr/bin/env bash

file="$1"

if [ -f "$file" ]; then
  echo "File exists: $file"
else
  echo "File not found: $file"
  exit 1
fi
```

Проверка `[ -f "$file" ]` отвечает на вопрос «существует ли такой обычный файл». Если да — скрипт сообщает об этом и работает дальше; если нет — печатает ошибку и завершается с `exit 1`, давая понять вызывающему, что что-то пошло не так. Обратите внимание на кавычки вокруг `"$file"`: они обязательны. Без них имя файла с пробелом развалится на несколько слов, а пустой аргумент превратит проверку в бессмыслицу — кавычки защищают от обоих случаев.

> **Типичные ошибки.**
>
> - **Пробелы внутри `[ ]`.** `[$file -f]` или `[ -f$file ]` не работают: `[` — это команда, а её аргументы разделяются пробелами. Правильно — `[ -f "$file" ]` с пробелами *вокруг* скобок и операторов.
> - **`=` вместо `==` при сравнении и наоборот.** В `[ ... ]` для строк используется одинарный `=`, для чисел — `-eq`. `[ $a == $b ]` сработает в bash, но не в POSIX-`sh`; `[ $a = $b ]` — переносимо. Сравнивать строки через `-eq` нельзя — будет ошибка.
> - **Переменные без кавычек.** `if [ -f $file ]` при пустом `$file` превратится в `[ -f ]` и вернёт «истину», потому что одиночный аргумент `-f` `test` интерпретирует как «непустая строка». Кавычки `"$file"` решают и эту проблему, и проблему пробелов в именах.
> - **`if команда; then` против `if [ ... ]`.** `if grep -q ERROR file; then ...` — это полноценный идиоматический способ проверить «нашлось ли». Не нужно прогонять вывод через `[ -n "$(...)" ]` — `if` смотрит на код завершения.

## Циклы

Если действие нужно повторить для каждого файла или каждой строки, выручают циклы. Цикл `for` перебирает список — например, все лог-файлы в каталоге:

```shell
for file in *.log; do
  echo "$file"
  wc -l "$file"
done
```

Цикл `while` в связке с `read` идёт по файлу построчно — это типовой способ обработать содержимое файла строка за строкой:

```shell
while IFS= read -r line; do
  echo "$line"
done < input.txt
```

Конструкция выглядит непривычно, но её стоит запомнить как готовый рецепт: `IFS=` не даёт обрезать пробелы по краям строки, а `read -r` запрещает интерпретировать обратные слэши — вместе они гарантируют, что строка прочитается ровно так, как записана.

## Код завершения

Каждая команда в Linux, завершаясь, оставляет после себя число — код завершения. По соглашению `0` означает «всё прошло успешно», а любое ненулевое значение — ошибку. Прочитать код последней команды можно через специальную переменную `$?`:

```shell
$ grep "ERROR" app.log
$ echo $?
```

Именно на кодах завершения держится надёжность скриптов: по ним `if` понимает, удалась команда или нет. Чтобы скрипт не продолжал работу после первой же ошибки, в его начало часто добавляют строгий режим:

```shell
set -euo pipefail
```

Коротко: `-e` останавливает скрипт при первой ошибке, `-u` ругается на необъявленные переменные, `pipefail` замечает ошибку в середине конвейера. Режим помогает ловить проблемы рано, но требует аккуратности — некоторые команды (например, `grep`, который ничего не нашёл) возвращают ненулевой код вполне законно, и это придётся учитывать.

## Разобранный пример: скрипт ежедневной выгрузки

**Задача.** Написать `daily-export.sh`, который принимает первым аргументом каталог с CSV-файлами, упаковывает его в архив с датой в имени, складывает архив в `/backup/` и удаляет файлы старше 14 дней. Скрипт должен останавливаться при первой ошибке и не оставлять временного мусора.

**Решение.**

1. Включить строгий режим — чтобы первая же ошибка остановила скрипт:
   ```shell
   #!/usr/bin/env bash
   set -euo pipefail
   ```
2. Проверить аргумент. Без явной проверки `set -u` поймает не ту ошибку — лучше вывести понятное сообщение:
   ```shell
   src="${1:?usage: daily-export.sh <src-dir>}"
   [ -d "$src" ] || { echo "Not a directory: $src" >&2; exit 1; }
   ```
3. Подготовить переменные. Имя архива — с датой; `date +%F` даст `2026-05-19`:
   ```shell
   stamp=$(date +%F)
   dest="/backup/$(basename "$src")-$stamp.tar.gz"
   ```
4. Создать архив. Кавычки вокруг `"$src"` и `"$dest"` обязательны — путь может содержать пробелы:
   ```shell
   tar -czf "$dest" "$src"
   ```
5. Удалить старые архивы. `find -mtime +14` отбирает файлы старше 14 дней; `-delete` удаляет их на месте:
   ```shell
   find /backup -name "$(basename "$src")-*.tar.gz" -mtime +14 -delete
   ```
6. Сообщить об успехе. Без этой строки трудно отличить «всё прошло» от «упало где-то посередине»:
   ```shell
   echo "Export OK: $dest"
   ```

**Проверка.** Запустите `./daily-export.sh ./data` — в `/backup/` появится `data-2026-05-19.tar.gz`, `echo $?` вернёт `0`. Передайте несуществующий каталог — скрипт ругнётся и завершится с кодом `1`. Создайте в `/backup/` пустой файл с датой двухмесячной давности — следующий запуск его удалит.

## Функции

Когда часть скрипта повторяется или внутри одной задачи легко выделяются «маленькие подзадачи», их выносят в функции. Это то же, что функция в любом языке программирования, только синтаксис своеобразный:

```shell
#!/usr/bin/env bash

log() {
  echo "[$(date '+%F %T')] $*"
}

backup() {
  local src="$1"
  local dst="$2"
  log "Backing up $src to $dst"
  tar -czf "$dst" "$src"
}

backup /var/www /backup/www.tar.gz
```

Аргументы передаются в функцию так же, как в скрипт: `$1`, `$2`, `$@`, и не пересекаются с аргументами самого скрипта. `local` объявляет переменную «локальной» — она не утечёт за пределы функции; без `local` любая переменная глобальна, и это самый частый источник трудных багов в bash-скриптах.

## `case`: выбор из нескольких вариантов

Когда нужно сравнить значение с несколькими шаблонами, `if/elif/elif/else` быстро становится громоздким. Для таких случаев есть `case` — он короче и сразу видит структуру:

```shell
#!/usr/bin/env bash

cmd="$1"

case "$cmd" in
  start)
    echo "Starting service"
    ;;
  stop)
    echo "Stopping service"
    ;;
  restart|reload)
    echo "Restarting service"
    ;;
  *)
    echo "Usage: $0 {start|stop|restart|reload}" >&2
    exit 1
    ;;
esac
```

Каждый блок завершается `;;`, шаблон `*` ловит «всё остальное» — это эквивалент `default` из других языков. В шаблонах работают те же glob-конструкции, что и в именах файлов: `*.log`, `[0-9]*`, `start|begin`.

## `trap`: реакция на сигналы и выход

Скрипт может оставить после себя «мусор»: временные файлы, частично записанные результаты, заблокированные ресурсы. Если пользователь нажмёт `Ctrl+C` посреди выполнения, мусор просто останется. `trap` решает эту проблему — вешает обработчик на сигнал (или на сам выход скрипта):

```shell
#!/usr/bin/env bash
set -euo pipefail

tmpdir=$(mktemp -d)
trap 'rm -rf "$tmpdir"' EXIT

# ... работа со $tmpdir ...
```

Псевдо-«сигнал» `EXIT` срабатывает при любом завершении скрипта — нормальном, при ошибке (`set -e`), при `Ctrl+C` или `kill`. Один `trap ... EXIT` в начале — самый надёжный способ убрать за собой временные файлы.

Можно вешать обработчики и на конкретные сигналы:

```shell
trap 'echo "Interrupted!"; exit 130' INT
trap 'echo "Reload config"; reload' HUP
```

## Here-doc: многострочные блоки

Иногда внутри скрипта нужно передать команде многострочный ввод — например, отправить SQL в `psql` или письмо в `mail`. Для этого есть конструкция «here-doc»:

```shell
psql -U postgres <<EOF
CREATE DATABASE mercury;
GRANT ALL ON DATABASE mercury TO mercury_user;
EOF
```

Всё между первой `EOF` и второй передаётся команде как stdin. Метка `EOF` может быть любой — `END`, `SQL`, `JSON`, лишь бы она не встречалась внутри блока. Внутри блока работают подстановки переменных: `$user`, `$(date)` и т.п. Если подстановки не нужны (например, передаёте код с кучей долларов), окружите метку кавычками — `<<'EOF'` — тогда содержимое передаётся буквально.

## Отладка скриптов

Когда скрипт ведёт себя не так, как ожидаешь, самый простой способ заглянуть внутрь — `set -x`: bash начнёт печатать каждую команду с подставленными значениями переменных до её выполнения. Включить можно для всего скрипта (`set -x` в начале) или для конкретного блока:

```shell
set -x
backup /var/www /backup/www.tar.gz
set +x
```

Видя на экране точный вид каждой команды, ошибку обычно находишь за секунды. После отладки `set -x` либо снимают (`set +x`), либо включают условно: `[[ "${DEBUG:-}" ]] && set -x` — тогда отладочный режим включается переменной окружения `DEBUG=1 ./script.sh`.
