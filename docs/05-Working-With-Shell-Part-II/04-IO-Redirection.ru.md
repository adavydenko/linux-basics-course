# Перенаправление ввода-вывода

- [Доп. видео](https://kodekloud.com/topic/io-redirection/)

По умолчанию команда печатает результат на экран, а ввод ждёт с клавиатуры. Но это поведение не высечено в камне: вывод можно направить в файл, ошибки — отделить от обычного результата, а несколько команд — соединить в конвейер, где одна передаёт работу следующей. Всё это вместе называется перенаправлением ввода-вывода, и в основе лежит простая идея трёх стандартных потоков. С неё и начнём.

## Цели раздела

После раздела студент должен уметь:

- объяснить, что такое `STDIN`, `STDOUT`, `STDERR`;
- перенаправлять обычный вывод в файл;
- перенаправлять ошибки в файл;
- добавлять вывод в конец файла;
- использовать `/dev/null`;
- соединять команды через pipe;
- использовать `tee`.

## Стандартные потоки

Каждый раз, когда вы запускаете команду, Linux открывает для неё три канала связи с внешним миром — стандартные потоки:

- `STDIN` (standard input) — стандартный ввод, откуда команда читает данные;
- `STDOUT` (standard output) — стандартный вывод, куда уходит обычный результат;
- `STDERR` (standard error) — отдельный поток для сообщений об ошибках.

![io](../images/ru_io.svg)

Ключевая мысль: ошибки идут не вперемешку с результатом, а по своему каналу. Именно поэтому их можно перенаправлять отдельно. Перенаправление — это и есть переключение любого из трёх потоков с экрана или клавиатуры на файл или на другую команду.

## Перенаправление `STDOUT`

Начнём с обычного вывода. Символ `>` перехватывает `STDOUT` и записывает его в файл вместо экрана:

```shell
$ echo $SHELL > shell.txt
```

Важный нюанс: `>` каждый раз создаёт файл заново, стирая прежнее содержимое. Если нужно дописать вывод в конец, не затирая старое, используйте удвоенный символ `>>`:

```shell
$ echo $SHELL >> shell.txt
```

## Перенаправление `STDERR`

Поскольку ошибки текут по отдельному потоку, у него есть свой номер — дескриптор `2`. Записав `2>`, вы перенаправляете именно ошибки, не трогая обычный вывод:

```shell
$ cat missing_file 2> error.txt
```

По той же логике, что и с обычным выводом, `2>>` дописывает ошибки в конец файла:

```shell
$ cat missing_file 2>> error.txt
```

Иногда ошибки не нужно ни видеть, ни хранить. Тогда их отправляют в `/dev/null` — специальный «чёрный ящик» системы, который молча поглощает всё, что в него попадает:

```shell
$ cat missing_file 2> /dev/null
```

## Pipe

Перенаправление в файл — это про сохранение результата. Но потоки можно соединять и между командами. Pipe (символ `|`) связывает две команды так, что `STDOUT` первой становится `STDIN` второй — результат одной команды сразу уходит на вход следующей:

```shell
$ grep Hello sample.txt | less
```

Здесь `grep` находит строки, а `less` тут же принимает их и показывает постранично — без промежуточного файла.

![pipe](../images/ru_pipe.svg)

## `tee`

Иногда хочется и сохранить вывод, и сразу увидеть его на экране. Обычный `>` так не умеет — он уводит весь поток в файл. Эту задачу решает `tee`: он читает данные из `STDIN` и раздваивает их — пишет одновременно и в файл, и в `STDOUT`:

```shell
$ echo $SHELL | tee shell.txt
```

Как и `>`, по умолчанию `tee` перезаписывает файл. Флаг `-a` (append) меняет это на дописывание в конец:

```shell
$ echo "This is the bash shell" | tee -a shell.txt
```

## Разобранный пример: ищем ошибку в большом логе и одновременно сохраняем результат

**Задача.** В файле `/var/log/syslog` нужно найти все строки со словом `error` (без учёта регистра), посчитать их количество и одновременно сохранить сам список ошибок в файл `errors.log` — чтобы потом отправить коллеге.

**Решение.**

1. Собираем строки с ошибками: `grep -i error /var/log/syslog`. Флаг `-i` — потому что в логах попадается и `Error`, и `ERROR`, и `error`.
2. Часть строк может быть в файлах, к которым нет доступа без `sudo`. Чтобы не засорять вывод сообщениями «Permission denied», перенаправляем `STDERR` в «чёрную дыру»: `grep -i error /var/log/syslog 2>/dev/null`.
3. Раздваиваем поток через `tee`: одна копия идёт в файл `errors.log`, вторая — дальше по конвейеру: `grep -i error /var/log/syslog 2>/dev/null | tee errors.log`.
4. На вход следующей команде нам нужен не сам список, а его длина — добавляем `wc -l` (word count, флаг `-l` — считать строки): `grep -i error /var/log/syslog 2>/dev/null | tee errors.log | wc -l`.
5. На экране остаётся только число — количество найденных строк, — а полный список ошибок уже лежит в `errors.log`.

**Проверка.** `wc -l errors.log` должен показать ровно то число, которое вывел конвейер. Если значения не совпадают — где-то в конвейере поток ушёл не туда, чаще всего из-за того, что `tee` стоит до или после нужной команды.

## Практическое задание

1. Запишите вывод `echo $SHELL` в файл.
2. Добавьте ещё одну строку в тот же файл через `>>`.
3. Выполните команду с ошибкой и перенаправьте `STDERR` в отдельный файл.
4. Соедините `grep` и `less` через pipe.
5. Запишите вывод команды одновременно на экран и в файл через `tee`.
