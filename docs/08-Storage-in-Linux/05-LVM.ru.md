# LVM: Logical Volume Manager

- [Доп. видео](https://kodekloud.com/topic/lvm/)

Представьте, что у вас есть несколько дисков разного размера, а нужен один большой раздел — или наоборот, один диск, который удобно делить на части и менять их размер по ходу работы. Обычные разделы для этого слишком жёсткие: их границы заданы раз и навсегда. LVM (Logical Volume Manager) добавляет между физическими дисками и файловыми системами гибкий промежуточный слой. Несколько дисков или разделов объединяются в общий «пул» — группу томов, — а уже из этого пула нарезаются логические тома нужного размера, которые при необходимости можно увеличить.

## Цели раздела

После раздела студент должен уметь:

- объяснить термины PV, VG и LV;
- создать physical volume;
- создать volume group;
- создать logical volume;
- создать файловую систему на LV;
- смонтировать LV;
- увеличить логический том и файловую систему.

## Основные понятия

LVM строится из трёх уровней, и схема выше показывает, как они вложены друг в друга:

- **`PV` (Physical Volume)** — физический том: реальный диск или раздел, отданный под управление LVM.
- **`VG` (Volume Group)** — группа томов: общий пул дискового пространства, собранный из одного или нескольких PV.
- **`LV` (Logical Volume)** — логический том: «виртуальный раздел», нарезанный из VG. Именно на нём создаётся файловая система, и именно его монтирует система.

## Работа с LVM

Пройдём весь путь — от чистого диска до смонтированного логического тома.

Сначала установим пакет с инструментами LVM (в большинстве систем он называется `lvm2`):

```shell
$ sudo apt install lvm2
```

Подготовим физический диск `/dev/sdb` для LVM — превратим его в physical volume. После успешного выполнения команда подтвердит результат:

```shell
$ sudo pvcreate /dev/sdb
Physical volume "/dev/sdb" successfully created
```

Соберём из этого PV группу томов и назовём её `caleston_vg`. В одну VG можно включить сразу несколько дисков, но для начала достаточно одного:

```shell
$ sudo vgcreate caleston_vg /dev/sdb
Volume group "caleston_vg" successfully created
```

Проверить, что получилось, помогают парные команды: `*display` показывает подробности, `*s` — краткую сводку. Посмотрим физические тома:

```shell
$ sudo pvdisplay
```

и группы томов:

```shell
$ sudo vgdisplay
```

Теперь из группы томов нарежем логический том. Создадим том `vol1` размером 1G в группе `caleston_vg`:

```shell
$ sudo lvcreate -L 1G -n vol1 caleston_vg
Logical volume "vol1" created.
```

Посмотреть логические тома можно так же — подробно или кратко:

```shell
$ sudo lvdisplay
$ sudo lvs
```

Логический том готов, но пока это «пустой раздел». Чтобы на нём можно было хранить файлы, нужно создать файловую систему — например, ext4:

```shell
$ sudo mkfs.ext4 /dev/caleston_vg/vol1
```

Осталось подключить том к дереву каталогов. Создадим точку монтирования и смонтируем в неё наш том:

```shell
$ sudo mkdir -p /mnt/vol1
$ sudo mount -t ext4 /dev/caleston_vg/vol1 /mnt/vol1
```

Теперь всё, что записывается в `/mnt/vol1`, попадает на логический том `vol1`.

## Увеличение logical volume

Главное преимущество LVM — тома можно расширять, не пересоздавая. Но прежде чем увеличивать том, убедимся, что в группе есть свободное место:

```shell
$ sudo vgs
VG #PV #LV #SN Attr VSize VFree
caleston_vg 1 1 0 wz--n- 20.00g 19.00g
```

В этом выводе нас интересуют две последние колонки: `VSize` — общий размер группы томов, `VFree` — сколько в ней пока свободно. Здесь свободно 19 из 20 ГБ, так что место для расширения есть. Если бы `VFree` был равен нулю, сначала пришлось бы добавить в группу новый физический том.

Увеличим логический том на 1G:

```shell
$ sudo lvresize -L +1G /dev/caleston_vg/vol1
Logical volume vol1 successfully resized.
```

Здесь есть важный нюанс. Команда `lvresize` увеличила сам том, но файловая система внутри него об этом ещё не знает — она по-прежнему занимает прежний размер. Убедиться в этом можно командой `df`:

```shell
$ df -hP /mnt/vol1
```

Чтобы файловая система «растянулась» на весь увеличенный том, для ext4 используется `resize2fs`:

```shell
$ sudo resize2fs /dev/caleston_vg/vol1
```

Теперь проверим результат ещё раз — размер файловой системы должен вырасти:

```shell
$ df -hP /mnt/vol1
Filesystem Size Used Avail Use% Mounted on
/dev/mapper/caleston_vg-vol1 2.0G 1.6M 1.9G 1% /mnt/vol1
```

Здесь есть один нюанс именования: к одному и тому же логическому тому в `/dev` ведут сразу два пути — они эквивалентны, но называются по-разному:

| Логический том | Путь в файловой системе          |
|----------------|----------------------------------|
| `vol1`         | `/dev/caleston_vg/vol1`          |
| `vol1`         | `/dev/mapper/caleston_vg-vol1`   |

Первый путь — «человеко-читаемый», создаётся через символическую ссылку на удобство пользователя. Второй — каноническое имя из подсистемы device-mapper, на которое и указывает первая ссылка. Использовать можно любой; в утилитах LVM (`mkfs`, `mount`, `lvresize`) обычно встречается первый вариант.

## Разобранный пример: «логи на `/var/log` упёрлись в 100 %»

**Задача.** Сервис на сервере начал валиться, потому что раздел `/var/log` заполнился. Раздел лежит на логическом томе `system_vg/log_lv` размером 2 ГБ; в группе томов есть свободные 5 ГБ. Нужно увеличить логический том и файловую систему на нём без остановки сервиса.

**Решение.**

1. Убеждаемся, что в группе томов есть свободное место: `sudo vgs`. В колонке `VFree` должно быть не меньше нужного запаса. Если `VFree` маленький — расширять группу через `vgextend` (это уже отдельная история, требующая нового PV).
2. Расширяем логический том на 2 ГБ: `sudo lvresize -L +2G /dev/system_vg/log_lv`. Знак `+` важен — без него `lvresize -L 2G` попытается *задать* размер 2 ГБ, то есть фактически ничего не изменит.
3. Проверяем, что том вырос: `sudo lvs system_vg/log_lv`. В колонке `LSize` должно быть `4.00g`. При этом файловая система внутри тома пока остаётся прежнего размера — `df -hP /var/log` всё ещё покажет 2 ГБ.
4. Расширяем файловую систему. Для ext4 — `sudo resize2fs /dev/system_vg/log_lv`; команда умеет работать «на горячую», размонтировать раздел не нужно. Для xfs пришлось бы использовать `xfs_growfs /var/log`.
5. Снова смотрим `df -hP /var/log` — размер должен совпасть с новым размером тома.

**Проверка.** `df -hP /var/log` показывает 4 ГБ и заметно меньшую загрузку. Сервис, который писал в `/var/log`, перестаёт падать. Если в шаге 4 пропустить `resize2fs`, том будет «формально» больше, но файловая система останется прежнего размера — самая частая ошибка при расширении.

## Важное предупреждение

Команды LVM меняют структуру хранения данных, и ошибка здесь может стоить целой файловой системы. Выполняйте их только на учебных дисках или в изолированной лабораторной среде — никогда на рабочем диске с нужными данными.
