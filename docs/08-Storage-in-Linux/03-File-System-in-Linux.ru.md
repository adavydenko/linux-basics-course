# Файловые системы в Linux

- [Доп. видео](https://kodekloud.com/topic/file-systems-in-linux/)

Размеченный раздел — это всё ещё просто кусок диска: записать в него файл нельзя. Чтобы на разделе появились каталоги, имена файлов и сами файлы, на нём нужно создать файловую систему — структуру, которая решает, где и как хранятся данные. В Linux таких систем несколько, но чаще всего встречается семейство EXT, и с самой распространённой из них — ext4 — мы и поработаем: создадим её, подключим к дереву каталогов и сделаем так, чтобы она поднималась после перезагрузки.

## Цели раздела

После раздела студент должен уметь:

- создать файловую систему ext4;
- создать точку монтирования;
- смонтировать файловую систему;
- проверить монтирование;
- добавить запись в `/etc/fstab`.

Семейство EXT развивалось постепенно — каждая версия добавляла к предыдущей что-то полезное. Краткое сравнение:

| Свойство          | EXT2                | EXT3                | EXT4                  |
|-------------------|---------------------|---------------------|-----------------------|
| Размер файла      | до 2 ТБ             | до 2 ТБ             | до 16 ТБ              |
| Размер тома       | до 4 ТБ             | до 4 ТБ             | до 1 ЭБ (экзабайт)    |
| Журналирование    | нет                 | есть                | есть                  |
| Контрольные суммы | нет                 | нет                 | есть (для журнала)    |
| Совместимость     | базовая             | обратная с EXT2     | обратная с EXT2/EXT3  |

В современных системах по умолчанию используют ext4: она поддерживает большие тома, ведёт журнал изменений (что упрощает восстановление после сбоя) и при этом совместима с предыдущими версиями.

## Работа с ext4

Первый шаг — создать на разделе файловую систему. Этим занимается семейство команд `mkfs` (make filesystem); для ext4 нужная команда — `mkfs.ext4`:

```shell
$ sudo mkfs.ext4 /dev/sdb1
```

Файловая система создана, но обращаться к ней пока не через что. В Linux диски не получают отдельных букв, как в Windows, — вместо этого файловую систему «подключают» к какому-то каталогу в общем дереве. Этот каталог называют точкой монтирования. Создадим его и смонтируем туда наш раздел:

```shell
$ sudo mkdir /mnt/ext4
$ sudo mount /dev/sdb1 /mnt/ext4
```

Теперь всё, что пишется в `/mnt/ext4`, попадает на раздел `/dev/sdb1`. Убедиться, что монтирование прошло, можно двумя командами: `mount` показывает список смонтированных файловых систем, а `df` — ещё и сколько на них места:

```shell
$ mount | grep /dev/sdb1
$ df -hP | grep /dev/sdb1
```

Есть одна тонкость: команда `mount` действует только до перезагрузки. Чтобы файловая система поднималась автоматически при каждом старте, её нужно описать в файле `/etc/fstab` — система читает его при загрузке и монтирует всё перечисленное.

Каждая строка `/etc/fstab` описывает одну файловую систему, поля идут в фиксированном порядке:

```text
# <file system> <mount point> <type> <options> <dump> <pass>
/dev/sda1 / ext4 defaults,errors=remount-ro 0 1
```

По порядку: что монтировать, куда, тип файловой системы, опции монтирования и два служебных числа (для резервного копирования и проверки при загрузке). Опции выше говорят: монтировать с разумными значениями по умолчанию (`defaults`), а при ошибке файловой системы переключить её в режим только для чтения (`errors=remount-ro`) — это стандартное поведение в Ubuntu. Добавим строку для нашего `/dev/sdb1` — `tee -a` дописывает её в конец файла:

```shell
$ echo "/dev/sdb1 /mnt/ext4 ext4 defaults 0 0" | sudo tee -a /etc/fstab
```

Но имя `/dev/sdb1` — ненадёжная привязка: стоит переставить диски или добавить новый, и буква может измениться, а запись в `/etc/fstab` начнёт указывать не туда. Поэтому в реальной работе вместо имени указывают UUID — уникальный идентификатор файловой системы, который не зависит от порядка подключения. Узнать его можно командой `blkid`:

```shell
$ blkid /dev/sdb1
```

Для удобства соберём назначение всех полей `/etc/fstab` в одну таблицу:

| Поле           | Назначение                                                              |
|----------------|-------------------------------------------------------------------------|
| `<file system>`| Что монтировать: имя устройства (`/dev/sdb1`) или UUID                  |
| `<mount point>`| Куда монтировать: путь к точке монтирования                             |
| `<type>`       | Тип файловой системы: `ext2`, `ext3`, `ext4`, `xfs`, …                  |
| `<options>`    | Опции монтирования: `rw` (read-write), `ro` (read-only), `defaults`, …  |
| `<dump>`       | Резервное копирование `dump`: `0` — пропускать, `1` — включать          |
| `<pass>`       | Порядок проверки `fsck` при загрузке: `0` — не проверять, `1` или `2` — проверять |

> **Типичные ошибки.**
>
> - **Опечатка в `/etc/fstab` ломает загрузку.** Если в строке `/etc/fstab` неверно указан путь к устройству, тип файловой системы или опции, система при загрузке может вообще не подняться и упадёт в emergency-режим. Перед перезагрузкой проверьте конфигурацию командой `sudo mount -a` — она попробует смонтировать всё описанное в `/etc/fstab`, и любые ошибки покажет сразу, без последствий для загрузки.
> - **Привязка по `/dev/sdb1` вместо UUID.** Имена устройств зависят от порядка обнаружения дисков ядром. Стоит переставить кабели или добавить новый диск — и `/dev/sdb1` может стать `/dev/sdc1`, а запись в `/etc/fstab` начнёт указывать не туда. В рабочих системах привязываются по UUID (узнаётся через `blkid`).
> - **Файлы «исчезли после монтирования».** Если в точке монтирования (`/mnt/ext4`) уже лежали файлы, после `mount` они станут невидимы — они никуда не делись, просто прикрыты содержимым смонтированной файловой системы. Размонтируйте — увидите их снова.
> - **Поле `<pass>` равно `1` для не-корневой ФС.** Значение `1` зарезервировано за корневой файловой системой (для проверки `fsck` при загрузке). Для всех остальных ставят `2` (проверять после корня) или `0` (не проверять).
